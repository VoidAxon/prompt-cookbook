# pr-desc

## Overview

ステージング済みの変更、直近のコミット、または指定コミットIDから、レビュアーが理解しやすい日本語のPR説明文を生成するスラッシュコマンド。変更の意図・影響範囲・確認ポイントを簡潔に伝えることを最優先とする。

## Usage

```
/pr-desc <mode> [context]
```

### mode

| 値 | 説明 |
|----|------|
| `stage` | ステージングされた差分から生成（mode 省略時のデフォルト） |
| `latest` | 直近のコミット履歴から生成 |
| `commit <id>` | 指定したコミットIDから生成。`commit` 直後の最初の引数を commit-id、それ以降を context として解釈する |

### context（任意）

チケットURL、仕様書リンク、注意点などの補足情報。

## Rules

### 出力形式
- Markdown形式・日本語（「である調」または体言止め）で記述する
- 出力全体を ```markdown コードブロックで囲み、PR本文のみを出力する（説明・分析・前置きは含めない）
- セクション見出しは Output Format と完全一致させる

### 内容の原則
- 変更目的・背景・影響範囲・挙動変化は、根拠（コミットメッセージ・差分・context・調査）なしに断定しない（→「推測と事実の区別」）
- コード識別子（クラス名・メソッド名・プロパティ名など）は変更箇所の特定に必要な場合のみ最小限引用し、コード断片（条件式・代入式・SQL・ファイルパスなど）は書かない
- ticket・spec URL だけで本文を省略しない（リンクは補強情報。Why・What・動作確認は本文に書く）
- コミットメッセージを箇条書きでそのまま貼り付けない（複数コミットは意味のある単位に再構成して要約する）
- コミットメッセージと最終差分が矛盾する場合（メッセージが squash 前の中間作業や削除済みの変更に言及している等）は、**最終差分に実在する変更を正**とし、メッセージは意図の参考に留める。差分に存在しない変更を `## 主要な変更点` に書かない
- `(Fix)` は症状・発生条件・対象画面を含め「何の問題を修正したか」が一文で伝わるように書く（コード層の原因は `## 変更目的・背景` 側に書く）。NG/OK例は examples.md 参照
- 単一PRに独立した複数目的（不具合修正＋機能追加＋大規模リファクタ等）が混在する場合、本文出力の前に `> **注意：** このPRは分割を推奨します（理由：◯◯）` を1行で提示する

### セクション分離（混在禁止）
情報が複数カテゴリにまたがる場合は、最も適切な1セクションへ振り分ける：

- `## 変更目的・背景`：Why（なぜ変更するのか）のみ。実装内容・修正箇所は `## 主要な変更点` へ
- `## 主要な変更点`：What（何を変更したか）のみ
- `## 動作確認`：How（どう確認したか）のみ。コード知識のないレビュアー/QAが**独立して再現・判定できる具体的な事実**で構成し、内部識別子（設定キー名・内部状態名・コード上の分類）を判定基準にしない。差分理解の補助情報は補足側へ
- `## レビュアーへの補足`：影響範囲・リスク・ロールバック方針・レビュー観点・推測情報を集約

### URL変換（pmed.cybozu.com）
- `pmed.cybozu.com` ドメイン かつ `#record=<整数>` を含む場合のみ、表示テキストを `<整数>` に変えた Markdown リンク形式に変換する
- 上記以外のドメイン、または `record=` の値が整数でない場合は、変換せずそのまま出力する
- 複数URLが存在する場合は各URLを個別に判定する

## 推測と事実の区別

差分・コミットメッセージ・context だけで断定できない情報は、**まず事実化を試みる**：変更された関数・メソッドの呼び出し元／呼び出し先・関連する処理フローを確認し、変更点が機能全体のどこに位置するかを把握する。変更差分に現れない識別子（呼び出すクエリ・メソッド・定数の定義など）に挙動が依存する場合も、その定義・実装をリポジトリ内で参照して確認する。

調査しても確定できないものだけを「推測」として扱い、次のとおり区別する：

| 推測の対象 | 記載先 | 記法 |
|---|---|---|
| 動作確認の確認項目 | `## 動作確認`（原処に残す） | `- [ ]`（未確認）形式。必要なら「※推測」を付す |
| 影響範囲・挙動変化・リスク | `## レビュアーへの補足` | 文中に「推測」と明示 |

- 確認できた事実は推測扱いにしない
- 推測すらできない場合のみ、`- [ ] テスト内容を記載` 等のプレースホルダーで代替してよい（必須項目には使用不可）

## Behavior

### 入力の優先順位

1. `commit <id>` 指定時：指定コミットの変更内容を最優先で分析し、コミットメッセージと差分を統合して要約する
2. `mode = latest`：直近の複数コミットを整理し、「直近の変更（概要）」として要約する
3. `mode = stage`（または mode 省略）：ステージングされた差分から変更意図を推測する

### 情報充足判定

差分・コミットメッセージ・context を解析し、以下の基準で不足項目を判定する：

| フィールド | 分類 | 不足時の対応 |
|-----------|------|-------------|
| 変更目的・背景 | 必須 | 追問する |
| 動作確認 | 重要 | 関連ファイル等を調査して確認項目を補う（→「推測と事実の区別」） |
| 影響範囲 | 重要 | 調査して推測する。推測不可なら追問する |
| 期待する挙動変化 | 重要 | 調査して推測する。推測不可なら追問する。判明分は `## 主要な変更点`（必要に応じて「変更前 / 変更後」）に記載する |
| リスク・ロールバック方針 | 任意 | 高リスク差分（DB移行・権限変更・データ削除・決済など）がある場合のみ追問する |

**必須フィールドが不足している場合：PR本文を出力せず、不足項目への質問のみを出力すること。**

質問優先順位（最大3問）：変更目的 → 影響範囲 → 挙動変化

質問文の例：
- 変更目的：「何の課題・要望に対応する変更ですか？」
- 影響範囲：「影響する画面・ユーザー・後方互換性について教えてください」
- 挙動変化：「挙動の変化を一言で教えてください（例：登録できなかった → 登録できる）」

ユーザー回答後：回答を context として取り込み、テンプレートに従って PR本文を生成する。

## Output Format

セクション見出しは以下と完全一致させること。`## 変更目的・背景` と `## 関連チケット・仕様書` は必須セクション。

```markdown
## 変更目的・背景
（変更の目的・背景）

## 主要な変更点
- (カテゴリ) 変更内容
- (Logic) 変更内容 ※以下の条件を満たす場合のみ変更前後を追記
  - 変更前: X
  - 変更後: Y

## 動作確認
- （確認済みの事実を記載。推測・未確認の項目は `- [ ] ...` 形式で区別する。空の `- ` やプレースホルダーを残さない）

## 関連チケット・仕様書
- （あれば記載、なければ「なし」）

## レビュアーへの補足
- （該当する内容があれば記載、なければ「なし」。以下の観点を含めてよい）
  - レビュー観点：重点確認してほしいファイル・箇所
  - リスク・影響範囲：後方互換性、データ移行、ロールバック方針
  - 推測情報：差分から推測した内容は「推測」と明示
```

### 「変更前 / 変更後」を書く方針

原則は「変更点（何を変えたか）」だけを書く。以下を**すべて満たす**「複雑な挙動変更」の場合のみ追記する：

1. 既存の挙動が変わる（新規追加・削除・リファクタは対象外）
2. 条件分岐・優先順位・例外系などが絡み、変更点の一文だけでは挙動差が伝わりにくい

| 区分 | 例 |
|------|----|
| 追記する | 条件付きで初期表示や判定ロジックを切り替える／特定条件下でのみ従来処理をスキップ・上書き／複数条件の優先順位が絡みケースごとに結果が変わる |
| 追記しない | 新機能追加（変更前が存在しない）／リネーム・整理・責務分離などのリファクタ／期待値が自明な軽微な不具合修正・誤字修正／文言・スタイルなどUIの軽微変更 |

記載ルール：
- 「変更前 / 変更後」は `Logic` の項目にのみ付ける（原則）
- 追記する場合も、詳細な実装名・内部変数・コード断片は書かない
- 追記不要なら行自体を出さない（空行・プレースホルダー禁止）

### 主要な変更点の粒度

**機能/挙動の単位**でまとめ、実装作業の単位（関数分割、移動、微調整の羅列）では書かない。レビュアーが短時間で「何がどう変わったか」を把握できる粒度に揃える。

| 同一項目に統合する | 別項目に分ける |
|---|---|
| 同じ画面/ユースケースに関する変更 | 目的が異なる（例：不具合修正 vs 性能改善） |
| 同じ原因に対する一連の修正（例外回避 + 入力チェック + メッセージ調整） | 影響範囲が異なる（例：管理者向け vs 一般ユーザー向け） |
| 1つの挙動変更を成立させるための付随変更（ログ追加・軽微な調整・リネーム等） | リスク・レビュー観点が異なる（例：UI変更 vs 権限制御・データ更新を伴う変更） |

書き方：
- 1項目は原則1〜2行（補足は箇条書きで追加）
- 「作業内容」ではなく「結果」を書く（NG：整理した／分割した　OK：保守しやすい構造にした）
- 数値・条件が重要なときだけ具体化、それ以外は抽象化
- 同種の軽微変更が複数ある場合は「〜など」で束ねる
- diff上の変更が多くても「主要」以外は書かない。細かい列挙が必要なら `## レビュアーへの補足` に回し「詳細は差分参照」と留める

### カテゴリ

`Logic` / `UI` / `Refactor` / `Fix` / `Test` / `Docs` — 変更内容に応じて付与する。必要に応じて追加可。

## 出力前チェック

PR本文を出力する前に、以下を自己点検する：

- 各記述が正しいセクションにあるか（Why＝目的・背景／What＝主要な変更点／How＝動作確認 の混在がないか）
- 最終差分に実在しない変更を `## 主要な変更点` に書いていないか（コミットメッセージ由来の記述に注意）
- 断定した影響範囲・挙動変化に根拠（差分・調査）があるか。確定できないものは「推測」と明示したか
- `## 動作確認` がコード知識なしで再現・判定できる事実で構成されているか（未確認項は `- [ ]`）
- コード断片・不要な内部識別子を書いていないか
- 必須セクション（変更目的・背景／関連チケット・仕様書）があり、見出しがテンプレートと完全一致するか

## Notes

- チケット番号（例：ABC-123）が含まれている場合は、関連チケット欄に記載すること
- NG/OK の具体例は examples.md を参照

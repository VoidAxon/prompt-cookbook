# pr-desc

## Overview

ステージング済みの変更、直近のコミット、または指定コミットIDから、レビュアーが理解しやすい日本語のPR説明文を生成するスラッシュコマンド。

あなたは、チーム開発の効率を最大化させるシニアソフトウェアエンジニアです。変更の意図・影響範囲・確認ポイントを簡潔に伝える、プロフェッショナルなPR説明文を作成してください。

## Usage

```
/pr-desc <mode> [context]
/pr-desc commit <commit-id> [context]
```

### mode

| 値 | 説明 |
|----|------|
| `stage` | ステージングされた差分から生成 |
| `latest` | 直近のコミット履歴から生成 |
| `commit <id>` | 指定したコミットIDから生成 |

### context（任意）

チケットURL、仕様書リンク、注意点などの補足情報。

## Rules

- Markdown形式で出力すること
- 日本語で記述すること（「である調」または体言止め）
- PR本文のみを出力し、説明や分析は含めないこと
- 変更目的・背景・影響範囲・挙動変化は根拠（コミットメッセージ、差分、context）なしに断定しないこと
- 調査から推測した影響範囲・挙動変化は `## レビュアーへの補足` に「推測」と明記すること
- コード識別子（クラス名・メソッド名・プロパティ名・設定キー名・変数名など）は、変更箇所の特定に必要な場合のみ最小限の引用を許可する
- コード断片（条件式・代入式・SQL・ファイルパスなど）は記載しない
- 出力は必ずコードブロック（```markdown）で囲んで出力すること
- pmed.cybozu.com のURL変換ルール：
  - `pmed.cybozu.com` ドメイン かつ `#record=<整数>` を含む場合のみ変換し、表示テキストを `<整数>` に変えMarkdownリンク形式で記載すること
  - 複数URLが存在する場合は各URLを個別に判定すること
  - `pmed.cybozu.com` 以外のドメインは変換せずそのまま出力すること
  - `record=` の値が整数でない場合はURLをそのまま出力すること
- PRの各セクションは役割を厳密に分離すること（情報の混在禁止）
  - `## 変更目的・背景`：Why（なぜ変更するのか）のみ
  - `## 主要な変更点`：What（何を変更したか）のみ
  - `## 動作確認`：How（どう確認したか）のみ
  - `## レビュアーへの補足`：影響範囲・リスク・ロールバック方針はここに記載すること
- 各セクションに他カテゴリの情報（背景・実装・検証手順など）を混在させないこと
- 情報が複数カテゴリにまたがる場合は、最も適切なセクションに自動的に振り分けること
- `## 変更目的・背景` に実装内容や修正箇所が含まれる場合は、それを削除し `## 主要な変更点` に移動すること

## Behavior

### 入力の優先順位

1. `commit <id>` が指定されている場合：指定コミットの変更内容を最優先で分析し、コミットメッセージと差分を統合して要約する
2. `mode = latest` の場合：直近の複数コミットを整理し、「直近の変更（概要）」として要約する
3. `mode = stage` の場合：ステージングされた差分から変更意図を推測する

### 情報充足判定

差分・コミットメッセージ・context を解析し、以下の基準で不足項目を判定する：

| フィールド | 分類 | 不足時の対応 |
|-----------|------|-------------|
| 変更目的・背景 | 必須 | 追問する |
| 動作確認 | 重要 | 差分・コミットメッセージ・context・関連ファイルを調査し、変更箇所に対応するテストケースを推測して記載する |
| 影響範囲 | 重要 | 差分・コミットメッセージ・context・関連ファイルを調査して推測する。推測不可なら追問する |
| 期待する挙動変化 | 重要 | 差分・コミットメッセージ・context・関連ファイルを調査して推測する。推測不可なら追問する |
| リスク・ロールバック方針 | 任意 | DB移行・権限変更・データ削除・決済など高リスクの差分がある場合のみ追問する |

**必須フィールドが不足している場合：PR本文を出力せず、不足項目への質問のみを出力すること。**

質問優先順位（最大3問）：変更目的 → 影響範囲 → 挙動変化

質問文の例：
- 変更目的 が不足：「何の課題・要望に対応する変更ですか？」
- 影響範囲 が不足：「影響する画面・ユーザー・後方互換性について教えてください」
- 挙動変化 が不足：「挙動の変化を一言で教えてください（例：登録できなかった → 登録できる）」

ユーザー回答後：回答を context として取り込み、テンプレートに従って PR本文を生成する。

## Output Format

```markdown
## 変更目的・背景
（変更の目的・背景）

## 主要な変更点
- (カテゴリ) 変更内容
- (Logic) 変更内容 ※以下の条件を満たす場合のみ変更前後を追記
  - 変更前: X
  - 変更後: Y

## 動作確認
- 
- 

## 関連チケット・仕様書
- （あれば記載、なければ「なし」）

## レビュアーへの補足
- （あれば記載、なければ「なし」） 
```

### 「変更前 / 変更後」を書く方針

- 原則：各項目は「変更点（何を変えたか）」だけを書く
- 例外：**複雑な挙動変更**が含まれ、変更点の一文だけではレビュアーが理解しづらい場合に限り、「変更前 / 変更後」を追記する

### 「複雑な挙動変更」とみなして追記する条件

以下を **すべて満たす** 場合のみ追記する：

1. 既存の挙動が変わる（新規追加・削除・リファクタは対象外）
2. 条件分岐・優先順位・例外系などが絡み、変更点の一文だけでは挙動差が伝わりにくい

### 追記しない（＝変更点だけでよい）例

- 新機能追加（変更前が存在しない）
- リネーム・整理・責務分離などのリファクタ
- 軽微な不具合修正で期待値が自明なもの（例：例外発生を防ぐ、誤字修正）
- 文言・スタイルなどUIの軽微変更

### 追記する例（変更前 / 変更後を併記）

- 既存の初期表示や判定ロジックを、条件付きで切り替える
- 特定条件下でのみ従来の処理をスキップ/上書きする
- 複数の条件や優先順位が絡み、結果がケースごとに変わるような変更

### 記載ルール

- 「変更前 / 変更後」は `Logic` の項目にのみ付ける（原則）
- 追記する場合も、詳細な実装名・内部変数・コード断片は書かない
- 追記が不要な場合は、テンプレ上の「変更前 / 変更後」行自体を出さない（空行・プレースホルダー禁止）

### 主要な変更点の粒度（細かすぎる記載を避ける方針）

- 目的：レビュアーが「何がどう変わったか」を短時間で把握できる粒度にそろえる
- 原則：**機能/挙動の単位**でまとめ、実装作業の単位（関数分割、移動、微調整の羅列）では書かない

**1項目にまとめる基準（細分化しない）**

以下は可能な限り同一項目に統合する：
- 同じ画面/同じユースケースに関する変更
- 同じ原因に対する一連の修正（例外回避 + 入力チェック + メッセージ調整）
- 1つの挙動変更を成立させるための付随変更（ログ追加、軽微な調整、リネーム等）

**分ける基準（項目を分離する）**

以下は別項目にする：
- 目的が異なる（例：不具合修正 と 性能改善）
- 影響範囲が明確に異なる（例：管理者向け と 一般ユーザー向け）
- リスクが異なる（例：データ更新を伴う変更）
- レビュー観点が異なる（例：UI変更 と 権限制御の変更）

**書き方のルール（過度に細かくしない）**

- 1項目は原則1〜2行に収める（補足が必要な場合のみ箇条書きで追加）
- 「作業内容」ではなく「結果」を書く（NG：整理した／分割した　OK：保守しやすい構造にした）
- 数値・条件が重要な場合だけ具体化し、それ以外は抽象化する
- 同種の軽微変更が複数ある場合は「〜など」として束ねる（列挙しすぎない）

**出力側の制約**

- diff上の変更が多くても、「主要」以外は原則書かない
- 細かい変更の列挙が必要な場合は `## レビュアーへの補足` に回し、「詳細は差分参照」として簡潔に留める

### カテゴリ

`Logic` / `UI` / `Refactor` / `Fix` / `Test` / `Docs` — 変更内容に応じて付与する。必要に応じて追加可。

### 必須セクション

`## 変更目的・背景` と `## 関連チケット・仕様書` は必須。セクション見出しは Output Format と完全一致させること。

## Missing Information Handling

可選フィールドや重要フィールドの推測が困難な場合のみ、以下のプレースホルダーを使用する：

- `- [ ] テスト内容を記載`
- `- [ ] 確認手順を記載`
- `- [ ] 影響範囲を記載`

必填フィールドにはプレースホルダーを使用しない（追問すること）。

## Notes

- コミットIDが指定されている場合は、その情報を最優先で使用すること
- チケット番号（例：ABC-123）が含まれている場合は、関連チケット欄に記載すること

# pr-desc examples

## stage モード — ステージ差分から生成

**コマンド**
```
/pr-desc stage
```

**出力例**
````markdown
## 変更目的・背景
ログイン画面でパスワード要件を視覚的に示す機能がなく、ユーザーがエラーになる理由を把握できなかった。

## 主要な変更点
- (UI) パスワード入力欄の下に強度インジケーターを追加
- (Logic) 8文字未満・記号なしの場合は送信ボタンを無効化
- (Refactor) バリデーションロジックを PasswordValidator クラスに分離

## 動作確認
- 4文字入力 → インジケーターが「弱」表示・送信ボタン無効
- 12文字以上・記号含む → インジケーターが「強」表示・送信ボタン有効

## 関連チケット・仕様書
- なし

## レビュアーへの補足
- なし
````

## latest モード — 直近コミット履歴から生成

**コマンド**
```
/pr-desc latest
```

**出力例**
````markdown
## 変更目的・背景
各ルートに散在していた認証ロジックを整理し、保守性を高める。

## 主要な変更点
- (Refactor) 認証・認可ロジックを AuthMiddleware クラスに集約
- (Refactor) トークン検証・ロール確認をメソッドとして分離
- (Test) AuthMiddleware のユニットテストを追加

## 動作確認
- 既存の全エンドポイントで認証が正常に動作することを確認
- 未認証リクエストが 401 を返すことを確認

## 関連チケット・仕様書
- なし

## レビュアーへの補足
- 動作仕様の変更はなし。リファクタリングのみ。
````

## commit モード — context（チケットURL）付き

**コマンド**
```
/pr-desc commit a1b2c3d https://example.pmed.cybozu.com/k/12/#record=456
```

**出力例**
````markdown
## 変更目的・背景
[456](https://example.pmed.cybozu.com/k/12/#record=456) の対応として、カート削除時のエラーを修正。

## 主要な変更点
- (Fix) CartItem が null の場合に発生していた NullPointerException を修正
- (Logic) 削除処理前に null ガードを追加

## 動作確認
- カートが空の状態で削除ボタンを押してもエラーが発生しないことを確認

## 関連チケット・仕様書
- [456](https://example.pmed.cybozu.com/k/12/#record=456)

## レビュアーへの補足
- なし
````

## NG / OK 対比

### 1. ticket-only 本文（リンクだけで内容を省略）

**NG**
````markdown
## 変更目的・背景
[456](https://example.pmed.cybozu.com/k/12/#record=456) の対応。

## 主要な変更点
- 詳細は ticket 参照

## 動作確認
- ticket の受入条件を満たすことを確認

## 関連チケット・仕様書
- [456](https://example.pmed.cybozu.com/k/12/#record=456)
````

**OK** — リンクは補強情報。Why・What・How を本文で明確に記述する：
````markdown
## 変更目的・背景
カート削除時に発生していた NullPointerException を解消し、空カート状態でも操作を継続できるようにする。

## 主要な変更点
- (Fix) CartItem が null の場合に発生していた NullPointerException を修正
- (Logic) 削除処理前に null ガードを追加

## 動作確認
- 空カート状態で削除ボタンを押下しエラーが出ないことを確認
- 通常の削除フローに影響がないことを確認

## 関連チケット・仕様書
- [456](https://example.pmed.cybozu.com/k/12/#record=456)
````

### 2. コミットメッセージのそのまま貼り付け

**NG** — コミット単位の作業ログをそのまま列挙：
````markdown
## 主要な変更点
- fix: nil チェック追加
- refactor: メソッド名を変更
- fix: テストを修正
- chore: 不要なログを削除
- fix: typo
````

**OK** — 機能/挙動単位に再構成し「結果」で書く：
````markdown
## 主要な変更点
- (Fix) 通知送信時の nil 参照エラーを修正（原因：受信者リストが空のケースが未考慮）
- (Refactor) 通知送信処理のメソッド名を責務に合わせて整理
````

---

### 3. (Fix) が抽象的すぎる

**NG** — 「何の問題」が伝わらない：
````markdown
## 主要な変更点
- (Fix) 注文画面のエラーを修正
- (Fix) 検索結果が出ない問題を修正
- (Fix) 不具合対応
````

**OK** — 症状・発生条件・対象を明示（コード層の実装原因は書かない、それは `## 変更目的・背景` 側）：
````markdown
## 変更目的・背景
在庫数取得や検索結果の表示で、特定条件下に画面が機能しなくなる事象が報告されていたため対応する。

## 主要な変更点
- (Fix) 注文画面で在庫数取得に失敗した際、エラー画面ではなく空配列として扱われ続けていた問題を修正
- (Fix) 検索キーワードを空欄のまま送信すると、検索結果が常に0件になる問題を修正
````

### 4. セクションの情報混在

**NG** — Why に実装詳細が混入：
````markdown
## 変更目的・背景
パスワード強度が分かりづらかったため、PasswordValidator クラスを新設し、`validateStrength()` メソッドで判定するように実装した。8文字未満は弱、12文字以上＋記号で強と判定する。

## 主要な変更点
- パスワード強度判定を実装
````

**OK** — Why は理由のみ、What は変更点に集約：
````markdown
## 変更目的・背景
ユーザーがパスワード要件を満たさずに登録に失敗する事例が多く、入力時点で強度を可視化する必要があった。

## 主要な変更点
- (UI) パスワード入力欄の下に強度インジケーターを追加
- (Logic) 8文字未満・記号なしの場合は送信ボタンを無効化
- (Refactor) 強度判定ロジックを PasswordValidator クラスへ分離
````

### 5. 複数目的が混在する巨大PR

**NG** — 不具合修正と新機能追加とリファクタが1つのPRに同居（警告なし）：
````markdown
## 主要な変更点
- (Fix) 決済時のタイムアウトを修正
- (Logic) クーポン機能を新規追加
- (Refactor) 注文ドメイン全体を DDD 構造へ全面移行
````

**OK** — 出力前に分割推奨を明示：
````markdown
> **注意：** このPRは分割を推奨します（理由：不具合修正・新機能追加・大規模リファクタが同居しており、レビュー観点とリスクが大きく異なるため）

## 変更目的・背景
...
````

# commit-msg

## Overview

ステージング済みの変更、直近のコミット、または指定コミットIDから、規約に沿った日本語のコミットメッセージを生成するスラッシュコマンド。

あなたは、チーム開発の効率を最大化するシニアソフトウェアエンジニアです。変更の意図と影響範囲を簡潔に伝える、プロフェッショナルなコミットメッセージを作成してください。

## Usage

```
/commit-msg <mode> [context]
/commit-msg commit <commit-id> [context]
```

### mode

| 値 | 説明 |
|----|------|
| `stage` | ステージングされた差分から生成 |
| `latest` | 直近のコミット履歴から生成 |
| `commit <id>` | 指定したコミットIDから生成 |

### context（任意）

チケット番号、背景・目的、注意点、仕様変更の理由など、差分だけでは読み取れない補足情報。

## Rules

- 日本語で記述すること（「体言止め」または「〜する」調）
- コミットメッセージ本文のみを出力し、説明や分析は一切含めないこと
- 出力は必ずコードブロック（` ```  `）で囲んで出力すること
- 技術的な実装詳細（クラス名・メソッド名・SQLなど）は必要最小限にとどめること
- 変更の「何を・なぜ」を中心に記述し、「どのように」は補足程度にとどめること
- context が指定されている場合は、その情報を差分より優先して意図・背景の解釈に使用すること

## Behavior

### プレフィックス選択

現在の Git ブランチ名を確認し、その目的に合致するプレフィックスを優先して選択する。

| プレフィックス | 用途 |
|---------------|------|
| `feat`     | 新機能の追加 |
| `fix`      | バグ修正 |
| `docs`     | ドキュメントのみの変更 |
| `refactor` | 仕様を変えないコードの整理 |
| `style`    | フォーマット修正（インデント・スペースなど） |
| `test`     | テストコードの追加・修正 |
| `perf`     | パフォーマンス改善 |
| `ci`       | CI/CD設定・スクリプトの変更 |
| `chore`    | ビルド設定やライブラリ更新など補助的な変更 |

### 入力の優先順位

1. `commit <id>` が指定されている場合：指定コミットの変更内容を最優先で分析し、コミットメッセージと差分を統合して要約する
2. `mode = latest` の場合：直近の複数コミットを整理し、変更内容を1コミットとして集約・要約する
3. `mode = stage` の場合：ステージングされた差分から変更意図を推測する

context が指定されている場合は、どのモードでも背景・目的の解釈に最優先で活用する。

## Output Format

```
<プレフィックス>: <50文字以内の要約>

- <変更の詳細1>
- <変更の詳細2>
- <変更の詳細3>
```

### フォーマット規則

- **1行目**：`<プレフィックス>: <要約>`（要約は50文字以内）
- **2行目**：空行（必須）
- **3行目以降**：`- ` で始まる箇条書きで変更詳細を列挙

## Output Example

```
feat: ログイン画面にパスワード強度チェック機能を追加

- 入力中にリアルタイムで強度インジケーターを表示
- 8文字未満・記号なしの場合は送信ボタンを無効化
- バリデーションロジックをPasswordValidatorクラスに分離
```

## Missing Information Handling

差分のみでは変更意図が不明確な場合：

- 推測で補完せず、読み取れる事実のみを記述する
- context の指定を促すコメントを出力の後に1行追記する：
  - `<!-- 変更の背景・目的を /commit-msg stage <補足情報> で指定するとより正確に生成できます -->`

## Notes

- チケット番号（例：#123、ABC-123）が context に含まれている場合は、要約行または詳細行に括弧書きで付記すること（例：`fix: 〇〇のバグを修正 (#123)`）
- 複数の目的が混在する差分の場合、最も主要な変更を1行目に据え、その他は箇条書きに含める
- リファクタリングと機能追加が混在している場合は `feat` を優先する

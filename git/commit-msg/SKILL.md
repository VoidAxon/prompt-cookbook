# commit-msg

## Overview

ステージング済みの変更、直近のコミット、または指定コミットIDから、規約に沿った日本語のコミットメッセージを生成するスラッシュコマンド。

あなたは、チーム開発の効率を最大化するシニアソフトウェアエンジニアです。変更の意図と影響範囲を簡潔に伝える、プロフェッショナルなコミットメッセージを作成してください。

## Usage

```
/commit-msg [mode] [context]
/commit-msg commit <commit-id> [context]
```

### mode

| 値 | 説明 |
|----|------|
| `stage` | ステージングされた差分から生成 |
| `latest` | 直近のコミット履歴から生成 |
| `commit <id>` | 指定したコミットIDから生成 |

mode が省略された場合は以下のフォールバック順で自動判定する：

1. staged 差分が存在する → `stage`
2. staged 差分なし／unstaged 差分あり → 警告を出力（`git add` を促す）
3. 作業ツリーがクリーン → `latest`

### context（任意）

チケット番号、背景・目的、注意点、仕様変更の理由など、差分だけでは読み取れない補足情報。

## Rules

### 表記・出力

- 日本語で記述すること（「体言止め」または「〜する」調）
- 日本語以外の自然言語は混入させないこと（コード識別子・固有名詞・チケットIDは除く）
- 絵文字は使用しないこと
- 1行目は句点（。）で終わらせないこと
- コミットメッセージ本文のみを出力し、説明や分析は一切含めないこと
- 出力は必ずコードブロック（` ``` `）で囲んで出力すること

### 本文の内容（WHY を書け、HOW を書くな）

コミットメッセージの本文は **diff から読み取れない情報** を伝えるためにある。実装の HOW は diff が説明するので書かない。

- **本文に書くこと**（いずれか該当するもの）：
  - **WHY**：変更の動機・解決する問題・その方針を選んだ理由
  - **影響**：既存挙動との差分・副作用・前提条件・互換性
  - **注意**：レビュアーや将来の自分が気にすべき落とし穴
- **本文に書かないこと**（diff を読めば分かるため冗長）：
  - クラス名・メソッド名・引数・SQL 等の実装識別子の羅列
  - 「〜を呼び出す」「〜に渡す」「〜に分離する」等の制御フロー・構造説明
  - 改名・整形・using 並び替え・null チェック追加等の機械的リファクタの記述
  - subject の言い換え

判断に迷ったら「これを読まないと diff だけでは理解できないか？」を問う。Yes なら書く、No なら削る。

### body 省略

- subject 行だけで変更の WHAT と WHY が伝わる単純な変更は body を省略してよい
  - 例：typo 修正、依存ライブラリのバージョン更新、自明な軽微 fix、フォーマット適用のみ

### context

- context が指定されている場合は、その情報を差分より優先して意図・背景の解釈に使用すること

## Behavior

### プレフィックス選択

現在の Git ブランチ名を確認し、その目的に合致するプレフィックスを優先して選択する。

| プレフィックス | 用途 |
|---------------|------|
| `feat`     | 新機能の追加 |
| `fix`      | バグ修正 |
| `docs`     | ドキュメントのみの変更 |
| `refactor` | 仕様を変えないコードの整理 |
| `style`    | フォーマット修正（インデント・スペースなど） |
| `test`     | テストコードの追加・修正 |
| `perf`     | パフォーマンス改善 |
| `ci`       | CI/CD設定・スクリプトの変更 |
| `chore`    | ビルド設定やライブラリ更新など補助的な変更 |

複数目的が混在する差分の場合は、最も主要な変更に対応するプレフィックスを採用する。リファクタリングと機能追加が混在している場合は `feat` を優先する。

### チケット番号の自動付記

context にチケット番号（例：`#123`、`ABC-123`）が含まれている場合は、要約行の末尾に括弧書きで付記する（例：`fix: 〇〇のバグを修正 (#123)`）。

context が指定されていない場合でも、現在のブランチ名から以下のパターンを検出できた場合は自動で付記する：

- `ABC-\d+` 形式（JIRA等のプロジェクトキー）
- `#\d+` 形式（GitHub Issue等）

例：`feature/ABC-456-login-fix` → `(ABC-456)` を付記

### 入力ソースの解釈

| mode | 解釈方針 |
|------|----------|
| `commit <id>` | 指定コミットのメッセージと差分を統合して要約する |
| `latest` | 直近の複数コミットを整理し、1コミットとして集約・要約する |
| `stage` | ステージングされた差分から変更意図を推測する |

context が指定されている場合は、どのモードでも背景・目的の解釈に最優先で活用する。

## Output Format

```
<プレフィックス>: <50文字以内の要約>

[<本文段落（任意）：WHY を中心に簡潔に。複数行可>]

[- <箇条書き（任意）：影響・前提・注意点を中心に、1〜3件目安>]
```

### フォーマット規則

- **1行目**：`<プレフィックス>: <要約>`（要約は**50文字以内・硬上限**）
- **2行目**：空行（必須）
- **3行目以降（任意）**：本文段落・箇条書き、またはその両方。WHY/影響/注意のみ
- 箇条書きを使う場合は **1〜3 件目安**、1 行あたり **72 文字以内**
- 1 行目の要約が 50 文字に収まらない場合は、副次的な情報を本文側に逃がす
- 本文に書く内容が無ければ subject だけで終わらせる（無理に箇条書きを埋めない）

## Error / Empty States

差分やコミットが取得できない場合は、コミットメッセージを生成せず、状況に応じて以下のいずれかを出力する：

- `stage` で staged 差分が空 → `<!-- ステージング済みの変更がありません。git add 後に再実行してください -->`
- `commit <id>` の id が存在しない → `<!-- 指定されたコミットIDが見つかりません: <id> -->`
- `latest` でコミット履歴が空 → `<!-- コミット履歴が存在しません -->`

## Missing Information Handling

差分のみでは変更意図が不明確な場合：

- 推測で補完せず、読み取れる事実のみを記述する
- context の指定を促すコメントを出力の後に1行追記する：
  - `<!-- 変更の背景・目的を /commit-msg stage <補足情報> で指定するとより正確に生成できます -->`

## Notes

- 1 コミットで明らかに独立した複数の目的が含まれている場合は、出力末尾に分割コミットを提案するコメントを追記する：
  - `<!-- 独立した変更が含まれています。分割コミットを検討してください -->`
- 出力例は [examples.md](examples.md) を参照

# 設計: generate-testdoc に「機能の全体把握」前置ステップを追加する

- 日付: 2026-06-10
- 対象: `tools/generate-testdoc/SKILL.md`
- ユースケース: **PR レビュー時の変更テスト**(差分ベース)

## 背景・動機

現状の `generate-testdoc` の Step 2「影響範囲を確認」は、**diff の変更行を起点に上流・下流へ辿る bottom-up 構造**になっている。機能理解に最も近いのは 2.3「業務名称の特定」だが、これも「変更に登場した識別子の業務名称を引く」だけで、**変更が属する機能そのものが何のための機能か**を先に押さえる top-down のステップが欠けている。

結果として、調用チェーンや影響範囲は trace できても「この機能が業務上何をするものか」を理解しないまま観点を書くと、技術的には正しくても**業務的に意味の薄い観点・分組**になりやすい。テストを書く前に機能を理解させる前置が必要、というのが本変更の動機。

## 決定事項(ブレインストーミングで確定)

| 論点 | 決定 |
|------|------|
| 産物の形態 | **AI 内部の理解にとどめる。単独の出力・ケースにはしない**(JSON スキーマ・テンプレート・出力ファイルは変更しない) |
| 理解の深さ | **変更規模に自適応**。小規模修正は機能目的を 1〜2 文、業務ロジック/フロー変更は業務フロー層まで |
| 情報源 | **自適応 + 明示的な歯止め**。既定は PR 説明文・diff・2.3 で読む範囲。意図が掴めない時のみ周辺コードを定向に読み、読んだ理由をメモ |
| 配置方式(採用案) | **方案 A**: Step 2 の冒頭に軽量な前置小節を追加 |

不採用案: 方案 B(2.4 へ追記のみ→「先に理解、次に差分」の順序が表現できない)、方案 C(独立 Step に昇格→既存 Step 番号と参照を全面改番、コスト過大)。

## 設計

### 配置と番号

- 配置: `SKILL.md` の `## Step 2: 影響範囲を確認(最重要)` 直下、**導入段落の後・`### 2.1 変更箇所の列挙` の前**。
- 番号: **`### 2.0 機能の全体把握(差分を見る前に)`**。
- `2.0` を採用する理由:
  - 既存の **2.3「業務名称特定」・2.5「スコープ/AI レビュー扱い」は外部参照を持つ**(`references/test-writing.md` に「Step 2.3」「Step 2.5 の原則」の参照あり)。
  - `2.1` を占有して以降を +1 すると、これらの参照と Step 2 内部参照(「2.2 で把握した」等)を全て改める必要があり、改動面が大きい。
  - `2.0` は「2.1 より前に行う」意味を表しつつ、**既存 2.1〜2.5 の番号と全参照を一字も変えない**。
- 既存の Step 2 導入段落(「差分の行だけを見るテストは本質的なリスクを見落とす…」)は**変更しない**。2.0 はその後ろに自然に続く。

### 2.0 と 2.4 の分担(重複回避)

- **2.0** = 機能は**何か**(対象ユーザー・業務目的・到達する業務フロー = what / why / who)
- **2.4** = 機能の**状態と組み合わせ**(データ有無・権限・フラグ・状態遷移 = テストの文脈)

角度が異なり重複しない。

### 追加する本文(確定草案)

````markdown
### 2.0 機能の全体把握(差分を見る前に)

変更行を見る前に、**この変更が属する機能そのものを先に理解する**。何のための機能か分からないまま差分だけを追うと、技術的には正しくても業務的に意味のない観点になる。最低限、次を一文で言えるようにする：

- **誰が・何のために**使う機能か(対象ユーザー・業務上の目的)
- **どんな業務フロー**の中で、どの画面・操作から到達するか

**深さは変更規模に合わせる(自適応)：**

- 小規模修正(表示文言・軽微なバグ修正など)：機能の目的を 1〜2 文で押さえれば十分。深追いしない
- 業務ロジック・フロー・状態遷移に関わる変更：到達経路と前後の業務フローまで広げて把握する

**情報源は軽い順に。発散させない：**

1. まず PR 説明文・diff・2.3 で業務名称を確認する際に読むコードの範囲で理解する
2. それでも**変更の業務的意図が掴めない場合のみ**、変更箇所の周辺コード(同一画面・クラスの他部分、主要な呼出元)を**定向に**読む。広く読み漁らない
3. 周辺コードを読んだ時は、**なぜ読んだかを一言メモ**する(例：`// 機能把握: 到達経路がdiffから不明なため呼出元を確認`)

ここで得た理解は **3.1 のグループ軸(業務機能エリア)と Step 4 の観点・期待値に反映する**。この把握自体は単独のテストケース・出力にはしない(内部理解にとどめる)。
````

## スコープ

### やること
- `SKILL.md` に上記 `2.0` 小節を 1 つ追加する。

### やらないこと(非目標)
- JSON スキーマ(`test_viewpoints` / `test_cases`)の変更 — しない
- Excel テンプレート・スクリプト・出力ファイルの変更 — しない
- 既存 2.1〜2.5 の番号・本文・外部参照の変更 — しない
- 「機能概要」を出力物として書き出す機能 — しない(内部理解にとどめる決定)

## 検証

- **差分の最小性**: `git diff` で SKILL.md の変更が `2.0` 小節の追加のみ(既存行の改変なし)であることを確認。
- **参照の健全性**: `references/test-writing.md` の「Step 2.3」「Step 2.5」参照が依然有効であることを確認(番号を動かしていないので維持されるはず)。
- **行為実跑**: クリーンな subagent に「SKILL.md に従って実在の merged PR を処理せよ」とだけ指示し(2.0 の存在を伝えない)、差分を見る前に機能の目的・業務フローを自発的に把握してから観点設計へ進むかを確認する。

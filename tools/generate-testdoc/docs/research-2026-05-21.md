## 2. 詳細：方向 1 — テスト観点理論体系

### 2.1 ISTQB / JSTQB Foundation Level

テスト技法を 3 カテゴリで分類：

- **ブラックボックス**：同値分割、境界値分析、デシジョンテーブル、状態遷移
- **ホワイトボックス**：ステートメント / ブランチカバレッジ、上位レベルで MC/DC
- **経験ベース**：エラー推測、探索的テスト、チェックリストベース

経験ベースは黒箱・白箱で見落としがちな欠陥を補完する位置づけ。

### 2.2 ISO/IEC/IEEE 29119-4 (2021)

Specification-based / Structure-based / Experience-based の 3 カテゴリで標準化。ISTQB と枠組みは同じだが、規格として Scenario testing / Random testing / Data flow testing / MCDC など、より広範な技法カタログを提供。

### 2.3 VSTeP（西康晴, 電気通信大学）

Viewpoint-based Software Test Engineering Process。「テスト観点」を NGT 記法でツリー構造化し、機能 / 入出力 / 品質特性 / 非機能 / 環境 などの軸で MECE に分解。JaSST/NaITE/WACATE で議論。

### 2.4 ISO/IEC 25010 製品品質モデル

8 つの品質特性：
1. 機能適合性
2. 性能効率性
3. 互換性
4. 使用性
5. 信頼性
6. セキュリティ
7. 保守性
8. 移植性

「正常系/異常系」だけでは捉えられない非機能品質を網羅。

### 2.5 現行 SKILL.md 7 観点への評価

| 現行観点 | 対応理論 | 評価 |
|----------|---------|------|
| 正常系 | ISTQB 有効同値クラス / ISO25010 機能適合性 | 妥当 |
| 異常系・堅牢性 | ISTQB 無効同値 / ISO25010 信頼性 | 信頼性まで踏み込めていない |
| 境界値・極限 | ISTQB BVA | 教科書的に正当 |
| 権限・セキュリティ | ISO25010 セキュリティ（5 副特性）| 副特性まで詰めると深堀可能 |
| 操作性・UX | ISO25010 使用性（6 副特性）| 妥当 |
| 連携・複合条件 | ISTQB DT / 状態遷移、ISO25010 互換性 | 「複合条件」と「外部連携」分離推奨 |
| 回帰確認 | テストレベル概念 / CI/CD 観点 | 妥当 |

**不足観点**：性能効率性、信頼性（成熟性・可用性・耐故障性・回復性）、互換性（共存性・相互運用性）、保守性・移植性、状態遷移の独立化

## 3. 詳細：方向 2 — 「テストすべきでない」工程実践

### 3.1 「テストすべきでない」典型カテゴリ

| # | カテゴリ | 理由 |
|---|---------|------|
| 1 | Private メソッド・内部 state | 公開 API の契約だけ守ればよい。Fragile Test 化 |
| 2 | 実装詳細（メソッド呼出順・内部分岐）| Change-Detector Test。refactor で落ちる |
| 3 | フレームワーク・サードパーティ自体の挙動 | 信頼前提なので冗長 |
| 4 | null チェック・例外伝播など自明な短絡条件 | 防御的実装の都合、業務要件ではない |
| 5 | 副次効果・自明前提・「エラーが出ない」型期待値 | Specific（失敗時の原因明示）に反する |
| 6 | モック呼出検証 | The Mockery、外側の振る舞いが壊れても通る |
| 7 | 同じ振る舞いを複数視点で重複検証 | Eager Test、ROI 低下 |
| 8 | コード識別子で書かれた assertion | 業務語でないと読めない、リネームで破綻 |

### 3.2 名前のついた反パターン

- **Eager Test**：1 テストで複数振る舞い
- **Mystery Guest**：fixture が外部で cause-effect 不明
- **Fragile Test**：本質と無関係な変更で壊れる
- **Change-Detector Test**：モック呼出記録だけ
- **The Mockery**：モック濫用
- **The Inspector**：getter/reflection で内部状態覗き見
- **General Fixture**：1 つの巨大 fixture を全テストで共有
- **アイスクリームコーン**：E2E/GUI 偏重ピラミッド逆転

### 3.3 コードレビュー vs テストの境界

| カテゴリ | 主担当 |
|----------|--------|
| 業務要件の充足・振る舞い | テスト |
| 境界値・edge case | テスト |
| 命名・可読性・設計意図 | レビュー |
| 副作用の妥当性・責務分離 | レビュー |
| null チェック・防御的実装 | レビュー |
| サードパーティの正しい使い方 | レビュー（+ 少数 integration test）|
| 既存テストでカバー済 | どちらも不要 |

### 3.4 現行 SKILL.md ルールへの評価

| 既存ルール | 整合性 |
|-----------|-------|
| 実装詳細はテストしない | 完全整合（Google・Fowler・MS・Beck）|
| 期待値は業務要件のみ | 整合（Beck "Behavioral"・"Specific"）|
| コード内部識別子禁止 | 整合（"Prefer Testing Public APIs"）|
| OR/AND は代表値 1 件 | 整合（補強余地：境界値別途必要を明示）|

### 3.5 推奨追記事項

1. フレームワーク自体の挙動はテストしない（明文化）
2. モック呼出検証は禁止（Change-Detector Test の例として）
3. 既に他テストでカバー済の振る舞いは PR テスト文書に再掲しない（ROI）
4. 境界値と同値分割の使い分け（同値は代表 1 件、境界は別ケース）
5. 「自明前提・副次効果・例外伝播」を NG 例付き禁止リスト化

## 4. 詳細：方向 3 — PR/差分駆動の粒度

### 4.1 Change-impact analysis の現代的実践

- **Microsoft Azure TIA**：実行時に「テスト ↔ ソース」依存マップを構築、PR 単位で影響テストだけ走らせる
- **Symflower (2024)**：`git diff` → パッケージ依存解析 → 影響テスト実行。共通設定変更時は full run フォールバック
- **Springer 2024 PR-based CIA**：call graph + history mining（共修正履歴）を PR 粒度で組合せ
- 粒度階層：environment → project → module → package → file → type → function → control flow

### 4.2 Risk-based testing で粒度を決める基準

- **Risk = Likelihood × Impact**（ISO 29119、Rex Black RBT）
- Likelihood：技術複雑度・チーム経験・変更頻度
- Impact：使用頻度・業務停止・法的責任・データ整合性
- **高リスク変更の典型**：
  - 業務ロジック・計算式
  - 認可・権限制御
  - 金額・在庫等のデータ整合性
  - 多数の呼出元を持つ共通ヘルパー
  - 状態遷移・トランザクション
  - 外部 IF / 契約変更
- 「件数より業務シナリオ数」の根拠：Rex Black の縦型/横型探索モデル。横型では「リスクアイテムごとに代表 1 件」

### 4.3 「テストケース粒度」を業界はどう決めているか

- **「1 シナリオ = 1 ケース」**（BDD/Cucumber, Automation Panda BDD 101）：
  - When-Then ペアが 1 つだけ
  - ステップ数 1 桁（< 10）
  - 複数 When/Then は分割サイン
- **「同値分割で代表値 1 件」**（ISTQB EP）：partition 内の他値は冗長、BVA と組合せて境界は別途

### 4.4 現行 SKILL.md 各 Step への評価

- **Step 2 影響範囲確認**：方向性は MS TIA・Springer 2024 と一致。ただし「直接/1-hop/N-hop/共通基盤」の段階区分が未明文化
- **Step 3.2.1 回帰要否判定**：MS TIA の「新規導入テストは無条件・既存影響テストは依存マップ選定」と整合
- **Step 3.5 変更種別 × リスク度**：10 種類定義は Rex Black より具体的で実用的。ただし Likelihood × Impact 計算が暗黙
- **Step 4.4 粒度ルール**：BDD「One Scenario, One Behavior」+ ISTQB EP と完全整合

### 4.5 推奨追記事項

1. **影響範囲の階層タグ化**：Step 2 に「直接 / 1-hop / N-hop / 共通基盤」4 段タグ。共通基盤変更時は full regression フォールバック明示
2. **リスクスコアリングの明文化**：3×3 簡易マトリクス、変更種別ごとに典型 L/I 値併記
3. **粒度判定フロー**：Step 4.4 前段に「業務シナリオ抽出 → 同値クラス → 代表値 → 境界値追加 → リスク Top で深掘り」5 ステップ ASCII フロー
4. **「やらないテスト」明示**：「省略ケース + 省略理由」セクション必須化
5. **PR サイズ閾値連動**：Google "Small CLs"（100 行妥当・1000 行大すぎ）引用、超過時 Excel 分割

## 5. 詳細：方向 4 — AI 辅助测试用例生成

### 5.1 主要ツール・研究

- **Diffblue Cover**：RL ベース Java 単体テスト 100% コンパイル保証。Copilot 比 20×
- **Qodo (旧 CodiumAI) Flow Engineering**：「Prompt Engineering より Flow Engineering」。`/test` 生成 + PR 段階 coverage gap 検出
- **arXiv 2407.00225**：GToT (Guided Tree-of-Thought) が CoT/FSL を上回り MSR 最高。Few-Shot が構造化応答を**悪化**するケースあり。`Cannot Find Symbol` hallucination が 64-86% で支配的
- **SmartHR Tech Blog (2025/09)**：探索的テスト補助。「Examples / Steps / Constraints」3 セクション、各カテゴリ 1-6 件制約で過剰生成抑制
- **SHIFT シナリオテスト AI (JaSST'25 Tokai)**：SQF ベース 6 フェーズ（目的明確化 → 仕様整理 → 業務フロー → シナリオ → 観点 → 因子水準）
- **gihyo (2024/10)**：日本語仕様 → Markdown 化 → prompt 添付。プロダクションコード起点はカバレッジ不足が頻出

### 5.2 AI テスト生成の典型的失敗パターン

| 失敗 | 症状 | Prompt 側対策 |
|------|------|--------------|
| Hallucination | 関数・API・テーブル名の捏造（64-86%）| 「diff に出現しない名前を作らない」「不明は推測せず情報なしと書く」|
| 実装詳細混入 | コード識別子・内部メソッド名残存 | 業務用語のみ・コード識別子禁止リスト |
| 過剰生成・冗長 | overthinking phenomenon (CANDOR) | カテゴリ毎件数上限・優先度付け |
| 正常系偏り | エッジ・異常系欠落 | エッジ・複数機能組合せ重視を明示 |
| 曖昧表現 | 「正しく表示される」等 | 期待値を「観測可能事象」に限定 |

### 5.3 効果的な prompt 設計パターン

1. **段階分け (Tree-of-Thought)**：「観点抽出 → ケース骨子 → 詳細化」
2. **チェックリスト型 Constraints**：Happy/Negative/Edge 必須列挙
3. **構造化区切り**：`###Test START###` 等の delimiter で抽出性改善
4. **業務用語強制 + 用語辞書**：日本語設計書 → Markdown 化
5. **不確実性の明示化**：「情報不足時は推測せず明記」→ hallucination 抑制
6. **Few-Shot は慎重に**：単体テストでは逆効果あり。1-shot のフォーマット安定化目的は有効
7. **Flow Engineering / 反復 loop**：「生成 → 自己検証 → 修正」サイクル

### 5.4 現行 SKILL.md への評価

| 項目 | 評価 |
|------|------|
| Step 2-6 構成 | 強い（SHIFT・GToT と同思想）|
| Think Negatively 5 自問 | 強い（CoT self-detection 21.9% → 58.2%）|
| name 書式「【種別】対象/条件 + 動作」 | 強い（構造化 delimiter 効果）|
| 業務用語強制 | 強い（gihyo・SmartHR と一致）|
| 最終チェックリスト 5 項目 | 強い（ただし不確実性表明ルール欠落）|

**弱点**：

- グループ毎件数上限が明示されていない
- 1-shot 理想ケースが固定埋め込みされていない
- 「情報不足時の振る舞い」が未規定

### 5.5 推奨追記事項

1. **グループ／観点ごとのケース数上限**（SmartHR 1-6 件踏襲）
2. **不確実性表明ルール**（`// 要確認: <理由>`）→ hallucination 遮断
3. **1-shot 理想ケース埋め込み**（フォーマット安定化）
4. **期待値の「観測可能事象」原則**（「正しく」「適切に」禁止）
5. **Self-review プロンプトの明文化**（生成済み全ケースを再走査）

## 6. 採用決定（5 項）と SKILL.md 反映方針

| ID | 補強 | SKILL.md 反映予定箇所 |
|----|------|---------------------|
| **C1** | 不確実性表明ルール | Step 4.3 書き方ルールに追加 + 最終チェック項目追加 |
| **C2** | 観測可能事象原則の強化 | Step 4.3「具体性」ルールを格上げ、禁止語リスト化 |
| **A1** | ISO 25010 非機能観点 | Step 3.4 観点角度の拡張、3.5 変更種別マッピング |
| **B1** | 影響範囲の階層タグ化 | Step 2.2 呼出チェーンに「直接 / 1-hop / N-hop / 共通基盤」追加 |
| **C5** | 独立 self-review フェーズ | Step 4.6 最終チェックを「実行ステップ」として独立化 or Step 5 として昇格 |

具体的な diff レベル設計は次の design ドキュメントで行う。

# テスト観点グループ軸に「品質特性軸（非機能）」を追加する

- 設計日：2026-06-17
- 対象：`SKILL.md` Step 3.1 / `references/viewpoint-design.md`
- 契機：PR `medley-inc/mall4#1417`（電子カルテの処理が遅い）の生成結果で、機能（メモ表示・行高さ）と非機能（応答時間＝性能）が 1 つのテスト観点グループに混在した。

## 1. 問題

現行 3.1 はグループの「軸」を **業務機能エリア単一軸**（業務機能 / 業務状態遷移 / 画面エリア）に限定していた。一方 ⑧性能・⑨信頼性・⑩互換性（ISO 25010 非機能、`viewpoint-design.md`）は **item レベルの観点角度**としてしか定義されていない。

結果、非機能の懸念は機能エリアグループの item（`（性能）` タグ付き）として吸収され、検証方法・合格基準・テストデータが根本的に異なる機能確認と非機能確認が同一グループに同居する。#1417 では性能改善が PR の主目的であるにもかかわらず、性能観点が機能グループの 3 番目の item に埋没した。

## 2. 根拠（調査）

- **VSTeP（西康晴, JaSST'13）**：テスト観点をツリー化し、複数観点をテストコンテナにグルーピングして俯瞰する。グルーピングは単一軸ではなく、機能 / 入出力 / 品質特性 / 非機能 / 環境 などの**多軸 MECE 分解**。
- **業界標準のテスト観点整理**：まず「機能テスト」と「非機能テスト（性能・信頼性・セキュリティ・使用性）」に大別。観点リストも「機能要素＋確認ポイント」と「評価アングル＋テストパラメータ」の 2 系統に分けるのが定石。
- **ISO/IEC 25010**：品質特性は機能適合性と並んで性能効率性・信頼性・互換性・セキュリティ・使用性等を独立特性として持つ。

→ 機能観点（業務機能エリア）と品質特性観点（非機能）は**別系統の軸**であり、後者は機能エリアに従属する item ではなく、独立したグループ軸になり得る。なお「正常系/異常系」は軸ではなく item（既存どおり Step 4）で正しい。

## 3. 変更

- **3.1 を 2 系統軸に再構成**：軸A（機能エリア）＋軸B（品質特性・非機能）。良い軸/悪い軸表に軸の別を明記し、「機能と非機能の混在」を悪い軸として追加。
- **3.1.1 を新設**：非機能を独立グループにする判定（検証方法／合格基準／テストデータが機能確認と異なる、またはその非機能改善が PR の主目的、のいずれか）。軽微なものは item に残してよい例外も明記。
- **3.2**：軸B グループは該当時のみ追加し統合する旨を追記。
- **`viewpoint-design.md`**：⑧⑨⑩ 注記から 3.1.1 を参照し、機能グループへの混在を禁止。

## 4. #1417 への当てはめ（期待される出力）

1. 外来患者一覧（三段）のメモ表示・行高さ（軸A・機能）
2. 外来患者一覧（三段）の大量件数表示時の応答時間（軸B・性能）← PR 主目的・検証方法/基準/データが別
3. 設定に依存する画面表示・動作の維持（軸A・回帰）

## 5. 非互換・副作用

- 軸の追加のみで、機能グループの切り方・粒度ルール・テストケース記述ルールは不変。
- 非機能変更がない PR の出力は従来と同一（軸B は該当時のみ）。

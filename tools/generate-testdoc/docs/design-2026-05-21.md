## 1. 背景

`generate-testdoc` skill は、PR/差分から QA 担当者向けの手動テストケース仕様書（Excel）を生成するための Claude Code 用 skill prompt である。現行 SKILL.md は 6 Step 構成で、業務用語強制・Think Negatively・粒度ルールなど主要な業界 BP と整合済み。

しかし、業界の最新実践（ISO 25010、Microsoft TIA、SmartHR/SHIFT/Diffblue 等の LLM-based テスト生成、arXiv 2407.00225 で 64-86% を占める hallucination 問題）に照らすと、5 つの補強余地がある。本 design はそれを最小侵襲で SKILL.md に反映する diff 設計である。

## 2. 採用方針

### 採用する 5 項

| ID | 補強内容 | ROI | 主根拠 |
|----|---------|-----|--------|
| C1 | 不確実性表明ルール | 高 | arXiv 2407.00225 hallucination 64-86%、LDI 事例 |
| C2 | 観測可能事象原則の強化 | 高 | SmartHR Constraints、業界共通 |
| A1 | ISO 25010 非機能観点（性能・信頼性・互換性）| 高 | ISO/IEC 25010 |
| B1 | 影響範囲の階層タグ化（L0/L1/LSHARED）| 高 | Microsoft TIA、Fowler "Rise of TIA" |
| C5 | 独立 Self-Review フェーズ（Step 4.7 新設）| 高 | CoT self-detection 効果 |

### 不採用（次回以降に再検討）

- B2 Risk = L × I 3×3 マトリクス：暗黙のままでも実用上問題なし
- C3 グループ毎ケース数上限：既存「業務シナリオ単位」で代用
- B3 省略ケース + 理由セクション：当面はチャット上で十分
- A2 状態遷移を「複合条件」から独立：現運用で混乱なし
- C4 1-shot 理想例固定埋め込み：skill が長くなる副作用
- A3 境界値 vs 同値分割の使い分け明示：既存規則で含意
- D2 既存カバー済再掲しないルール：影響軽微
- B4 PR サイズ閾値連動：既存 Excel 分割ルールで代用

### 整合策略：Approach A「分散精修」

5 項を各 Step の最適箇所に挿入し、骨格は維持する。

理由：
- 現有 6 Step 骨格は業界 BP と整合済みで動かす必要なし
- メモリに保持された Step 参照（2.5/3.5 等）と整合
- git diff が読みやすい

## 3. 具体的 Diff 設計

### 3.1 C1: 不確実性表明ルール

**目的**：hallucination の最大原因（diff に出ていない仕様の推測）を遮断する。

**修正 1**：Step 4.3「書き方ルール」表（[SKILL.md:249-258](../SKILL.md)）に行を追加。

```markdown
| **不確実性表明** | diff・関連コードから読み取れない仕様は推測しない。`// 要確認: <理由>` で明記し、テスター/開発者の確認を待つ | ❌「画面遷移後にダイアログ表示される（推測）」<br>✅「// 要確認: 保存成功後の画面遷移仕様が diff に明示されていない」 |
```

**修正 2**：Step 4.6 最終チェック表（[SKILL.md:293-302](../SKILL.md)）に行を追加。

```markdown
| **推測仕様の混入** | 「`// 要確認:` で明記されていない推測仕様」がケース文中にないか走査。あれば削除 or `// 要確認:` 化 |
```

### 3.2 C2: 観測可能事象原則の強化

**目的**：「正しく」「適切に」等の主観表現を全面排除し、検証可能事象に限定。

**修正 1**：Step 4.3「具体性」行（[SKILL.md:255](../SKILL.md)）を以下に置換。

置換前：
```markdown
| **具体性** | 「正しく」「適切に」「すぐに」「問題なく」「正常に」は禁止。**誰が読んでも同じ判断ができる**状態・数値・文言で書く | ❌「作成日が正しく表示」<br>✅「開始日・終了日が空欄かつ非活性（グレーアウト）」 |
```

置換後：
```markdown
| **観測可能事象で書く** | 期待値は「画面表示／DB 値／通知／ログ／帳票」など人が直接確認できる事象に限定。主観表現は全面禁止 | **禁止語**：「正しく」「適切に」「すぐに」「問題なく」「正常に」「期待通り」「うまく」「適切な順序で」「合理的に」<br>❌「作成日が正しく表示」<br>✅「作成日が `2025/06/01` 形式で表示」 |
```

**修正 2**：Step 4.6 既存項目「『正しく』『適切に』『正常に』残存」を以下に格上げ。

置換前：
```markdown
| **「正しく」「適切に」「正常に」残存** | 具体的な状態・文言に置換 |
```

置換後：
```markdown
| **主観表現残存（観測可能事象原則違反）** | 禁止語 9 個（4.3 参照）を grep。1 件でも残っていれば「画面・DB・通知・ログ・帳票」のいずれかの観測可能事象に置換 |
```

### 3.3 A1: ISO 25010 非機能観点

**目的**：機能観点偏重を是正し、性能・信頼性・互換性をカバー対象に明示。

**修正 1**：Step 3.4 観点角度表（[SKILL.md:156-167](../SKILL.md)）末尾に行を追加。

```markdown
| ⑧ | 性能・効率（ISO25010 #2）| DB 拡張・外部 API・大量データ処理が含まれる時。N+1、タイムアウト、スループット |
| ⑨ | 信頼性（ISO25010 #5）| エラー処理・トランザクション・リトライ・データ整合性に関わる変更時 |
| ⑩ | 互換性（ISO25010 #3）| API 契約・データフォーマット・DB スキーマ・依存ライブラリ更新時 |
```

表後に注釈を 1 行追加：

```markdown
※ ⑧⑨⑩ は ISO/IEC 25010 品質特性に基づく非機能観点。該当変更時のみ追加し、ない時は省略する。
```

**修正 2**：Step 3.5 変更種別 × リスク度表（[SKILL.md:170-184](../SKILL.md)）の「厚くする角度」列を更新。

| 変更種別 | 変更前 | 変更後 |
|----------|--------|--------|
| DB更新（SQL/migration）| ①② | **①②⑨⑩** |
| API連携変更 | ①② | **①②⑧⑩** |
| エラー処理変更 | ② | **②⑨** |

**修正 3**：Step 3.5 末尾の生成方針箇条書きに 1 件追加。

```markdown
- 非機能観点（⑧⑨⑩）は該当変更が**存在する場合のみ**追加。差分にない種別の観点は生成しない（既存ルールと整合）。
```

### 3.4 B1: 影響範囲の階層タグ化

**目的**：共通基盤変更時の full regression フォールバックを明示し、影響評価を再現可能にする。

**修正 1**：Step 2.2 呼出チェーン表（[SKILL.md:54-64](../SKILL.md)）を以下に置換。

置換前（既存 4 行）：
```markdown
| 方向 | 確認内容 | テスト設計への反映 |
|------|---------|-----------------|
| **上流（呼び出し元）** | UI イベント・ボタン操作・バッチ起動など、変更メソッドへの入口 | 操作手順の入口・前提条件を具体化 |
| **業務フロー** | どの**画面**→どの**メニュー/ボタン**を経由して到達するか。複数経路（一覧行クリック / ボタン押下など）は全て列挙 | 操作手順を「システムを知らない担当者が迷わず実施できる」粒度に |
| **下流（呼び出し先）** | DB 操作・外部 API・別モジュールへの呼び出し | 期待値の範囲（DB 状態・画面反映・連携先）を具体化 |
| **条件分岐** | if/switch の全分岐が差分にあるか。差分外の分岐への影響は？ | 差分に現れない分岐パスのテストを追加 |
```

置換後（6 行 + 段階タグ列追加）：
```markdown
| 方向 | 確認内容 | 段階タグ | テスト設計への反映 |
|------|---------|---------|-----------------|
| 直接変更 | diff の変更行そのもの | **L0** | 必ず全パスをカバー |
| **上流（呼び出し元）** | UI イベント・ボタン操作・バッチ起動など、変更メソッドへの入口 | **L1** | 操作手順の入口・前提条件を具体化 |
| **業務フロー** | どの**画面**→どの**メニュー/ボタン**を経由して到達するか。複数経路（一覧行クリック / ボタン押下など）は全て列挙 | **L1** | 操作手順を「システムを知らない担当者が迷わず実施できる」粒度に |
| **下流（呼び出し先）** | DB 操作・外部 API・別モジュールへの呼び出し | **L1** | 期待値の範囲（DB 状態・画面反映・連携先）を具体化 |
| **共通ヘルパー・基底クラス** | 3 箇所以上の呼出元を持つ共通コード | **LSHARED** | **full regression フォールバック必須** |
| **条件分岐** | if/switch の全分岐が差分にあるか。差分外の分岐への影響は？ | **L0** | 差分に現れない分岐パスのテストを追加 |
```

**修正 2**：表後に小節を追加。

```markdown
**段階タグ運用ルール：**

- **L0**（直接変更）：差分そのもの。必ず全パスカバー
- **L1**（1-hop）：直接の呼出元・呼出先。代表シナリオ 1 ケース
- **LSHARED**（共通基盤）：3 箇所以上の呼出元を持つヘルパー・基底クラス・共通定数。**変更時は回帰グループに「主要呼出元への影響なし」を全列挙**して full regression 相当をカバーする

参考：Microsoft Azure Test Impact Analysis、Martin Fowler "Rise of TIA"
```

### 3.5 C5: 独立 Self-Review フェーズ（Step 4.7 新設）

**目的**：機械的チェック（4.6）の後に、人間視点での再走査を 1 つの能動ステップとして独立化。CoT self-detection の効果を取り込む。

**修正**：Step 4.6 の後（[SKILL.md:302](../SKILL.md) の次）に Step 4.7 を新設。

```markdown
### 4.7 Self-Review 実行（必須）

Step 4.6 のチェック項目を機械的に走らせるのではなく、**生成済み全テストケースをあらためて読み直し**、以下を自問する。

1. **「このケースが落ちたとき、なぜ落ちたか即座にわかるか？」**
   → 答えが NO なら expected が曖昧 or 複数振る舞いが混入。分割 or 具体化
2. **「テスター（コードを知らない人）がこの手順で迷わず実施できるか？」**
   → 答えが NO なら手順を画面・ボタン・方向まで具体化
3. **「この期待値は画面 / DB / 通知 / ログ / 帳票のどれで確認できるか？」**
   → 即答できなければ観測可能事象原則違反（4.3 参照）
4. **「このケースは他のケースと業務シナリオとして本当に独立か？」**
   → 統合可能なら統合
5. **「`// 要確認:` を 1 件も書いていないが、本当に推測ゼロか？」**
   → 推測仕様があるなら `// 要確認:` 化（4.3 参照）

走査結果を踏まえて修正してから Step 5 のプレビュー or Step 6 の出力に進む。
```

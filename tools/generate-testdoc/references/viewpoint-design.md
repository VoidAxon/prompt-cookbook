# 観点設計の詳細ルール（generate-testdoc Step 3）

generate-testdoc の SKILL.md「Step 3: テストグループを設計」から参照される詳細ルール集。Step 3 の本文（グループの軸・グループ数の目安・items の書き方）は SKILL.md 側に残し、ここでは観点を漏れなく洗い出すための判定基準・思考フレーム・追加観点を定義する。

## 回帰確認の要否判定

「他コードへの業務機能上の影響」がある場合のみ回帰グループを追加する。完全な新規追加・既存処理の挙動が変わらない変更では**省略する**。

| 含める判定 | 含めない判定（回帰省略） |
|-----------|----------------------|
| 既存メソッドのシグネチャ・戻り値・例外パスが変わった | 完全に新規メソッド・新規フィールドの追加のみ |
| 共通ヘルパー・基底クラスを変更した | 既存処理の前後に新処理を挿入しただけ（既存処理の挙動不変） |
| 既存の if/switch 条件・分岐が変わった | 新規 if/switch の追加のみ |
| DB スキーマ・既存クエリが変わった | 新規テーブル・新規クエリの追加のみ |
| 既存 UI コンポーネントを変更した | 新規 UI コンポーネントの追加のみ |
| 共通定数・enum 値が変わった | 新規定数・新規 enum 値の追加のみ |

## 観点角度（思考フレーム）

グループ名・items に「正常系」「異常系」等のラベル文字をそのまま書かない。**Step 4 でテストケースを漏れなく生成するための思考チェックリスト**として使う。

| # | 角度 | 使うタイミング |
|---|------|---------------|
| ① | 正常系 | 通常フロー・設定通りの動作 |
| ② | 異常系・堅牢性 | エラー・不正入力・null・空文字・例外 |
| ③ | 境界値・極限 | 最大/最小・ゼロ件・連続操作・日付境界 |
| ④ | 権限・セキュリティ | 権限あり/なしの差異 |
| ⑤ | 操作性・UX | リアルタイムフィードバック・即時反映 |
| ⑥ | 連携・複合条件 | 他機能との組み合わせ |
| ⑦ | 回帰確認 | 他コードに業務機能上の影響がある場合のみ（回帰確認の要否判定 参照） |
| ⑧ | 性能・効率（ISO25010 #2） | DB 拡張・外部 API・大量データ処理が含まれる時。N+1、タイムアウト、スループット |
| ⑨ | 信頼性（ISO25010 #5） | エラー処理・トランザクション・リトライ・データ整合性に関わる変更時 |
| ⑩ | 互換性（ISO25010 #3） | API 契約・データフォーマット・DB スキーマ・依存ライブラリ更新時 |

※ ⑧⑨⑩ は ISO/IEC 25010 品質特性に基づく非機能観点。該当変更時のみ追加し、ない時は省略する。

## 変更種別 × リスク度

変更種別ごとに「どの角度を厚くするか」を決める。**差分にない変更種別のテストは生成しない**。

| 変更種別 | リスク | 厚くする角度 |
|----------|-------|--------------|
| 業務ロジック変更 | 高 | ①②③⑥ |
| 権限チェック変更 | 高 | ①④ |
| 入力バリデーション変更 | 高 | ②③ |
| DB更新（SQL/migration） | 高 | ①②⑨⑩ |
| 排他制御変更 | 高 | ①⑥（多重実行） |
| API連携変更 | 中 | ①②⑧⑩（タイムアウト・エラー応答） |
| エラー処理変更 | 中 | ②⑨ |
| 一覧表示変更 | 中 | ①（ソート・フィルタ・件数） |
| CSV出力変更 | 中 | ①③（空・大量） |
| UI変更（テンプレ/CSS） | 低 | ⑤ |

**生成方針：**

- **件数で測らない。「業務シナリオ数」で測る。** 1 業務シナリオ = 1 ケースを基本に。
- 業務シナリオとは「ユーザーが業務文脈で『これが動くこと』を期待する独立した挙動」。実装の派生シナリオ（null チェック・生成失敗時の挙動・短絡評価条件・例外伝播・前面表示順序など）はカウントしない。
- **OR / AND 等の論理条件は同値分割で代表値 1 件に集約**する（例: `設定A OR 設定B OR 設定C` → 「いずれか有効」1 ケース + 「全て無効」1 ケース）。設定ごとに別ケースは作らない。
- ⑦回帰は **回帰確認の要否判定 の判定で必要な場合のみ 1 ケース**（完全な新規追加では省略）。追加する場合も細かく分散させず、回帰グループに統合。
- ⑧⑨⑩ の非機能観点は**該当変更が存在する場合のみ**追加。差分にない種別の観点は生成しない（既存ルールと整合）。

## マイグレーション時の追加観点（該当時のみ）

diff に `db/migrate` / `schema.rb` / `*.sql` が含まれる場合のみ：

| 観点 | 確認内容 |
|------|---------|
| データ移行整合性 | 既存レコードがある状態でマイグレーション実行時、データ破損・欠落がないこと |
| ロールバック可否 | Down マイグレーション後にデータが正しく復元されること |
| NULL/デフォルト値 | NOT NULL 新規カラムに既存レコード分のデフォルト値が設定されること |
| 外部キー制約 | 追加された外部キー制約が既存データに違反しないこと |

## 逆方向テスト（制御緩和時に必須）

if 条件削除・ガード節撤廃・短絡評価条件除去を含む変更では、**緩和されたパスが通過できることを確認するケースを必須追加**：

```
例: IsBulletinBoardHistoryEmpty チェックを削除した変更
  → 「全データ表示期間外でも編集履歴画面が開けること」を追加
    （削除前は弾かれていたパスが、削除後に通過できるかを確認）
```

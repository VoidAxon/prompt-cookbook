# generate-testdoc

## Overview

PRまたはコミットの差分を解析し、**変更された箇所のみ**を対象としたテスト観点・テストケースをExcelテンプレートへ出力する。未変更コードへのテスト生成は行わない。

## 入力パラメータ

| 引数 | 必須 | 説明 |
|------|------|------|
| `KintoneID` | `pr`指定時は省略可 | Kintoneチケット ID |
| `Title` | `pr`指定時は省略可 | 変更タイトル |
| `pr <number>` | 任意 | PR番号（`gh` CLI で差分と説明文を取得） |
| `commit <sha>` | 任意 | 特定コミットのハッシュ |
| `repo <owner/repo>` | 任意 | 対象リポジトリ（デフォルト: `medley-inc/mall4`） |

**`pr` と `commit` は同時指定不可。両方省略時は `git diff --staged` を使用。**

**`pr` 指定時の KintoneID・Title 自動抽出：**
PR タイトルが `1234 患者一覧の検索条件修正` のように `数字 タイトル文字列` の形式の場合、KintoneID と Title を省略できる。スキルが PR タイトルの先頭数字を KintoneID、残りを Title として自動的に使用する。

呼び出し例：
```
/generate-testdoc pr 567                           # PRタイトルからKintoneID・Title自動抽出
/generate-testdoc pr 567 repo medley-inc/mall3     # 別リポジトリ
/generate-testdoc 1234 患者一覧の検索条件修正 commit abc1234
/generate-testdoc 1234 患者一覧の検索条件修正
```

## 実行フロー

### Step 1: パラメータ検証

- `pr` と `commit` の同時指定をチェック（エラーハンドリング参照）
- `pr` が指定されていない場合、`KintoneID` と `Title` が両方指定されているか確認
- `pr` が指定されている場合、KintoneID・Title は Step 2 で PR タイトルから自動抽出するため省略可

### Step 2: 差分・説明文を収集

デフォルトリポジトリ: `medley-inc/mall4`（`repo` で上書き可）

| 条件 | 差分取得コマンド | 説明文取得 |
|------|----------------|-----------|
| `pr <no>` | `gh pr diff <no> --repo <repo>` | `gh pr view <no> --repo <repo> --json body,title` |
| `commit <sha>` | `git diff <sha>^ <sha>` | なし |
| 両方省略 | `git diff --staged` | なし |

**`pr` 指定時の KintoneID・Title 自動抽出：**
`KintoneID` または `Title` が引数として渡されていない場合、取得した PR タイトルから抽出する。
- PR タイトルの先頭が数字のみのトークン（例: `162251`）→ `KintoneID` として使用
- それ以降の文字列 → `Title` として使用
- 先頭トークンが数字でない場合は KintoneID を空文字・PR タイトル全体を Title として使用

### Step 2.5: Confluence から機能仕様を取得

差分の変更対象（画面名・機能名）をキーワードとして Confluence（MALLDOC スペース）を検索し、関連する機能仕様ページを取得する。取得した内容は Step 5・Step 6 のテスト観点・テストケース生成の背景知識として活用する。

**検索手順：**

1. PR タイトル・diff のファイルパス・コメントから検索キーワードを 1〜2 語に絞って抽出する
2. `searchConfluenceUsingCql` で以下の CQL を実行し、**上位 3 件まで**取得する（`limit: 3`）：
   ```
   (space = "MALL" OR space = "MALLDOC") AND text ~ "<キーワード>" AND type = page
   ```
3. ヒットした 3 件のタイトルを確認し、変更対象に**最も関連性の高い 1 件だけ** `getConfluencePage` で読み取る
4. 3 件いずれも明らかに無関係な場合は読み取りを行わずスキップして Step 3 へ進む

**取得した仕様情報の活用ルール：**

| 活用目的 | 具体的な使い方 |
|----------|--------------|
| 画面・項目の正式名称 | テストケースの操作手順・期待値に仕様書の日本語正式名称を使う |
| 正常系の動作仕様・acceptance criteria | 期待値を「〇〇が表示されること」と具体化する根拠にする |
| 前提条件 | マスタ設定・権限・データ状態の具体的な条件を仕様書から拾う |
| **仕様とコードの不一致** | 仕様書に記載された動作がコードに実装されていない箇所 → 最も重要なテストケースになる |

差分コードと仕様書の記載が矛盾する場合は**差分（実装）を優先**しつつ、不一致を明記したテストケースを追加する。

### Step 2.7: コード全体の呼び出しフロー確認

差分で変更されたメソッド・ロジックを起点に、**上流（誰が呼ぶか）と下流（何を呼ぶか）** の呼び出しチェーンを確認し、差分に直接現れない影響範囲を把握する。

**確認手順：**

1. diff の変更行を一覧にする（修正されたメソッド名・クラス名・SQL を列挙）
2. 各変更箇所について以下を確認する：

| 方向 | 確認内容 | テスト設計への反映 |
|------|---------|-----------------|
| **上流（呼び出し元）** | UI イベント・ボタン操作・バッチ起動など、変更メソッドへの入口を特定 | テストケースの「操作手順」の入口と前提条件を具体化する |
| **業務フロー** | 「どの画面から」「どのメニュー・ボタンを経由して」その機能に到達するか | 操作手順ステップ1〜2を具体的な画面名・ボタン名で書けるようにする |
| **下流（呼び出し先）** | 変更メソッドが呼ぶ DB 操作・外部 API・別モジュールを特定 | 「期待値」の範囲（DB 状態変化・画面反映・連携先への影響）を具体化する |
| **条件分岐** | if/switch の全分岐が差分に含まれているか確認 | 差分に現れていない分岐パスへの影響テストを追加する |

3. 業務フローの確認では、diff のコードだけでなく**呼び出し元のファイル名・メソッド名をたどり**、以下を明確にする：
   - **起点画面**: どの画面（フォーム名）から始まるか
   - **到達経路**: メニュー → サブメニュー → ボタンクリック、など実際の操作パスを特定する
   - **複数経路**: 同じ機能に複数の入口がある場合（例: 一覧行クリック と ボタン押下の両方）は全て列挙する

   この情報が揃って初めて、テストケースの「操作手順」をシステムを知らない担当者が迷わず実施できる粒度で書ける。

4. **コード内の識別子（ID・定数・フラグ値）の業務的意味をコードから調べる：**

   diff に内部IDや定数が登場した場合、推測で業務名称を書いてはいけない。必ず以下の手順で実際の意味を確認する：

   - 定数・enum の定義元を `grep` で検索し、コメントや変数名から業務名称を特定する
   - 画面上の表示名（ラベル・メニュー項目・帳票タイトル）をコード（リソースファイル・View）から確認する
   - 不明な場合は diff のコメントや PR 説明文も参照する

   確認した業務名称を Step 6 のテストケース（precondition・steps・expected）に使う。内部IDや定数名はテストケースに書かない。

**出力：** この Step で発見した追加の影響範囲・観点を Step 3〜5 に反映させる。

---

### Step 2.8: 機能全体の状態・組み合わせを把握

差分を起点に、変更された機能が「どのような状態を持ち、どんな組み合わせで動作するか」を整理する。diff に現れた行だけを確認するテストでは本質的なリスクを見落とす。

**以下の4つの問いに答える：**

| 問い | 具体的に調べること |
|------|-----------------|
| **この機能はどんな状態を持つか** | データの有無・種類、ユーザー権限、設定フラグ、ステータス遷移など、変更箇所に関係するすべての状態をリストアップする |
| **重要な組み合わせはどれか** | 全組み合わせは不要。「変更されたロジックが通過するパス」に影響する組み合わせに絞る |
| **どこに影響するか** | Step 2.7 で把握した呼び出し元・呼び出し先・間接影響エリアを参照する |
| **以前のロジックは何を担っていたか** | 削除・変更されたコードの責務を確認する。「なぜそのコードが存在したか」を考え、削除後にその役割が別の場所で担保されているかを確認する |

**出力：** 整理した状態リスト・組み合わせ・影響エリアを Step 5 のグループ設計に反映させる。

---

### Step 3: 変更スコープを確定

差分とPR説明文から「何が変わったか」を特定する。以下はスコープ外として除外：

1. diffに変更行が含まれていないコード
2. PR説明文に「スコープ外」「今回は対応しない」「別チケット」と明記されたもの
3. コメント・空白・import整理のみの変更（ロジック変更なし）

差分コードとPR説明文が矛盾する場合は**差分を優先**する。

### Step 4: 変更影響分析とテストケースの絞り込み

テスト実施時間・工数は有限。「漏れなく網羅」より「効率よく重要なものを確認」を優先する。**差分に含まれない変更種別のテストは生成しない。**

**変更種別とリスク度：**

| 変更種別 | リスク度 | 使用するテスト観点（C列） |
|----------|---------|------------------------|
| 業務ロジック変更 | 高 | ①正常系・②異常系・③境界値・⑥連携 |
| 権限チェック変更 | 高 | ①正常系・④権限・セキュリティ |
| 入力バリデーション変更 | 高 | ②異常系・堅牢性・③境界値・極限 |
| DB更新（SQL/migration） | 高 | ①正常系・②異常系（整合性エラー） |
| 排他制御変更 | 高 | ①正常系・⑥連携・複合条件（多重実行） |
| API連携変更 | 中 | ①正常系・②異常系（タイムアウト・エラー応答） |
| エラー処理変更 | 中 | ②異常系・堅牢性 |
| 一覧表示変更 | 中 | ①正常系（ソート・フィルタ・件数） |
| CSV出力変更 | 中 | ①正常系・③境界値（空データ・大量データ） |
| UI変更（テンプレート/CSS） | 低 | ⑤操作性・UX |

**リスク度別の生成方針：**
- 高：最低3つの観点角度・計4行以上
- 中：最低2つの観点角度・計2行以上
- 低：最低1つの観点角度・計1行以上
- ⑦回帰確認は常に1行必ず追加

**テスト技法を使って効率よく絞り込む：**

| 技法 | 使いどころ |
|------|---------|
| 同値分割・境界値分析 | 入力値に範囲がある場合（文字数・数値・日付）。同じ結果になるグループは代表値1つで十分 |
| デシジョンテーブル | 複数の条件（ON/OFF）が組み合わさって動作が変わる場合。条件の組み合わせを表で整理し、重複を排除する |
| 組み合わせテスト | 設定項目が3つ以上ある場合。全組み合わせではなくペアワイズ（2因子間網羅）で絞り込む |

**状態遷移は双方向で確認する：** A→B だけでなく B→A も生成する。条件分岐は全分岐（if節とelse節の両方）を確認する。

**逆方向テスト（制御強化パターンの変更時に必須）：**

if条件の削除・短絡評価条件の除去・ガード節の撤廃を含む変更では、以下を確認するテストケースを追加する：
- 変更前に可能だった操作が、変更後も引き続き可能であること
- 制御が緩和された側のパスが正しく動作すること

```
例: IsBulletinBoardHistoryEmptyチェックを削除した変更
  → 「全データ表示期間外でも編集履歴画面が開けること」を確認するテストが必須
  　（削除前はチェックで弾かれていたパスが、削除後は通過できるかを確認する）
```

### Step 5: テスト観点グループを設計

Step 2.8 で整理した状態・組み合わせ・影響エリアを起点にグループを設計する。「diff で変わった行だけ」ではなく「変更によって動作が変わりうる全パス」を網羅するように意識する。

**グループ定義の方針：**
- Step 2.8 で列挙した状態・組み合わせを単位としてグループ化する
- 1PRで複数の機能が変わった場合は、それぞれ別グループにする
- 回帰確認は常に1グループとして追加する（「既存機能への影響」等）
- グループ数は変更の規模・影響範囲に応じて決める（小規模修正：2〜3、複数状態・パスあり：4〜7、大規模改修：それ以上も許容）

**グループ名は機能エリアだけ。条件・ケースは入れない：**

グループ名は「何の機能をテストするか」のみを表す。テストケースのパターン（変更あり・なし・異常時等）はグループ名に含めず、テストケースの `name` や `precondition` で区別する。

```
❌ 悪い例（条件をグループ名に入れている）:
  退院時の必要文書表示（入院形態変更あり）
  退院時の必要文書表示（入院形態変更なし）

✅ 良い例（機能エリアだけ）:
  退院時の必要文書表示
    ← 変更あり/なし/複数回変更の各ケースはテストケース側で区別する
```

**確認項目（items）の書き方：**

各グループで「何を確認すべきか」のキーポイントを簡潔に箇条書きする。長い説明を避け、テスト担当者が一目で把握できる粒度にする。

確認項目は、確認する対象の種類（**画面/DB/IF/ログ/通知** 等）が分かるように書く。「〇〇が画面に表示されること」「DBのXXカラムがYYになること」のように種別を明示することで、テスト担当者がどこを確認すればよいかが明確になる。

```
グループ: 「編集履歴」ボタンの動作
  確認項目:
    ・表示期間内/期間外/データなしの各条件で編集履歴画面が開くこと
    ・「掲示板の履歴がありません。」警告が表示されないこと

グループ: 既存機能への影響
  確認項目:
    ・フィルタ・新規作成・編集・削除への影響なし
```

**テスト設計の7角度（テストケース生成時の思考フレームワーク）：**

テスト観点シートのグループ名・確認項目には「正常系」「異常系」等の7角度ラベルをそのまま書かない。7角度はStep 6でテストケースを漏れなく生成するための思考チェックリストとして使う。

| # | 角度 | 使うタイミング |
|---|------|-------------|
| ① | 正常系 | 通常フロー・設定通りの動作 |
| ② | 異常系・堅牢性 | エラー・不正入力・null・空文字・例外スロー |
| ③ | 境界値・極限 | 最大/最小・ゼロ件・高速連続操作・日付境界 |
| ④ | 権限・セキュリティ | 権限あり/なしの動作差異 |
| ⑤ | 操作性・UX | リアルタイムフィードバック・即時反映 |
| ⑥ | 連携・複合条件 | 他機能との組み合わせ |
| ⑦ | 回帰確認 | 既存機能への影響なし（常にグループとして追加） |

### Step 5.6: インフラ・DB変更の特別観点（マイグレーションが含まれる場合）

diff に `db/migrate`・`schema.rb`・`*.sql` などのマイグレーションファイルが含まれる場合のみ適用する。含まれない変更には適用しない。

| 観点 | 確認内容 |
|------|---------|
| **データ移行の整合性** | 既存レコードがある状態でマイグレーションを実行した際、既存データが破損・欠落しないこと |
| **ロールバック可否** | 障害時に以前のバージョンに戻せること。Down マイグレーションがある場合は実行後にデータが正しく復元されること |
| **NULL / デフォルト値** | 新規追加カラムが NOT NULL の場合、既存レコードにデフォルト値が正しく設定されること |
| **外部キー制約** | 追加された外部キー制約が既存データに違反しないこと |

### Step 6: テストケースを生成

各グループについて、7角度を思考チェックリストとして使いながらテストケースを生成する。

| フィールド | 内容 | ルール |
|-----------|------|--------|
| group | 所属グループ | `test_viewpoints[].group` と**完全一致** |
| name | テスト名 | このテストケースの簡潔な名前（テストケースシートのテスト観点列に入る） |
| precondition | 設定・前提条件 | **このテストケース固有の条件のみ**記載（後述） |
| steps | 入力値・操作手順 | **必ずリスト形式**（スクリプトが自動で `1. / 2. ...` を付与）。文字列で書くと1文字ずつ展開されるバグが発生する |
| expected | 期待値 | 画面表示/DB更新/エラーメッセージを具体的に記載 |
| evidence | 証跡メモ（任意） | 何をエビデンスとして残すべきか。省略可。**使いどころ後述** |

**`evidence` フィールドの使いどころ：**

すべてのテストケースに書く必要はない。DB 状態変化の確認や画面だけでは判断しにくい場合にのみ付記する。単純な画面確認（「ボタンを押して成功メッセージが出る」）には書かなくてよい。

**テストケースは業務用語で書く：**

テストケース（precondition / steps / expected）はテスターが業務の視点で読む文書。コード内部の詳細を書いてはいけない。

| 書いてはいけないもの | 代わりに使う業務用語の例 |
|---------------------|--------------------------|
| フィールド名・カラム名（`is_renewed=false`） | 「初回の〇〇」「継続の〇〇」 |
| NULL値の条件（`is_renewed=NULL`） | 「改修前に作成された既存データ」「今回の機能追加前に作成された〇〇」 |
| 内部ID・定数（RecordTypeIdなど） | **コードを検索して実際の業務名称を確認してから**書く。推測で書かない |
| コード定数名（`NewRule2025`） | 「制度改正日（20XX/X/X）」「20XX/X/Xを境に」 |
| フラグ値（`is_renewed=true`） | 「継続として登録された」「継続扱いの」 |

**テストケースの粒度ルール：**

テストケースは「担当者が実施しやすい単位」で切る。細かすぎる分割は工数増・見通しの悪化を招く。以下のルールに従う。

| 場面 | ルール | 理由 |
|------|--------|------|
| **編集画面（入力フォーム）** | 入力項目を1項目ずつテストケースにしない。「保存成功」「保存後の表示確認」など動作単位で1〜2件にまとめる | 全項目を個別に書くと膨大になり本質的なリスクが埋もれる |
| **帳票（印刷プレビュー）** | 個々の印字項目をチェックしない。「保存した内容が帳票に正しく反映されること」を1件にまとめる | 帳票項目は多数あり、全体整合性を1件で確認するのが実務的 |
| **処理が複雑な機能** | 機能全体を1グループとしてまとめ、確認項目（items）にサブ観点を列挙する。テストケースはそのグループ内で複数件に分けて詳述してよい | 複雑な処理は分解して観点を整理した方がテスト漏れを防げる |

```
❌ やりすぎ:
  - 〇〇を選択して保存できること
  - △△を入力して保存できること
  ← 入力項目を1項目ずつケースにしない

✅ 適切:
  - 各項目を入力して保存が成功すること（1件）
  - 保存後に一覧・詳細画面で入力内容が反映されること（1件）
```

**1つのケースで複数観点を確認する場合：**

テスト観点（`name`）欄に複数の観点を併記するか、期待結果（`expected`）を箇条書きに分けて、どの観点に対応する確認かを明確にする。

```
例: 保存成功 + DB更新 + 画面反映を1件にまとめる場合
  name: 療養計画書保存・DB更新・画面反映
  expected:
    ・保存成功のトーストが表示されること
    ・DBのXXテーブルのYYカラムがZZに更新されること
    ・一覧画面に保存した内容が反映されること
```

**`name` フィールドの書き方：**

**体言止めの短いフレーズ（8〜15文字目安）**で書く。テストケースシートのテスト観点列に入る。「〇〇であること」で終わる文章形式は使わない。

「**対象/条件（圧縮）+ 動作キーワード**」の形が基本。条件は短縮し、動作（「保存」「表示」「遷移」「不表示」「影響なし」「無効」等）をキーワードとして末尾に残す。

```
❌ 長い文章形式: 入院形態変更あり患者の必要文書が正しく表示されること
❌ 動作なし:    入院形態変更あり患者（← 何を確認するか不明）

✅ 体言止めの短いフレーズ（会社サンプルスタイル）:
  初期データ投入          ← 動作名詞
  一覧画面表示            ← 画面＋動作
  表示フラグONで保存      ← 条件＋動作
  保存後の行選択          ← タイミング＋動作
  追加/削除/並替え無効    ← 操作名＋状態
  既存機能への影響なし    ← 回帰確認
  警告ダイアログ不表示    ← 否定確認
```

**前提条件・操作手順の書き方：**

前提条件はテストケース固有の条件のみ記載（共通前提は省略）。操作手順はシステムを知らない人が迷わない具体性で書く。

```
✅ 前提条件: 基本設定「全期間初期表示」がON、対象に表示期間外データが存在
✅ 操作手順: 「全期間」チェックボックスをクリックしてOFFにする
❌ 操作手順: チェックを切り替える（← どのチェック？どの方向？）
```

**E列（期待値）の書き方：**

「誰が読んでも同じ判断ができる」具体的な状態・数値・文言で書く。

```
✅ 開始日・終了日が空欄かつ非活性（グレーアウト）になっていること
❌ 作成日が正しく表示されること（← 「正しく」の基準が不明）

✅ 「〇〇」ダイアログが表示されないこと
❌ エラーが出ないこと（← どのエラー？）
```

**確認項目が複数ある場合は箇条書きで分ける：**

確認内容を短い箇条書きに分割し、テスト担当者が1項目ずつチェックできるようにする。

```
❌ 「〇〇」警告が表示されないこと。△△画面が直接開き、全履歴が一覧表示されること

✅ ・「〇〇」警告が表示されないこと
   ・△△画面が直接開くこと
   ・全履歴が一覧に表示されること
```

**禁止表現：**「正しく」「適切に」「すぐに」「問題なく」「正常に」（→ 何が・どんな状態・何秒以内かを具体的に書く）

**英語の画面名・設定名を使わない：**

日本語名が判明していれば日本語で書く。内部実装名（クラス名・メソッド名）は操作手順・期待値には書かない。

```
❌ RecordEditHistoryScreen が開くこと　→　✅ 〇〇編集履歴画面が開くこと
```

**"Think Negatively"（悪いシナリオを意図的に想定する）：**

正常フローだけを確認しても品質は担保できない。各テストケースを生成するたびに以下を自問し、見落としがちな失敗パターンを意図的に洗い出す：

| 問い | 具体例 |
|------|--------|
| この操作が**途中で失敗**したら何が起きるか？ | 保存ボタンを押した後に通信エラーが起きたら、データは残るか消えるか |
| **境界を1つ外れた**入力を渡したら？ | 上限値+1 / 下限値-1 / ゼロ件 / null を渡したときの挙動 |
| **同時操作**・**連続操作**したら？ | 同じボタンを素早く2回押す / 別ウィンドウから同じレコードを同時更新する |
| **権限のないユーザー**が操作したら？ | 閲覧専用ユーザーが更新ボタンを押したときにエラーが出るか |
| **データが空・最大・混在**しているとき？ | 一覧が0件 / 数千件 / 特殊文字を含む患者名のとき |

これらをすべてテストケースにする必要はない。「リスクが高い or 変更ロジックに近い」ものだけを選んで追加する。

### Step 6.5: テストケースの重複・統合チェック

JSON 出力前に、生成したテストケース一覧を見直して冗長なケースを削除・統合する。

**チェック項目：**

| 確認内容 | 対処 |
|---------|------|
| **前提条件・操作手順・期待値がほぼ同じケースが複数ある** | 1件に統合し、差異を `precondition` や `name` で区別する |
| **正常系のみで同じ画面を別グループから重複して確認している** | グループを統合するか、片方を削除する |
| **「影響なし」確認が細かく分散している** | 既存機能への影響グループにまとめ、操作手順を箇条書きで列挙する |
| **境界値ケースが同じ条件で別名で重複している** | 代表値1件に絞る（同値分割の原則） |

**目安：**
- 高リスク変更を含む PR で最終的なテストケース数が 30 件を超える場合は統合候補を探す
- テストケースを削るのではなく「同じリスクを別の切り口で確認している冗長なもの」だけを対象にする

### Step 6.7: プレビューとフィードバック（大型PR向け）

以下のいずれかに該当する場合は、JSON 出力前にテストグループ一覧をチャット上でユーザーに提示して確認を求める。該当しない場合はスキップして Step 7 へ進む。

**適用条件（いずれか1つを満たす場合）：**
- テストケース総数が **15件以上** になる見込み
- テスト観点グループ数が **5グループ以上**
- ユーザーが「確認してから進めてほしい」と明示した場合

**提示内容：**

```
【テスト観点プレビュー】

グループ1: <グループ名>
  確認項目: <items の箇条書き>
  ケース数（見込み）: N件

グループ2: ...

以上のグループ構成・観点でテストケースを詳細化します。
特定の画面やケースを厚くしたい場合、または除外・追加したいグループがあれば指示してください。
問題なければこのまま進めます。
```

ユーザーから追加・修正の指示があれば Step 5〜6 に戻って反映する。「問題ない」「進めて」という応答があれば Step 7 へ進む。

### Step 7: JSONを出力

以下のスキーマで**JSONのみ**をstdoutに出力する（前後の説明文・マークダウン記法なし）。

```json
{
  "summary": "string",
  "test_viewpoints": [
    {
      "group": "string",
      "items": ["string"]
    }
  ],
  "test_cases": [
    {
      "group": "string",
      "name": "string",
      "precondition": "string",
      "steps": ["string"],
      "expected": "string",
      "evidence": "string (省略可)"
    }
  ]
}
```

- `test_cases[].group` は `test_viewpoints[].group` と**完全一致**させること
- `test_cases[].name` はテストケースシートのテスト観点列に入るテスト名
- JSON生成に失敗した場合は不完全なJSONを出力せず、エラーを表示して停止

### Step 8: Pythonスクリプトを実行

JSONを一時ファイルに保存してスクリプトを呼び出す。

**注意：** Windows 環境では `/tmp/` が Python の `tempfile.gettempdir()` と異なる場合がある。Bash の `/tmp/` に書き込んだファイルは Python から参照できないため、Python の一時ディレクトリ（例: `C:/Users/<user>/AppData/Local/Temp/`）に直接保存して渡すこと。

**スキルのパス解決：** Claude Code はスキルをプロジェクト優先で探す（`.claude/skills/` → `~/.claude/skills/`）。スクリプト実行前に以下で実際のパスを確認すること：

```bash
# プロジェクト優先でスキルディレクトリを特定
SKILL_DIR=".claude/skills/generate-testdoc"
[ ! -d "$SKILL_DIR" ] && SKILL_DIR="$HOME/.claude/skills/generate-testdoc"
echo "$SKILL_DIR"
```

#### パターン A: 単一ファイル（通常ケース）

```bash
python "$SKILL_DIR/scripts/generate_testdoc.py" \
  --kintone-id "<KintoneID>" \
  --title "<Title>" \
  --input "<TempDir>/testdoc_<KintoneID>.json"
```

#### パターン B: 複数ファイルに分割（変更量が多いケース）

**分割の判断基準：**

- 変更が複数の独立した機能エリアにまたがり、テスト担当者が別々に実施する想定の場合
- 1つのJSONに20テストケースを大幅に超えるケースが集中する場合
- リリーススクリプト・設定系などのインフラ変更と、業務ロジック変更が混在する場合

**分割手順：**

1. Step 5〜6 でグループを設計する際に「分割軸」を決定する（例: 「新機能」と「既存改修」）
2. 各軸ごとに別のJSONファイルを作成する
3. `--title` の末尾に識別情報を付加して別ファイルとして出力する：

```bash
# 分割ファイル 1
python "$SKILL_DIR/scripts/generate_testdoc.py" \
  --kintone-id "<KintoneID>" \
  --title "<Title>（新機能）" \
  --input "<TempDir>/testdoc_<KintoneID>_1.json"

# 分割ファイル 2
python "$SKILL_DIR/scripts/generate_testdoc.py" \
  --kintone-id "<KintoneID>" \
  --title "<Title>（既存改修）" \
  --input "<TempDir>/testdoc_<KintoneID>_2.json"
```

→ 出力ファイルは `【KintoneID】 <Title>（新機能）.xlsx` / `【KintoneID】 <Title>（既存改修）.xlsx` として保存される。識別情報の命名は変更内容に合わせて適切に変えること（例: `（算定管理）`、`（印刷）`、`（DB移行）` など）。

## エラーハンドリング

| 状況 | 対応 |
|------|------|
| `git diff --staged` が空 | 「ステージングされた変更がありません。`pr` か `commit` を指定するか、`git add` 後に実行してください」と通知して停止 |
| `pr` と `commit` を同時指定 | 「`pr` と `commit` は同時に指定できません」と通知して停止 |
| `gh` CLI未インストール（`pr`指定時） | 「`gh` CLIが必要です。https://cli.github.com/ からインストールしてください」と通知して停止 |
| `gh` CLI未認証（`pr`指定時） | 「`gh auth login` で認証してください」と通知して停止 |
| 指定したPRが存在しない | 「PR #〇〇 が `<repo>` に見つかりません。番号またはリポジトリを確認してください（`repo` で指定可）」と通知して停止 |
| パラメータ不足 | 使用方法を表示して停止 |
| Pythonスクリプト不在 | パスを明示してエラー通知 |
| JSON生成失敗 | エラー内容を表示して停止（不完全なJSONは出力しない） |

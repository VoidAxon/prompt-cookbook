# generate-testdoc

PR またはコミットの差分を解析し、**変更箇所のみ**を対象としたテスト観点・テストケースを Excel テンプレートへ出力する。未変更コードへのテスト生成は行わない。

## 入力パラメータ

| 引数 | 必須 | 説明 |
|------|------|------|
| `KintoneID` | `pr`時は省略可 | Kintoneチケット ID |
| `Title` | `pr`時は省略可 | 変更タイトル |
| `pr <number>` | 任意 | PR番号（`gh` CLI で差分・説明文を取得） |
| `commit <sha>` | 任意 | コミットハッシュ |
| `repo <owner/repo>` | 任意 | 対象リポジトリ（デフォルト `medley-inc/mall4`） |

- `pr` と `commit` は同時指定不可。両方省略時は `git diff --staged`
- `pr` 指定時、PR タイトルが `1234 患者一覧の検索条件修正` 形式なら KintoneID・Title を自動抽出

呼び出し例：
```
/generate-testdoc pr 567                           # KintoneID・Title 自動抽出
/generate-testdoc pr 567 repo medley-inc/mall3
/generate-testdoc 1234 患者一覧の検索条件修正 commit abc1234
/generate-testdoc 1234 患者一覧の検索条件修正
```

---

## Step 1: 差分・説明文を取得

| 入力 | 差分 | 説明文 |
|------|------|--------|
| `pr <no>` | `gh pr diff <no> --repo <repo>` | `gh pr view <no> --repo <repo> --json body,title` |
| `commit <sha>` | `git diff <sha>^ <sha>` | なし |
| 省略 | `git diff --staged` | なし |

**`pr` 時の KintoneID・Title 自動抽出：**
PR タイトル先頭が数字トークンなら `KintoneID`、残り全体を `Title` に。先頭が数字でなければ KintoneID=空、Title=PR タイトル全体。

**`pr` 時の AI レビュー指摘の取得（任意）：**
`gh pr view <no> --repo <repo> --json reviews,comments` を実行し、**bot アカウント**（author type が Bot。例: copilot / claude / coderabbit）が投稿した review・comment を「AI レビュー指摘」として抽出する。Step 2 の影響範囲確認・Step 4.5 の Think Negatively で**参考情報**として活用する（扱いは 2.5 の原則に従う）。AI レビューが存在しない場合は何も取得せず黙ってスキップする。

---

## Step 2: 影響範囲を確認（最重要）

差分の行だけを見るテストは本質的なリスクを見落とす。**変更行を起点に「動作が変わりうる全パス」**を以下の順で確認する。

### 2.1 変更箇所の列挙

diff の変更行から、修正されたメソッド・クラス・SQL・定数を一覧化する。

### 2.2 呼び出しチェーンの確認

各変更箇所について、上流・下流をたどる：

| 方向 | 確認内容 | 段階タグ | テスト設計への反映 |
|------|---------|---------|-----------------|
| 直接変更 | diff の変更行そのもの | **L0** | 必ず全パスをカバー |
| **上流（呼び出し元）** | UI イベント・ボタン操作・バッチ起動など、変更メソッドへの入口 | **L1** | 操作手順の入口・前提条件を具体化 |
| **業務フロー** | どの**画面**→どの**メニュー/ボタン**を経由して到達するか。複数経路（一覧行クリック / ボタン押下など）は全て列挙 | **L1** | 操作手順を「システムを知らない担当者が迷わず実施できる」粒度に |
| **下流（呼び出し先）** | DB 操作・外部 API・別モジュールへの呼び出し | **L1** | 期待値の範囲（DB 状態・画面反映・連携先）を具体化 |
| **共通ヘルパー・基底クラス** | 3 箇所以上の呼出元を持つ共通コード | **LSHARED** | **full regression フォールバック必須** |
| **条件分岐** | if/switch の全分岐が差分にあるか。差分外の分岐への影響は？ | **L0** | 差分に現れない分岐パスのテストを追加 |

**段階タグ運用ルール：**

- **L0**（直接変更）：差分そのもの。必ず全パスカバー
- **L1**（1-hop）：直接の呼出元・呼出先。代表シナリオ 1 ケース
- **LSHARED**（共通基盤）：3 箇所以上の呼出元を持つヘルパー・基底クラス・共通定数。**変更時は回帰グループに「主要呼出元への影響なし」を全列挙**して full regression 相当をカバーする

参考：Microsoft Azure Test Impact Analysis、Martin Fowler "Rise of TIA"

### 2.3 コード識別子の業務名称を特定（推測禁止）

diff に内部 ID・定数・enum・フラグが登場したら、**必ずコードから業務名称を確認する**。推測で業務名称を書いてはいけない。

- 定数・enum の定義元を grep → コメント・変数名・リソースファイルから業務名称を取る
- 画面表示名（ラベル・メニュー項目・帳票タイトル）を View / リソースから確認
- 不明なら PR 説明文・diff コメントも参照

特定した業務名称を Step 4 の precondition / steps / expected に使う。

### 2.4 状態と組み合わせの整理

変更が動作する文脈を 4 つの問いで把握する：

| 問い | 調べること |
|------|-----------|
| この機能はどんな**状態**を持つか | データ有無・種類、ユーザー権限、設定フラグ、ステータス遷移 |
| 重要な**組み合わせ**はどれか | 変更ロジックが通過するパスに影響するもの**だけ**（全組み合わせは不要） |
| どこに**影響**するか | 2.2 で把握した上流・下流・間接影響エリア |
| **以前のロジック**は何を担っていたか | 削除・変更コードの責務。役割が別所で担保されているか確認 |

### 2.5 スコープを確定

以下はスコープ外として除外：
1. diff に変更行のないコード
2. PR 説明文で「スコープ外」「別チケット」と明記
3. コメント・空白・import 整理のみ（ロジック変更なし）

差分と PR 説明文が矛盾する場合は **差分を優先**。

**AI レビュー指摘の扱い（参考情報）：**
Step 1 で取得した AI レビュー指摘（境界・例外・影響範囲などの懸念点）は、テスト観点を洗い出す**ヒント**として活用する。ただし**権威ではなく参考**であり、以下に従う：

1. コードスタイル・リファクタ提案など、本変更のテストに無関係な指摘は無視する
2. diff と矛盾する指摘は **差分を優先**する
3. 指摘を逐条でテストケース化しない。無関係・誤報は捨てる

## Step 3: テストグループを設計

### 3.1 グループの「軸」を決める

**グループ = 業務上意味のあるエリア**。diff の塊・ファイル単位ではなく、テスト担当者が「この機能の確認」と理解できる単位で切る。

| ✅ 良い軸 | ❌ 悪い軸 |
|----------|-----------|
| 業務機能（例: 「退院時の必要文書表示」） | ファイル名・クラス名（例: 「DischargeServiceの修正」） |
| 業務状態の遷移（例: 「入院形態の変更時動作」） | diff の塊（例: 「コミット abc1234 の変更」） |
| 影響を受ける画面エリア（例: 「編集履歴画面」） | 観点ラベル（例: 「正常系」「異常系」← これは Step 4 で使う） |

**グループ名は機能エリアのみ。条件・ケースを名前に入れない：**

```
❌ 退院時の必要文書表示（入院形態変更あり）
❌ 退院時の必要文書表示（入院形態変更なし）
   ← 条件はテストケース側で区別する

✅ 退院時の必要文書表示
   ← 変更あり/なし/複数回変更は各テストケースで分ける
```

### 3.2 グループ数の目安

- 小規模修正：2〜3 グループ
- 複数状態・パスあり：4〜7 グループ
- 大規模改修：それ以上も許容
- **「回帰確認」は他コードに業務機能上の影響がある場合のみ追加**（下記 3.2.1 参照）

#### 3.2.1 回帰確認の要否判定

「他コードへの業務機能上の影響」がある場合のみ回帰グループを追加する。完全な新規追加・既存処理の挙動が変わらない変更では**省略する**。

| 含める判定 | 含めない判定（回帰省略） |
|-----------|----------------------|
| 既存メソッドのシグネチャ・戻り値・例外パスが変わった | 完全に新規メソッド・新規フィールドの追加のみ |
| 共通ヘルパー・基底クラスを変更した | 既存処理の前後に新処理を挿入しただけ（既存処理の挙動不変） |
| 既存の if/switch 条件・分岐が変わった | 新規 if/switch の追加のみ |
| DB スキーマ・既存クエリが変わった | 新規テーブル・新規クエリの追加のみ |
| 既存 UI コンポーネントを変更した | 新規 UI コンポーネントの追加のみ |
| 共通定数・enum 値が変わった | 新規定数・新規 enum 値の追加のみ |

### 3.3 確認項目（items）の書き方

各グループで「何を確認すべきか」のキーポイントを箇条書きで簡潔に。**確認対象の種別（画面 / DB / IF / ログ / 通知）を明示**する。

**最重要原則：items は「上位概念で束ねた確認軸」を書く。条件・値・パターンを逐条展開しない。**

- items は「この機能で何を担保すべきか」を **1 段抽象度を上げて** 記述する層。`表示期間内 / 期間外 / データなし` のような **個別条件の列挙は items に書かない**。
- 条件・境界値・パターンの **分解は Step 4 のテストケース側で行う**。items でそれを先取りすると、items とテストケースで同じ列挙が二重化し、グループの見通しが悪くなる。
- 判定基準：その項目が「条件 A の場合 / 条件 B の場合 / …」と**並列で増やせる**なら、それは上位概念（例: 「各表示条件で◯◯が開くこと」）に束ねるサイン。束ねた 1 行を items に置き、各条件は 4.4 の粒度ルールに従ってテストケースで分ける。

```
グループ: 編集履歴ボタンの動作
  ❌ 逐条展開（items でやらない）:
    ・表示期間内で編集履歴画面が開くこと（画面）
    ・表示期間外で編集履歴画面が開くこと（画面）
    ・データなしで編集履歴画面が開くこと（画面）
  ✅ 上位概念で束ねる:
    ・各表示条件（期間内/外・データ有無）で編集履歴画面が開くこと（画面）
       ← 個別条件はテストケースで分ける
    ・「掲示板の履歴がありません」警告が表示されないこと（画面）

グループ: 既存機能への影響
  ✅ 操作種別を 1 行に束ねる:
    ・既存の CRUD 操作（フィルタ・新規作成・編集・削除）への影響なし（画面）
```

### 3.4 観点角度（思考フレーム）

グループ名・items に「正常系」「異常系」等のラベル文字をそのまま書かない。**Step 4 でテストケースを漏れなく生成するための思考チェックリスト**として使う。

| # | 角度 | 使うタイミング |
|---|------|---------------|
| ① | 正常系 | 通常フロー・設定通りの動作 |
| ② | 異常系・堅牢性 | エラー・不正入力・null・空文字・例外 |
| ③ | 境界値・極限 | 最大/最小・ゼロ件・連続操作・日付境界 |
| ④ | 権限・セキュリティ | 権限あり/なしの差異 |
| ⑤ | 操作性・UX | リアルタイムフィードバック・即時反映 |
| ⑥ | 連携・複合条件 | 他機能との組み合わせ |
| ⑦ | 回帰確認 | 他コードに業務機能上の影響がある場合のみ（3.2.1 参照） |
| ⑧ | 性能・効率（ISO25010 #2） | DB 拡張・外部 API・大量データ処理が含まれる時。N+1、タイムアウト、スループット |
| ⑨ | 信頼性（ISO25010 #5） | エラー処理・トランザクション・リトライ・データ整合性に関わる変更時 |
| ⑩ | 互換性（ISO25010 #3） | API 契約・データフォーマット・DB スキーマ・依存ライブラリ更新時 |

※ ⑧⑨⑩ は ISO/IEC 25010 品質特性に基づく非機能観点。該当変更時のみ追加し、ない時は省略する。

### 3.5 変更種別 × リスク度

変更種別ごとに「どの角度を厚くするか」を決める。**差分にない変更種別のテストは生成しない**。

| 変更種別 | リスク | 厚くする角度 |
|----------|-------|--------------|
| 業務ロジック変更 | 高 | ①②③⑥ |
| 権限チェック変更 | 高 | ①④ |
| 入力バリデーション変更 | 高 | ②③ |
| DB更新（SQL/migration） | 高 | ①②⑨⑩ |
| 排他制御変更 | 高 | ①⑥（多重実行） |
| API連携変更 | 中 | ①②⑧⑩（タイムアウト・エラー応答） |
| エラー処理変更 | 中 | ②⑨ |
| 一覧表示変更 | 中 | ①（ソート・フィルタ・件数） |
| CSV出力変更 | 中 | ①③（空・大量） |
| UI変更（テンプレ/CSS） | 低 | ⑤ |

**生成方針：**

- **件数で測らない。「業務シナリオ数」で測る。** 1 業務シナリオ = 1 ケースを基本に。
- 業務シナリオとは「ユーザーが業務文脈で『これが動くこと』を期待する独立した挙動」。実装の派生シナリオ（null チェック・生成失敗時の挙動・短絡評価条件・例外伝播・前面表示順序など）はカウントしない。
- **OR / AND 等の論理条件は同値分割で代表値 1 件に集約**する（例: `設定A OR 設定B OR 設定C` → 「いずれか有効」1 ケース + 「全て無効」1 ケース）。設定ごとに別ケースは作らない。
- ⑦回帰は **3.2.1 の判定で必要な場合のみ 1 ケース**（完全な新規追加では省略）。追加する場合も細かく分散させず、回帰グループに統合。
- ⑧⑨⑩ の非機能観点は**該当変更が存在する場合のみ**追加。差分にない種別の観点は生成しない（既存ルールと整合）。

### 3.6 マイグレーション時の追加観点（該当時のみ）

diff に `db/migrate` / `schema.rb` / `*.sql` が含まれる場合のみ：

| 観点 | 確認内容 |
|------|---------|
| データ移行整合性 | 既存レコードがある状態でマイグレーション実行時、データ破損・欠落がないこと |
| ロールバック可否 | Down マイグレーション後にデータが正しく復元されること |
| NULL/デフォルト値 | NOT NULL 新規カラムに既存レコード分のデフォルト値が設定されること |
| 外部キー制約 | 追加された外部キー制約が既存データに違反しないこと |

### 3.7 逆方向テスト（制御緩和時に必須）

if 条件削除・ガード節撤廃・短絡評価条件除去を含む変更では、**緩和されたパスが通過できることを確認するケースを必須追加**：

```
例: IsBulletinBoardHistoryEmpty チェックを削除した変更
  → 「全データ表示期間外でも編集履歴画面が開けること」を追加
    （削除前は弾かれていたパスが、削除後に通過できるかを確認）
```

## Step 4: テストケースを生成

### 4.1 フィールド定義

| フィールド | 内容 |
|-----------|------|
| `group` | 所属グループ（`test_viewpoints[].group` と**完全一致**） |
| `name` | テスト観点列に入る簡潔な名前（後述 4.2） |
| `precondition` | このケース固有の前提条件のみ |
| `steps` | **必ずリスト形式**。文字列で書くと1文字ずつ展開されるバグ |
| `expected` | 画面表示 / DB 更新 / エラーメッセージを具体的に |
| `evidence` | 証跡メモ（任意）。DB 状態変化など、画面だけで判断しにくい場合のみ |

### 4.2 name の書式

**`【種別】対象/条件（圧縮）+ 動作キーワード`** で書く。体言止め。

prefix 種別：`【正常系】` `【異常系】` `【境界値】` `【権限】` `【回帰】`

動作キーワード：「保存」「表示」「遷移」「不表示」「影響なし」「無効」「開く」「閉じる」など。

```
❌ prefix なし:   全期間外患者で履歴画面が開く
❌ 文章形式:      【正常系】入院形態変更あり患者の必要文書が正しく表示されること
❌ 動作なし:      【正常系】入院形態変更あり患者
✅ 適切:
  【正常系】全期間外患者で履歴画面が開く
  【正常系】期間内データ患者で履歴画面が開く
  【異常系】DB例外時のエラーダイアログ表示
  【境界値】データなし患者の初期表示
  【回帰】掲示板操作・フィルターへの影響なし
```

### 4.3 書き方ルール

| 場面 | ルール | 例 |
|------|--------|-----|
| **業務用語で書く** | コード内部の識別子は禁止（カラム名・フラグ値・定数名・enum）。Step 2.3 で特定した業務名称を使う | ❌`is_renewed=false` → ✅「初回の療養計画書」<br>❌`NewRule2025` → ✅「制度改正日（2025/X/X）」<br>❌`RecordEditHistoryScreen` → ✅「編集履歴画面」 |
| **期待値は業務要件のみ** | 業務上の主目的だけを書く。実装で保証される副次効果（前面表示順・オーナー設定・null 安全・「エラーが出ない」等）は書かない。当然成立する前提（「画面が表示される」）も省く | ❌「ウィンドウが表示されること・前面に戻ること・オーナー設定されていること・エラーが出ないこと」<br>✅「メッセージ受信ウィンドウが表示されること」 |
| **観測可能事象で書く** | 期待値は「画面表示／DB 値／通知／ログ／帳票」など人が直接確認できる事象に限定。主観表現は全面禁止 | **禁止語**：「正しく」「適切に」「すぐに」「問題なく」「正常に」「期待通り」「うまく」「適切な順序で」「合理的に」<br>❌「作成日が正しく表示」<br>✅「作成日が `2025/06/01` 形式で表示」 |
| **操作手順** | システムを知らない人が迷わない粒度。どの画面・どのボタン・どの方向か明示。ただし「ログインする」のような当然の前段は省く | ❌「チェックを切り替える」<br>✅「『全期間』チェックボックスをクリックして OFF にする」 |
| **複数確認は箇条書き** | expected に複数の独立した確認項目があれば `・` で改行分割。1 項目で済むなら 1 行で書く | ✅「・警告が表示されないこと<br>・編集履歴画面が直接開くこと」 |
| **不確実性表明** | diff・関連コードから読み取れない仕様は推測しない。`// 要確認: <理由>` で明記し、テスター/開発者の確認を待つ | ❌「画面遷移後にダイアログ表示される（推測）」<br>✅「// 要確認: 保存成功後の画面遷移仕様が diff に明示されていない」 |

### 4.4 粒度ルール（テストケースを切る単位）

**「業務シナリオ単位」で切る。** 「ユーザーが業務文脈で『これが動くこと』を期待する独立した挙動」1 つにつき 1 ケース。実装の派生は別ケースにしない。

| 場面 | ルール |
|------|-------|
| **論理条件（OR/AND）** | 各値を別ケースにしない。「いずれか有効」「全て無効」のように業務シナリオ単位で集約 |
| **編集画面（入力フォーム）** | 入力項目を 1 項目ずつにしない。「保存成功」「保存後の表示確認」を 1〜2 件にまとめる |
| **帳票（印刷プレビュー）** | 印字項目を個別チェックしない。「保存内容が帳票に反映されること」を 1 件 |
| **実装詳細はテストしない** | null チェック・生成失敗時の挙動・短絡評価条件・例外伝播・前面表示順序など、コードで自明な実装詳細はケースにしない。業務挙動だけ |
| **複雑な処理** | 1 グループにまとめ、items にサブ観点を列挙。テストケースで複数件に分けて詳述してよい |
| **複数観点の併記** | 1 ケースで複数観点を確認する場合、name に併記 or expected を箇条書き分割で対応関係を明確化 |

```
❌ やりすぎ:
  - 〇〇を選択して保存できること
  - △△を入力して保存できること
✅ 適切:
  - 各項目を入力して保存が成功すること（1件）
  - 保存後に一覧・詳細画面で入力内容が反映されること（1件）
```

### 4.5 Think Negatively（必須チェックリスト）

各ケース生成後、以下の 5 項目を**必ず自問**する。**業務シナリオとして独立しており、かつリスクが高い or 変更ロジックに近いもののみ**ケース追加。実装で自明に保証される異常系（null チェック・例外伝播）はケースにしない。

加えて、Step 1 で AI レビュー指摘を取得している場合は**その懸念点も照合**し、業務シナリオとして独立かつリスクが高いものはケース化する（無関係・実装自明なものは従来どおり除外。扱いは 2.5 の原則に従う）。

| # | 自問 | 該当時の追加例 |
|---|------|---------------|
| 1 | この操作が**途中で失敗**したら？ | 保存後の通信エラー時、データが残るか消えるか |
| 2 | **境界を 1 つ外れた**入力なら？ | 上限値+1 / 下限値-1 / ゼロ件 / null |
| 3 | **同時操作・連続操作**なら？ | 同じボタンの2連打 / 別ウィンドウから同レコード同時更新 |
| 4 | **権限なしユーザー**が操作したら？ | 閲覧専用ユーザーが更新操作でエラー表示確認 |
| 5 | **データ空 / 最大 / 混在**時は？ | 一覧 0 件 / 数千件 / 特殊文字を含む患者名 |

### 4.6 最終チェック（出力前に必ず実行）

| チェック項目 | 不合格時の対応 |
|------------|---------------|
| **コード識別子残存** | テストケース全文を走査し、`snake_case` / `CamelCase` / `xxxFlag` / `xxxId` 等が残っていれば 4.3 表で業務用語に置換 |
| **実装詳細ケース残存** | 「null チェック」「生成失敗時の挙動」「複数 ON 時の重複表示なし」など実装で自明なケースが残っていないか確認。残っていれば削除 |
| **期待値の過剰記述** | 期待値に「前面に戻る」「オーナー設定される」「エラーが出ない」「画面が表示される」等の副次効果・自明な前提が含まれていないか。残っていれば削除し、業務要件のみ残す |
| **主観表現残存（観測可能事象原則違反）** | 禁止語 9 個（4.3 参照）を grep。1 件でも残っていれば「画面・DB・通知・ログ・帳票」のいずれかの観測可能事象に置換 |
| **推測仕様の混入** | 「`// 要確認:` で明記されていない推測仕様」がケース文中にないか走査。あれば削除 or `// 要確認:` 化 |
| **重複・冗長** | precondition / steps / expected がほぼ同じケースは 1 件に統合。「影響なし」の細かい分散は回帰グループにまとめる。同条件の境界値ケース重複は代表1件に絞る |
| **目安** | **業務シナリオ数を超えるケースがあれば統合候補を探す**（件数で判断しない）。**冗長なものだけ削る**。本質的カバレッジは削らない |

### 4.7 Self-Review 実行（必須）

Step 4.6 のチェック項目を機械的に走らせるのではなく、**生成済み全テストケースをあらためて読み直し**、以下を自問する。

1. **「このケースが落ちたとき、なぜ落ちたか即座にわかるか？」**
   → 答えが NO なら expected が曖昧 or 複数振る舞いが混入。分割 or 具体化
2. **「テスター（コードを知らない人）がこの手順で迷わず実施できるか？」**
   → 答えが NO なら手順を画面・ボタン・方向まで具体化
3. **「この期待値は画面 / DB / 通知 / ログ / 帳票のどれで確認できるか？」**
   → 即答できなければ観測可能事象原則違反（4.3 参照）
4. **「このケースは他のケースと業務シナリオとして本当に独立か？」**
   → 統合可能なら統合
5. **「`// 要確認:` を 1 件も書いていないが、本当に推測ゼロか？」**
   → 推測仕様があるなら `// 要確認:` 化（4.3 参照）

走査結果を踏まえて修正してから Step 5 のプレビュー or Step 6 の出力に進む。

## Step 5: プレビュー（大型 PR のみ）

以下のいずれかに該当時、JSON 出力前にテストグループ一覧をチャット上でユーザーに提示し、進めてよいか確認する：

- テストケース総数 **15 件以上** 見込み
- グループ数 **5 つ以上**
- ユーザーが「確認してから進めて」と明示

提示フォーマット：

```
【テスト観点プレビュー】

グループ1: <グループ名>
  確認項目: <items 箇条書き>
  ケース数（見込み）: N件

グループ2: ...

以上で詳細化を進めます。除外・追加・厚くしたい箇所があれば指示してください。
```

追加・修正指示があれば Step 3〜4 に戻る。「進めて」で Step 6 へ。

## Step 6: JSON 出力とスクリプト実行

### 6.1 JSON スキーマ

**JSON のみ**を stdout に出力（前後の説明文・マークダウン記法なし）：

```json
{
  "summary": "string",
  "test_viewpoints": [
    { "group": "string", "items": ["string"] }
  ],
  "test_cases": [
    {
      "group": "string",
      "name": "string",
      "precondition": "string",
      "steps": ["string"],
      "expected": "string",
      "evidence": "string (省略可)"
    }
  ]
}
```

- `test_cases[].group` は `test_viewpoints[].group` と**完全一致**
- JSON 生成失敗時は不完全な JSON を出力せず、エラーで停止

### 6.2 スクリプト実行

JSON を一時ファイルに保存してスクリプト呼び出し。

**注意：** Windows では Bash `/tmp/` と Python `tempfile.gettempdir()` が異なる場合がある。Python の一時ディレクトリ（例: `C:/Users/<user>/AppData/Local/Temp/`）に直接保存すること。

```bash
# プロジェクト優先でスキルディレクトリ特定
SKILL_DIR=".claude/skills/generate-testdoc"
[ ! -d "$SKILL_DIR" ] && SKILL_DIR="$HOME/.claude/skills/generate-testdoc"

# 通常ケース
python "$SKILL_DIR/scripts/generate_testdoc.py" \
  --kintone-id "<KintoneID>" \
  --title "<Title>" \
  --input "<TempDir>/testdoc_<KintoneID>.json"
```

### 6.3 ファイル分割（変更量が多いケース）

以下に該当する場合、JSON とタイトルを分けて複数ファイル出力：

- 変更が独立した機能エリアにまたがり、担当者が別々に実施
- 1 ファイルに 20 件大幅超のケースが集中
- インフラ変更と業務ロジック変更が混在

`--title` 末尾に識別情報を付加：

```bash
python "$SKILL_DIR/scripts/generate_testdoc.py" \
  --kintone-id "<KintoneID>" --title "<Title>（新機能）" \
  --input "<TempDir>/testdoc_<KintoneID>_1.json"

python "$SKILL_DIR/scripts/generate_testdoc.py" \
  --kintone-id "<KintoneID>" --title "<Title>（既存改修）" \
  --input "<TempDir>/testdoc_<KintoneID>_2.json"
```

→ `【KintoneID】 <Title>（新機能）.xlsx` / `（既存改修）.xlsx` で出力。識別情報は変更内容に合わせる（`（算定管理）`、`（印刷）`、`（DB移行）` 等）。

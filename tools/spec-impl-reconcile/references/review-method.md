# 突合の方法（実在ゲート・3レンズ・収束ゲート・見落観点）

SKILL.md の核心フロー詳細。**実装を唯一の事実**とし、推測でパス・行番号・シンボルを使わない。

## 1. 実在ゲート（最初に必ず通す）
仕様書が「ソース」として挙げるファイル/クラス/メソッド/関数を Glob/Grep で片端から実在確認する。

- **中核シンボル（本体クラス・保存/取得の中核メソッド・Service/Repository 等）が0件 → それ自体を最重要のハルシネーションとして即起票**（`分類B・深刻度高・対処 再抽出`）。
- 次に実体がどこにあるか（旧サーバー側処理・別クラス直呼び・UI直書き等）を特定し「帰属先の誤り」として記録する。
- 引用行番号・メソッド名も実ファイルで検証。系統的にズレていれば「行番号ベースの参照は信頼不可」と備考に明記し、シンボル名参照での再抽出を促す。
- **AI生成仕様書は、移行後アーキの空想（実在しない層を中核として描く）を犯しがち**。ここで早期検出する。
- **存在確認だけで止めない — 効果を主張する委譲先は本体を直読して裏取りする**：仕様書が委譲先（サーバー処理/クエリ/リポジトリ/関数）の**効果・副作用・戻り値・書込先を記述している場合、その主張は呼出側だけでは検証できない**。名指しされた本体を直読し、(a) 実際に書き込む全データ源（テーブル/エンティティ/ファイル） (b) 戻り値/出力が呼出側で使われるか を記述と突合する。仕様書の副作用・関連データに無い書込先＝欠落（A2/A3）、戻り値が使われない呼出＝デッドコール（C）。**これは仕様書が名指しした引用ソースの検証であり、リポジトリ横断探索ではない（§コスト規律の明示的例外）**。データを書き込む処理（挿入/更新/削除）は副作用の記載漏れが特に起きやすい。

## 2. 子画面・委譲先の実在確認（必須・事故多発点）
「この子画面は別仕様書で抽出済みだろう」という**未確認の思い込み**と、「渡された候補リストにある画面だけ見る」という**候補リスト依存**が二大漏れ要因。

- **★機械枚举が先（候補リストに依存しない）**：まず実装コードを grep して子画面/委譲先を**母集団ごと洗い出す**。検索対象は起動パターン：`new .*Form`／`new .*Dialog`／`ShowDialog`／`ShowModal`／`_Open` 系メソッド／`KickFunction`／`FormFactory.` 等。**呼出元や指示が事前に挙げた候補名だけを見て済ませない**——事前候補は必ず漏れる（実例: 予め列挙した候補にグリッドの主入力子画面が入っておらず、`_OpenReceSub`→`ReceSub` を丸ごと見落とした）。
- 枚举した各画面について**仕様書フォルダ内をクラス名（表示名でなく実クラス名）で Glob/Grep**。
  - 実在した → 差異一覧に**確認済みメモ**を1行残す（黙って省略しない）。
  - 実在しない → **A1／新規抽出**で起票。実画面の規模・主な入出力・外部連携有無を根拠に添える。
- **未確認の他担当範囲へ差異を丸投げしない**。その担当範囲が検証単位として実在するときのみ「〇〇で確認」と書いてよい。
- **分節並行で運用する場合の注意**：セクションを跨いで起動される子画面（グリッドから開く入力ダイアログ等）は分節の隙間に落ちやすい。子画面枚举レンズは1つのエージェントに機械枚举を必ず担わせ、他分節が触れた前提にしない。

## 3. 突合3レンズ
- **①子画面・委譲先の被覆**：「委譲/未確認/本体未定義」で止まっていないか、開いた先の入出力・副作用が語られているか、一覧内の委譲部品の実ロジックが追えるか。
- **②画面横断ロジック**：ライフサイクル（作成→送信→実施→中止→会計/在庫/帳票/外部連携）が途切れず追えるか、中止・変更の伝播先が明記か、サーバー側処理が反映されているか。**A2 の境界＝効果が本対象の外へ出るか**（Yes＝A2横断新設／No＝A3・C追補）。
- **③セクション網羅性**：一覧の列は列挙定義（enum/定数）と突合（主クエリの SELECT だけを根拠にしない）、設定/権限分岐・右クリック・エラーケースが代表例止まりでないか。

## 4. 見落観点集（一般原則。分節エージェントのチェックリストにも使う）

### 4.0 機械枚举母集団突合（最優先・構造カバレッジの土台）
分節並行（fan-out）の弱点は「分節の切り方・種子の偏りで透镜の隙間に落ちる」こと。インライン読解が"コードを辿るだけで"得ている構造カバレッジを、fan-out でも得るために、**機械で全量枚举できる母集団は必ず全量枚举して仕様書と双方向で突合する**（候補リスト・顕著性・他分節が見た前提に頼らない）。最低限の固定母集団は次の6つ：

| 母集団 | 枚举の仕方（grep） | 突合先（仕様書） | 見るもの |
|---|---|---|---|
| 子画面/委譲先 | `new .*Form`/`.*Dialog`・`ShowDialog`/`ShowModal`・`_Open`系・`KickFunction`・`FormFactory.` | §トリガー/§他フォーム/子画面別md | 開くだけで I/O・副作用が未記載＝A1（§2） |
| トリガー/イベント配線 | `.Click +=`・`DoubleClick +=`・`KeyDown +=` 等（designer含む） | §トリガー表 | 表に無い配線＝トリガー漏れ／配線無い表行＝死配線（両方向） |
| サーバーコントロール/データ源 | `Execute(typeof(SH00Karte_*/CommonQuery_*))`・直SQL | §外部連携表 | 表に無い呼出＝欠落（特に画面中心の主クエリ）／**効果・副作用・戻り値が記述された呼出は本体を直読して突合（副作用データ源の記載漏れ・デッドコール検出）** |
| 設定キー | `Setting*.*.Value`・設定参照 | §関連設定表 | 表に無いキー／補集合(false時)挙動が未記載 |
| 一覧の列 | `enum Col`/列挙定義 | §グリッド列網羅 | 列の過不足・供給元の相違（主SELECTだけを根拠にしない） |
| 多段分岐 | 多分岐構文（`switch`／`else if` 連鎖等）・同一変数への再代入 | 分岐が記述されたセクション（§関連設定・§出力等） | 既定・補集合分岐（`default`／`else`）の欠落＝無言スキップ／同一の出力変数を複数分岐で上書き＝優先順位の相違 |

**「表に無い実装」だけでなく「実装に無い表行」も両方向で見る**（死配線・死UIの検出）。

### 4.1 経路差・観点別（深掘りレンズ）
- **新規/更新/取消の経路差**：同一動作（受付・保存・送信等）が入口や状態（新規 KarteID=-1 / 既存更新 / 取消）で発火する処理が違わないか。片方の経路でしか呼ばれない副作用・連携を明示（実例: 健診受付/検査送信/放射線通知が受付更新経路のみ、新規受付経路では未発火）。
- **マージの勝敗規則**：統合・名寄せで「どちらの値が残るか」を追う。高リスク値の勝敗は深刻度高。
- **無言スキップ分岐**：存在チェック false 時に空ブロックで黙ってスキップ（例外もログもなし）。仕様に無ければ起票。**呼出元が指定した高リスク値（§高リスク領域。識別子・キー等）の確定パスで、条件不成立時に前値が据え置かれ上書き/クリアされない場合は、到達性が不確実でも要確認(E)または欠落(A3)で起票し、黙って落とさない**（据え置かれた識別子が後続処理に使われれば取り違え等の高リスク）。
- **委譲先の本体まで読んで効果を裏取りする**：仕様書が委譲先（サーバー処理/クエリ/リポジトリ/関数）の**効果・副作用・戻り値・書込先を記述している場合、その主張は呼出側だけでは検証できない**。名指しされた本体を直読し記述と突合する。特にデータを書き込む処理（挿入/更新/削除）は副作用データ源の記載漏れ、戻り値が使われない呼出は実質デッドコールを生みやすい（外部送信の購読側(subscriber)読解＝「通知」と書かれて実際はキー書換え・中止/件数ガードが購読側にある、はこの一例）。
- **UI状態の成功/失敗差**：finally で無条件にボタン状態が変わる、worker 完了の失敗分岐が無通知 等。
- **削除/付替対象の実体列挙 vs ワイルドカード**（不可逆操作は実体一致が必須）。
- **呼び出し元の逆引き**：機能固有シンボルで grep し、(a)トリガー表に無い起動経路 (b)呼出元ゼロ＝死にコード を両方向で確認。
- **多経路機能は「経路 × ガード/エラー処理」対比表**で突合（権限・閲覧制限・デバイス不在・成功/失敗通知・ログ）。
- **検索・抽出条件の AND/OR 結合規則**を実クエリで確定（一括操作の実効も）。
- **条件分岐は補集合側（else で何が出るか）**を確定する。
- **同一業務概念の多重定義突合**（フィルタSQL・バッジ判定・表示列・ラベルが別実装なら定義一致を確認）。

## 5. アドバサリアル収束ゲート（出力前・必須）
最終セットの全指摘に、変化が出なくなるまで反証を反復（loop-until-stable）。各巡で `維持／取下げ／深刻度・粒度調整／文言・ID修正` を適用し、**1巡まわして何も変わらなければ収束＝出力可**。

反証の観点：
1. 本当にコードでそうか（`file:line` を再確認）。
2. 設計書が既に正確に記載していないか（既記載なら差ではない。E に上げ直さない）。
3. 過大主張でないか（条件やコードパスが存在するだけで断定せず、実効経路を確認）。
4. 採番し直し後の相互参照・ID が解決するか（stale 参照）。

安全弁：4巡しても収束しない項目は無理に閉じず、揺れている指摘を明示して人へ相談する。

**コスト規律**：反証をサブエージェントに fan-out する場合は、環境で最も安い reasoning 可能な層を**明示指定**し（メイン高単価モデルの継承を防ぐ）、可読範囲を該当指摘の根拠ファイルへ限定する（横断探索禁止）。この降級は確認作業ゆえ recall に効かず、最終判断はメイン側が持つ。発見型 fan-out（§2 分節並行）は降級しない。→ subagent-prompt.md「モデル・コスト規律」

## 5.5 完整性クリティック（出力前・分節の隙間を塞ぐ）
収束ゲートの直前に、**「何を突合していないか」を能動的に自問**する（fan-out の構造盲点対策）。次を1件ずつ確認し、埋めるか④の「網羅性の残リスク」に明示する（silent cap 禁止）：
- §4.0 の6母集団それぞれについて、**全量枚举 → 仕様書と突合した**か（未実施なら、その母集団は未カバーと明示）。
- 分節をまたいで起動される要素（グリッドから開く主入力ダイアログ等）が、どの分節の担当でもなく素通りしていないか。
- 引用ソース（子画面本体・Service実体・サーバーコントロール内部SQL等）で未読のまま残したものはないか。**効果・副作用・戻り値を仕様書が主張している委譲先は、④の残リスクに逃さず必ず本体を読んで突合する**（未読のまま残リスクへ回してよいのは効果が記述されていない委譲先のみ）。
- 「表に無い実装」と「実装に無い表行」の**両方向**を各母集団で見たか。

## 6. スコープ厳守
起票は「記載漏れ・相違・要確認」のみ。現行挙動が正しく記載されていれば、設計・UX改善提案は対象外。設計書が既にバグ候補・既知事項として正確に記載しているものは E に上げ直さず、確認済みメモに留める。

# 出力フォーマット（①②③④・format パラメータ・機械チェック）

毎回この順で出力する。**表示用（②④＝人への報告）は丁寧語、台帳セル（②の指摘内容・備考／③）は常体**で書く。

## ① 結論（短く）
対象の一言説明 ＋ 突合結果（コードとの一致度・ハルシネーション有無）＋ 件数内訳（要対応=高/中・軽微=低）。

## ② 1指摘＝1ブロック
並び順＝**深刻度順（高→中→低）、同順位内は分類順（A1→A2→A3→B→C→D→E）**。上から `R-{対象名}-01` から採番（`-01` が最重要）。

```
### R-{対象名}-NN ｜ 分類 X・深刻度 Y・確度 Z・対処 W
- 指摘箇所：〈設計書の実見出し §名〉
- 指摘内容：
〈要点：問題＋影響を平易な言葉で1〜2文〉
                        ← 空白行を1行
〈根拠：コードで確認した証拠。現行はこう／仕様書はこう を改行で。行番号は「47行目」等プレーンに〉
- 備考：〈次アクション／移植観点〉
- 参照（クリック用・1行1リンク・ブロック末尾）：
〈`絶対パス:行` を1行1つ〉
```

- **参照リンクは散文に混ぜず、ブロック末尾に1行1リンク**（`絶対パス:行`）。
- **確度**（Confirmed/Inferred/Gap）をブロック頭に必ず表示。Inferred/Gap は根拠側に「なぜ直証できないか」を書く。
- **トーン：チェックは厳しく、伝え方はやさしく**。造語・専門用語はその場でかみくだく。
- **対象を数量で丸めない**（「3グリッド」ではなく対象を名指し。必要なら部品名/クラス名を括弧添え）。

## ③ 転記用（`|` 区切り・9列）
列順（**先頭・末尾にパイプを付けない。パイプ本数＝8本**）:

```
課題ID | 対象(クラス/画面) | 指摘箇所 | 指摘内容 | 分類 | 深刻度 | 確度 | 対処区分 | 備考
```

- 区切り＝**半角パイプ `|`**。本文（指摘内容・備考）に `|` を含めない（論理OR `||` は「または」へ言い換え）。
- 指摘内容は1セル内「要点。根拠。」の1行。**②の要点＋根拠と同一文言**（改行の有無だけが差）。
- 分類/深刻度/対処区分/状態など台帳の下拉許可値はプロジェクト依存。運用先の許可値に合わせる。

### 指摘箇所の許可値（既定＝Phase2 仕様書テンプレート。台帳プルダウンと完全一致させる）
`指摘箇所` は**下の一覧の値ちょうど1つ**にする（複数該当でも主セクション1つ。`§` プレフィックスや複数併記 `A / B` はしない）。**md に別見出しがあってもこの一覧から選ぶ**。

`機能名` / `概要` / `想定ユースケース` / `トリガー` / `右クリックメニュー` / `入力` / `出力` / `ツールチップ・UI表示挙動` / `関連設定と画面制御` / `例外・エラーケース` / `トランザクション境界` / `外部連携` / `非同期処理` / `ライフサイクル副作用` / `バグ候補` / `UIロジック混入箇所` / `E2Eテスト候補` / `（全体）` / `（仕様書なし）` / `（セクションなし）`

- **md 見出しのかっこ書き補足は必ず削って完全一致**にする（例: `外部連携（DB・他クラス・他フォーム）`→`外部連携`／`例外・エラーケース（MessageBox 表示を含む）`→`例外・エラーケース`／`非同期処理（BackgroundWorker / Invoke / スレッド）`→`非同期処理`／`想定ユースケース（業務シナリオ単位）`→`想定ユースケース`／`ライフサイクル副作用（Load / Closing）`→`ライフサイクル副作用`）。
- 一覧に無い見出し（`関連テーブル`／`前提条件`／`グリッド列網羅`／DDL 等）は**最も近い正式セクションへ寄せる**（前提条件→`入力`、関連テーブル/DDL→`トランザクション境界` or `外部連携`、グリッド列網羅→`出力`）。
- **領域非依存の運用**：上の一覧は mall-cloud Phase2 テンプレートのセクション集合＝この工程の既定許可値。別テンプレートの仕様書に使う場合は、そのテンプレートのセクション名一覧に差し替える（`--sections` で機械チェックにも渡せる。下記）。

## ④ 人による最終確認ポイント（③の後・丁寧語）
1. **全指摘の人対応表**（全件）：列＝`ID｜深刻度｜確度｜人の対応`。④だけ見れば取りこぼしに気づけるようにする。
2. **観点別の申し送り**（該当あるもののみ、各項目に1行要約＋参照）：
   - **裏取り必須**：高・高リスクの指摘の根拠 `file:line`。
   - **エスカレ判断**：高リスク領域の新規指摘を有識者へ回す候補。
   - **Gap の阻断性区分**：Gap（要確認）を **阻断性Gap（解決しないと安全に再実装できない）** と **非阻断Gap（装飾的・範囲外）** に分ける。移植優先度に直結。
   - **網羅性の残リスク**：未読の引用ソース・打ち切った探索を明示（silent cap を作らない）。

## format パラメータ（md / excel / both・既定 md）
- **md**：①〜④ の全レポートを `<出力先>\<対象名>_突合レポート.md` に書く。
- **excel**：③ のパイプ表を `../scripts/pipe_to_xlsx.py` へ渡して **真の .xlsx** を生成する。
  ```
  python scripts/pipe_to_xlsx.py <③を書き出したファイル> --name <対象名>   # 既定 %USERPROFILE%\Documents\<対象名>\<対象名>_issues.xlsx
  python scripts/pipe_to_xlsx.py <③を書き出したファイル> <出力先ベース>       # 明示パス（拡張子なし）
  ```
  スクリプトが機械チェック（下記）→ 出力先解決 → 既存を `old\` へ退避 → 生成まで行う。`--name` 指定時は既定の `%USERPROFILE%\Documents\<対象名>\` を自動作成して書く。`openpyxl` が無ければ **UTF-8 BOM 付きクォート CSV** へ自動降級し警告する（`pip install openpyxl` か PowerShell `ImportExcel` を促す）。
- **both**：md と excel を両方出す（md レポートも同じ `%USERPROFILE%\Documents\<対象名>\` へ）。
- 既定出力先は `%USERPROFILE%\Documents\<対象名>\`。スクリプトが既存の同名ファイルを `old\`（衝突時は連番）へ退避してから上書きする（黙って消さない）。

## 出力前の機械チェック（③確定後に必ず実行）
**③ は `format` の値に関わらず、確定前に必ず `scripts/pipe_to_xlsx.py` へ通して検査する**（`format=md` で xlsx を作らない場合も、③をスクラッチファイルに書き出してスクリプトに掛ける）。**「機械チェックを満たすよう記述した」という自己申告で済ませてはならない**——スクリプトが exit 0 を返して初めて③を確定できる。exit 2（構造エラー）なら③を直して再実行する。検査項目：
- (a) 各行のパイプ本数＝列数−1（9列なら8本）。ヘッダ・データ行すべて同数 → 不一致は構造エラーで停止。
- (b) 先頭・末尾パイプなし → あれば構造エラーで停止。
- (c) 本文に `|` が無い（論理OR `||` 等は「または」へ言い換え。②③を書く時点で担保）。
- (d) 非日本語スクリプト文字（キリル/ギリシャ/ハングル等）検出 → 警告（μ 等の正当な記号は人が確認）。
- (e) **`指摘箇所` 列が許可値一覧と完全一致**か（`--sections "機能名,概要,…"` を渡した時のみ照合）。一覧外は警告（かっこ書き残り・§付き・複数併記・寄せ漏れの検出）。既定の Phase2 テンプレート一覧を渡す運用にすると台帳プルダウンと必ず一致する。

スクリプトは `|` 区切りをトリムしてセル値化し、CSV 降級時はカンマ・改行を含む値を自動クォートする。

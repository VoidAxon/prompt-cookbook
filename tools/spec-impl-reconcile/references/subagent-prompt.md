# 分節サブエージェント・プロンプト雛形（汎用）

検証単位（タブ/セクション/機能単位）ごとに `general-purpose` サブエージェントを並行起動する。独立領域を1エージェント＝1担当に割り当て、各自が結果を Write し要点だけ返す（メインの文脈を節約）。ツール未使用で異常終了したら再起動する（成果物の存在/更新時刻で確認）。

## モデル・コスト規律（発見型と確認型で分ける）
コストの最大の浪費源は**子エージェントのリポジトリ横断探索**と**高単価モデルの継承**。次の3点で抑える（判断の最終責任は常にメイン側の収束ゲートが持つ）:
- **読み取りホワイトリスト**: 各エージェントには**メイン側の実在ゲートで確認済みの小さなファイル一覧だけ**を可読範囲として渡し、**リポジトリ横断の Glob/Grep/Explore を禁止**する（例外は特定シンボルの呼出元 grep を数回まで）。recall に影響しないので発見型・確認型の両方へ適用。
- **事前シード**: メイン側の実在ゲート結果・確定事実（「Xは呼出元ゼロ」「中核実体は旧サーバー側」等）・担当セクションの設計書該当記述をプロンプトへ埋め込み、「発見済みの再発見」を消す。両方へ適用。
- **モデルの使い分け**: **発見型 fan-out（分節の主発見）は能力のある層のまま**にする——安価モデルは発見レンズの recall が落ちるため降級しない。**降級してよいのは §5 収束ゲートの対抗/確認 pass だけ**（ホワイトリスト内を読んで突合する機械的作業で、最終判断はメイン側）。降級先は「環境で最も安い reasoning 可能な層」を**明示指定**する（無指定だとメインの高単価モデルを継承する）。

**レンズ配分（構造盲点を作らないため）**: 機能をセクションで切るだけだと透镜の隙間に落ちる。以下を**必ず誰かに割り当てる**——(A) 機械枚举母集団突合の6母集団（子画面/トリガー配線/サーバーコントロール・データ源/設定キー/一覧の列/多段分岐。§4.0）を担う「構造カバレッジ担当」を1つ。ここには**効果・副作用・戻り値が記述された委譲先本体の直読突合（副作用データ源の記載漏れ・デッドコール検出）**を含める。(B) 深掘りレンズ（トランザクション/採番・排他、委譲先本体まで読む効果裏取り（外部連携subscriber側を含む）、設定の補集合(else)、経路差）を機能の重さに応じて配分。子画面レンズは (A) が機械枚举するので、他分節が見た前提にしない。最後にメイン側で §5.5 完整性クリティックを回して隙間を塞ぐ。

```
## 読み取り範囲（コスト規律・厳守）
- 可読ファイルは上の {対象} に挙げた**検証済みファイルに限る**。**リポジトリ横断の Glob/Grep/Explore は禁止**（最大の浪費源）。
- 例外: 呼出元の逆引き等、**特定シンボル名に限った grep を数回**まで。子画面の起動パターン機械枚举は担当に含まれる場合のみ実施する。
- 例外（追加）: 仕様書が名指しし、かつ**効果・副作用・戻り値を記述している委譲先（サーバー処理/クエリ/リポジトリ/関数）の本体ファイルは直読してよい**（書込先データ源・戻り値の使われ方は本体を読まないと確認不能。横断探索ではなく引用ソースの検証）。読むのは名指しされた本体に限り、そこから芋づる式に他ファイルを開かない。

## 実在ゲート（最初に）
設計書が挙げる核心シンボル（ファイル/クラス/メソッド）を実在確認。中核が0件なら B・高・再抽出 で起票し、真の実体を「帰属先の誤り」として記録。

## 3レンズ
- A1 子画面/委譲先: 「開く」で終わり入出力・副作用が抜けていないか。委譲スタブを「未確認|委譲|本体未定義」で機械検出。★**まず起動パターンを grep で機械枚举**（`new .*Form`/`new .*Dialog`/`ShowDialog`/`ShowModal`/`_Open`系/`KickFunction`/`FormFactory.`）して子画面の母集団を確定する。**このプロンプトが例示した候補名だけを見て済ませない**——例示は不完全で、挙がっていない主入力子画面（グリッドから開くダイアログ等）を丸ごと見落とす。枚举した各子画面を仕様書フォルダでクラス名 Glob/Grep して別途抽出の有無を実在確認。実在=確認済みメモ1行、非実在=A1新規抽出。未確認の他担当へ丸投げ禁止。粒度は「画面まるごと1件」（内部機能を個別行に分解しない。ただし他システム/他画面への横断=A2、有識者確認要=E、親側=A3/B は別行）。
- A2 横断ロジック: 効果が本対象の外へ波及する処理（別画面/別システム/帳票/監査ログ/在庫・会計）が追えるか。サーバー側処理が反映されているか。
- A3 分岐/網羅: 一覧の列は列挙定義と突合（主クエリSELECTだけを根拠にしない）。設定/権限分岐・エラーケースが代表例止まりでないか。補集合側（else）も確定。

## 機械枚举母集団突合（最優先）
候補リスト・顕著性に頼らず、機械で全量枚举できる母集団を grep で洗い出し仕様書と双方向突合する（表に無い実装＝欠落／実装に無い表行＝死配線・死UI）: 子画面(`new .*Form`/`ShowDialog`/`ShowModal`/`_Open`系/`KickFunction`/`FormFactory.`) / トリガー配線(`.Click +=`/`DoubleClick +=` designer含む) / サーバーコントロール・データ源(`Execute(typeof(...))`/直SQL。**効果・副作用・戻り値が記述された呼出は本体を直読して書込先データ源・戻り値使用まで突合＝副作用データ源の記載漏れ・デッドコール検出**) / 設定キー(`Setting*.Value`) / 一覧の列(`enum Col`) / 多段分岐(`switch`/`else if`連鎖・同一変数への再代入＝`default`/`else`欠落の無言スキップ・複数分岐の変数上書き優先順位)。

## 見落観点（該当を当てる）
新規/更新/取消の経路差 / マージ勝敗規則 / 無言スキップ / 外部送信は購読側まで / 成功失敗のUI差 / 削除対象の実体列挙 / 呼出元逆引き / 多経路×ガード対比 / 検索条件のAND-OR / 同一概念の多重定義。

## 高リスク領域
{この機能の高リスク点。例: 医療=患者識別/薬剤/検査値, 金融=金額/口座}。該当は深刻度を上げ、理由を一文添える。**指摘内容に【高リスク】タグは付けない**（深刻度=高で表す）。高リスク該当かは出力表の `高リスク` 列（④エスカレ判定用）で保持する。

## 報告しない
実装バグ摘出そのもの / コード品質・規約違反そのもの / 行番号単純ズレ（末尾1行集約）。設計書が既に正確に記載済みの点は E に上げ直さない。

## 書き方（必須）
非開発者が読んで分かる平易な日本語で「何が / どう違うか(漏れているか) / なぜ問題か(影響)」。記号・略記・クラス名の羅列にしない。対象を数量で丸めない（名指しする）。`file:line` は根拠として併記。

## 成果物（必ず Write）
{出力先}\{NN_セクション名}.md に、1差異1行の表で出力（列: No｜分類｜指摘内容｜高リスク｜深刻度｜対処区分｜該当箇所(§/file:line)｜設計書の記述｜実コードの実際｜影響）。深刻度 高→中→低 順。差異が無ければ「差異なし」。Write 後、重要差異を最大3点で要約して返す。
```

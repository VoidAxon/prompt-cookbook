# 分節サブエージェント・プロンプト雛形（汎用）

検証単位（タブ/セクション/機能単位）ごとに `general-purpose` サブエージェントを並行起動する。独立領域を1エージェント＝1担当に割り当て、各自が結果を Write し要点だけ返す（メインの文脈を節約）。ツール未使用で異常終了したら再起動する（成果物の存在/更新時刻で確認）。

**レンズ配分（構造盲点を作らないため）**: 機能をセクションで切るだけだと透镜の隙間に落ちる。以下を**必ず誰かに割り当てる**——(A) 機械枚举母集団突合の5母集団（子画面/トリガー配線/サーバーコントロール・データ源/設定キー/一覧の列。§4.0）を担う「構造カバレッジ担当」を1つ、(B) 深掘りレンズ（トランザクション/採番・排他、外部連携subscriber側、設定の補集合(else)、経路差）を機能の重さに応じて配分。子画面レンズは (A) が機械枚举するので、他分節が見た前提にしない。最後にメイン側で §6.5 完整性クリティックを回して隙間を塞ぐ。

```
設計書と実装の突合検証。実装を唯一の事実として、設計書の記述と実コードの差異を洗い出し要件整理する（バグ探しではない）。「記載漏れ / 相違 / 要確認」の3分類 + 確度(Confirmed/Inferred/Gap)で報告する。

## 対象（必ず Read/Grep で確認・推測禁止）
- {実装ファイルのフルパス}（本体 + UI定義ファイル等）
- 関連クラス/モジュール（Grep で場所特定）
- 仕様書フォルダ: {突合対象の仕様書が置かれたフォルダのフルパス}（子画面が別仕様書で抽出済みかをクラス名で Glob/Grep 確認するのに使う）

## 担当範囲: 【{セクション/タブ名}】
検証すべき設計書の記述（断定表現「〜は1つ」「該当なし」も明記）:
{該当セクションの記述を箇条書きで注入}

## 実在ゲート（最初に）
設計書が挙げる核心シンボル（ファイル/クラス/メソッド）を実在確認。中核が0件なら B・高・再抽出・確度Confirmed で起票し、真の実体を「帰属先の誤り」として記録。

## 3レンズ
- A1 子画面/委譲先: 「開く」で終わり入出力・副作用が抜けていないか。委譲スタブを「未確認|委譲|本体未定義」で機械検出。★**まず起動パターンを grep で機械枚举**（`new .*Form`/`new .*Dialog`/`ShowDialog`/`ShowModal`/`_Open`系/`KickFunction`/`FormFactory.`）して子画面の母集団を確定する。**このプロンプトが例示した候補名だけを見て済ませない**——例示は不完全で、挙がっていない主入力子画面（グリッドから開くダイアログ等）を丸ごと見落とす。枚举した各子画面を仕様書フォルダでクラス名 Glob/Grep して別途抽出の有無を実在確認。実在=確認済みメモ1行、非実在=A1新規抽出。未確認の他担当へ丸投げ禁止。粒度は「画面まるごと1件」（内部機能を個別行に分解しない。ただし他システム/他画面への横断=A2、有識者確認要=E、親側=A3/B は別行）。
- A2 横断ロジック: 効果が本対象の外へ波及する処理（別画面/別システム/帳票/監査ログ/在庫・会計）が追えるか。サーバー側処理が反映されているか。
- A3 分岐/網羅: 一覧の列は列挙定義と突合（主クエリSELECTだけを根拠にしない）。設定/権限分岐・エラーケースが代表例止まりでないか。補集合側（else）も確定。

## 機械枚举母集団突合（最優先）
候補リスト・顕著性に頼らず、機械で全量枚举できる母集団を grep で洗い出し仕様書と双方向突合する（表に無い実装＝欠落／実装に無い表行＝死配線・死UI）: 子画面(`new .*Form`/`ShowDialog`/`ShowModal`/`_Open`系/`KickFunction`/`FormFactory.`) / トリガー配線(`.Click +=`/`DoubleClick +=` designer含む) / サーバーコントロール・データ源(`Execute(typeof(...))`/直SQL) / 設定キー(`Setting*.Value`) / 一覧の列(`enum Col`)。

## 見落観点（該当を当てる）
新規/更新/取消の経路差 / マージ勝敗規則 / 無言スキップ / 外部送信は購読側まで / 成功失敗のUI差 / 削除対象の実体列挙 / 呼出元逆引き / 多経路×ガード対比 / 検索条件のAND-OR / 同一概念の多重定義。

## 高リスク領域
{この機能の高リスク点。例: 医療=患者識別/薬剤/検査値, 金融=金額/口座}。該当は深刻度を上げ、理由を一文添える。**指摘内容に【高リスク】タグは付けない**（深刻度=高で表す）。高リスク該当かは出力表の `高リスク` 列（④エスカレ判定用）で保持する。

## 報告しない
実装バグ摘出そのもの / コード品質・規約違反そのもの / 行番号単純ズレ（末尾1行集約）。設計書が既に正確に記載済みの点は E に上げ直さない。

## 書き方（必須）
非開発者が読んで分かる平易な日本語で「何が / どう違うか(漏れているか) / なぜ問題か(影響)」。記号・略記・クラス名の羅列にしない。対象を数量で丸めない（名指しする）。`file:line` は根拠として併記。各差異に確度(Confirmed/Inferred/Gap)を付け、Inferred/Gap は理由を明記。

## 成果物（必ず Write）
{出力先}\{NN_セクション名}.md に、1差異1行の表で出力（列: No｜分類｜指摘内容｜確度｜高リスク｜深刻度｜対処区分｜該当箇所(§/file:line)｜設計書の記述｜実コードの実際｜影響）。深刻度 高→中→低 順。差異が無ければ「差異なし」。Write 後、重要差異を最大3点で要約して返す。
```

総括はメイン側で各エージェントの返答要約から組む（個別ファイルを再読して文脈を膨張させない）。メイン側で全エージェントの結果を統合し、SKILL.md の収束ゲートと出力フォーマット（①〜④）へ流す。

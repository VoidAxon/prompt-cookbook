# 設計書 × 実装 突合検証

## 目的
**実装を唯一の事実**として、設計書の記述と実装コードの**差異**を洗い出し、移植・テスト化に足る要件を整理する。バグ探しではなく要件整理が主眼。

- **read-only**：設計書・実装・成果物を編集しない。出力は差異レポートのみ。
- **領域・言語非依存**：本書の例（C#/医療 等）は例示。任意の設計書×実装に適用する。「医療安全」は後述「高リスク領域」の一実例。
- 報告する差異は3分類（**記載漏れ / 相違 / 要確認**）。各差異に**確度**（Confirmed/Inferred/Gap）を付ける。

## 使わない場面
- 実装バグの摘出そのもの（null 参照・境界値・トランザクション欠落等の不具合探し）→ QA/デバッグの領分。実装が明らかに仕様と食い違えば「相違」、仕様が沈黙なら「記載漏れ」として拾うにとどめる。
- コード品質・規約違反そのもの。
- 行番号の単純ズレ → 個別列挙せず末尾に「全体で約N行ズレ」1行のみ（別メソッドを指す誤参照は「相違」）。

## 核心フロー
1. **特定 + 実在ゲート** — 仕様書と実コードを Glob/Grep で特定（推測禁止）。仕様書が挙げる核心シンボル（ファイル/クラス/メソッド）を実在確認。**中核が0件＝ハルシネーション、即 `B・高・再抽出` で起票**し、真の実体を特定して「帰属先の誤り」として記録。→ [references/review-method.md](references/review-method.md)
2. **分節並行検証** — 検証単位（タブ/セクション）ごとに `general-purpose` サブエージェントを並行起動。各自が結果を Write し要点を返す。→ [references/subagent-prompt.md](references/subagent-prompt.md)
3. **機械枚举母集団突合 + 突合3レンズ + 高リスク領域** — まず機械で全量枚举できる5母集団（子画面/トリガー配線/サーバーコントロール・データ源/設定キー/一覧の列）を grep で洗い出し仕様書と双方向突合（fan-out の構造盲点対策・候補リストに頼らない）。加えて3レンズ（①子画面/委譲先 ②画面横断 ③分岐/網羅）＋経路差＋見落観点集。→ [references/review-method.md](references/review-method.md)
4. **完整性クリティック → アドバサリアル収束ゲート（出力前・必須）** — 先に「5母集団を全量突合したか・分節の隙間に落ちた要素・未読引用ソース」を自問して埋める（silent cap 禁止）。次に全指摘を反証（コード再確認/既知重複/過大主張/stale参照）し、変化が出なくなるまで反復（loop-until-stable）。→ [references/review-method.md](references/review-method.md)
5. **出力** — ①結論 ②1指摘=1ブロック ③転記用 ④人確認段。`format` = `md`（既定）/`excel`/`both`。excel は `scripts/pipe_to_xlsx.py` が③を機械チェック→真xlsx化（openpyxl無ければCSV降級）。→ [references/output-format.md](references/output-format.md)

## 分類・確度クイックリファレンス

**指摘分類コード**（`分類`列。**A1/A2 が主眼**）

| コード | 名称 | 意味 |
|---|---|---|
| A1 | 欠落・子画面 | 委譲先/子画面/ダイアログの仕様が未抽出、または「委譲/未確認」で止まっている |
| A2 | 欠落・横断ロジック | 効果が本対象の外（別画面/別システム/会計・在庫・帳票・監査ログ等）へ波及する処理が断片でしか書かれていない |
| A3 | 欠落・分岐/網羅 | セクションはあるが列挙が代表例止まり（設定/権限分岐・一覧の列・エラーケース等） |
| B | 記載誤り | 仕様記述が実コードと不一致（区分値・既定値・呼出先・メッセージ・誤参照等） |
| C | 粒度不足 | 記載はあるがテスト化に足る具体度がない |
| D | 様式違反 | テンプレ様式・必須セクション・ID書式の違反 |
| E | 要確認 | コードだけでは仕様か歴史的挙動か判断できず、有識者確認が要る |

**確度**（各指摘に必須。Inferred/Gap は「なぜ直証できないか」を明記）

| 確度 | 意味 |
|---|---|
| Confirmed | 実コードで直接確認済み（根拠 `file:line` を開いた） |
| Inferred | 命名/パターン/構造からの推断。未直証 |
| Gap | コードだけでは確定不可＝要確認（E と対応。④で阻断性を判定） |

**深刻度**：高（高リスク領域 or 移植時の機能欠落・挙動差）/ 中（テスト網羅・抽出完全性に影響）/ 低（様式・軽微）
**対処区分**：A1→新規抽出 / A2→横断新設 / A3・C→追補 / B→修正 / D→再抽出or追補 / E→確認

## 高リスク領域（旧「医療安全」の一般化）
呼出元がその領域の高リスク点を指定する（**例**：医療＝患者識別/薬剤量/検査値/外部送信、金融＝金額/口座/与信）。該当する差異は**深刻度を上げ**、④で有識者エスカレ候補にする。**指摘内容に `【高リスク】` タグは付けない**——深刻度=高で表現する（深刻度列と重複させない）。高リスク該当かは分節中間表の `高リスク` 列（④エスカレ判定用の内部フラグ）で保持する。

## Common Mistakes
- 推測でパスを組む → 必ず Glob/Grep で実在確認。
- バグ探しに逸れる → 要件整理であってデバッグではない。
- 実在ゲートを飛ばす → AI生成仕様書は実在しない中核構造を書きがち。最初に必ず通す。
- 子画面を「別で抽出済みだろう」と実在確認せず素通り／未確認の他担当へ丸投げ → [references/review-method.md](references/review-method.md) の必須ルール参照。
- 1パスで転記に回す → 収束ゲートを通すまで出力しない。
- 対象を数量で丸める（「3グリッド」等）→ 名指しする。詳細は [references/output-format.md](references/output-format.md)。
- **③ の機械チェックを自己申告で済ませる** → 必ず `scripts/pipe_to_xlsx.py` を実行し exit 0 を確認してから確定する（末尾パイプ・列数ズレが自己申告では見逃される。テストで実証済み）。

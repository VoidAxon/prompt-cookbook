# 仕様書 × 実装 突合検証

## このスキルの目的

**実装を唯一の事実**として、「機能仕様書（設計書）の記述」と「現行実装コード」の**差異**を洗い出し、移植・テスト化に足る**要件を整理する**。バグ探しではなく需要（仕様）の整理が主眼。報告する分類は次の3種。

1. **記載漏れ** — 実コードにあるが仕様書に無い、業務上意味のある処理・分岐・挙動・区分（「該当なし/空実装/〜は無い/〜は1つ」等の**断定が実コードで不成立**な断言型漏れを含む）。抽出仕様は次の構造的弱点で漏れやすいので、そこを重点的に見る。報告時は各記載漏れに ①/②/③ を付記すると集計しやすい。
   - **① 子画面・委譲先の漏れ** — 代表画面（親フォーム）中心の抽出のため、そこから呼ばれる子画面・ダイアログ・委譲先コントロール（ucXxx 等）の仕様が薄い/「委譲」「未確認」で止まっている。**子画面を「別仕様書で抽出済みだから対象外」と判断してよいのは、その仕様書ファイルが実在することを出力フォルダ内で確認できたときだけ**（後述「別仕様書カバレッジの実在確認」）。「開くだけ書いてある」子画面を、実在確認せずに黙って素通りしない。
   - **② 画面横断ロジックの漏れ** — 画面起点の抽出のため、画面をまたぐフロー（オーダーの作成→送信→実施→中止の伝播、会計・在庫・部門・外部連携・印刷への波及、サーバー側処理＝サーバーコントロール/バックエンド）が断片でしか書かれていない。
   - **③ 分岐/網羅の漏れ** — セクションはあるが列挙が代表例止まり（設定キー分岐・権限分岐・グリッド列（enum Col）・右クリック項目・エラーケース等）。
2. **相違** — 仕様書の記述が実コードと食い違う（処理内容・条件分岐・区分値/enum・既定値・DB制御名・設定キー・呼出先・メッセージ・参照先/別タブを指す誤参照・UI要素名 等）
3. **解釈が割れる箇所** — 仕様書の記述が曖昧で複数解釈でき、実装挙動が一意に読み取れない。**コードだけでは仕様か歴史的バグか判断できず、業務有識者・原開発者の確認が要る**ケースもここに含め、「要確認」として明示する。

**スコープ外（報告しない）**:
- **実装バグの摘出そのもの** — null 参照・年跨ぎ・境界値・数値精度・トランザクション欠落等の「不具合探し」はこのスキルの目的ではない（QA スキル／`superpowers:systematic-debugging` の領分）。ただし実装挙動が明らかに仕様と食い違う場合は「相違」、仕様が沈黙している場合は「記載漏れ」として、要件整理の観点で拾う。
- コード品質・規約違反そのもの（空catch/SQLi/カルチャ依存等）→ 純粋な品質指摘は QA スキルの領分
- 行番号の単純ズレ → 個別列挙せず末尾に「全体で約N行ズレ」1行のみ（別タブ/別メソッドを指す誤参照は「相違」として報告）
- 瑣末で実務上無関係なもの（表記ゆれ・軽微な命名・コメント差）
- **設計書が既に正確に記載している内容（バグ候補・既知事項として記載しているものを含む）** — 設計書の記述が実装挙動と一致しているなら、そこに突合上の差は無い。設計書が実装のバグを「バグ候補」「要検証」等として既に正確に書いている場合、それは**設計書と実装が一致**しているのであって「相違」でも「記載漏れ」でもない。**E 要確認として上げ直さない**（設計書が既に確認を促しているものを突合で重ねて確認要求するのは重複、かつバグ修正要否の判断は上記「実装バグの摘出」と同じくスコープ外）。残す価値がある場合は差異一覧に **`（差異なし）`／確認済メモ**として1行だけ置き、「設計書§で既記載済み・突合上の新規差異ではない」と明記する（黙って落とさない）。**判定の目安**: 「設計書の記述はこの点で実装と食い違っているか？」→ No（設計書が正しく書けている）なら差異ではない。E に上げてよいのは、設計書が沈黙 or 曖昧で**実装挙動が仕様か歴史的バグか設計書からは判断できない**ときに限る。

> スコープを狭めたい/広げたい指示（例「相違だけ」「漏れは不要」）があれば、その指示を優先する。

## 指摘分類コード（`分類` 列に必ず付与する）

上記の「記載漏れ①②③／相違／解釈割れ・要確認」は、本レビューの正式な**指摘分類コード**に対応する。`分類` 列にはこのコードを用いる（A1/A2 が本レビューの主眼＝背景①②に対応）。

| コード | 名称 | 説明 | 旧表記との対応 |
|-------|------|------|--------------|
| **A1** | 欠落・子画面 | 代表画面から呼ばれる子画面・ダイアログ・委譲先コントロールの仕様が未抽出、または「委譲」「未確認」で止まっている。例: 右クリックメニューが「ucGrid に委譲（本体未定義）」のまま委譲先の仕様書が存在しない | 記載漏れ① |
| **A2** | 欠落・横断ロジック | 画面をまたぐ処理フロー・サーバーコントロール側ロジックが画面単位の断片でしか記載されず、全体像が追えない。例: オーダー中止の会計・在庫・電子処方箋への伝播が中止画面側に記載なし | 記載漏れ② |

> **「横断（A2）」の境界 — 効果が本フォームの外へ出るか**: A2 は「効果が**本フォーム（本画面クラス）の外**へ波及する」ものに限る。波及先が **別フォーム／別サブシステム／外部システム／会計・在庫・帳票・監査ログ** など、画面単位の抽出では追えなくなる境界を越える場合が A2＝横断新設。一方、**同一フォーム内のタブ間表示同期・コントロール連動**（例: あるタブでの CRUD 後、同フォームの別タブのラベルを再計算表示する）は、効果がフォーム内に留まり、そのフォーム自身の仕様書に書けば追える。これは**横断ではなく A3/C＝追補**（当該フォーム自身の仕様へ追記）とする。判定目安「その連動の効果は本フォームの外へ出るか？ Yes＝A2 横断新設／No（フォーム内で完結）＝A3・C 追補」。
| **A3** | 欠落・分岐/網羅 | セクションは存在するが列挙が不足（設定分岐・権限分岐・グリッド列・右クリック項目・エラーケース等が代表例止まり）。例: 施設設定 OFF 時の列非表示が未記載、enum Col 全列と突合していない | 記載漏れ③ |
| **B** | 記載誤り | コード実態と仕様記載が不一致。例: 設定キー名の誤記、実際は入院のみ有効な分岐を外来共通と記載 | 相違 |
| **C** | 粒度不足 | 記載はあるが Phase5 テスト化に足る具体度がない（前提データ・操作・期待結果が書けない）。例: E2E 候補が「保存できること」レベル | （新規・従来未分類） |
| **D** | 様式違反 | テンプレ様式・規定への違反（tier 誤判定、必須セクション欠落、E2E ID 書式不正、状態行なし）。例: write-path を持つのに tier 簡易 | （新規・従来未分類） |
| **E** | 要確認 | コードだけでは判断できず業務有識者・原開発者への確認が必要。例: 意図不明な分岐が仕様か歴史的バグか判断できない。**ただし設計書が既にその点をバグ候補・既知事項として正確に記載している場合は E に上げない**（設計書＝実装で一致しており差ではない。→ スコープ外／確認済メモ扱い） | 解釈割れ／要確認 |

- `分類` 列は **コードを先頭**に置き、必要なら短い旧表記を併記する（例: `A1 記載漏れ①`、`B 相違`、`E 要確認`）。複合する場合はスラッシュ区切り（例: `A1/B`）。
- **C・D も新たに報告対象**とする。C は「記載はあるが具体度不足でテスト化できない」箇所、D は「設計書テンプレの様式・tier・必須セクション・E2E ID 書式の違反」。ただし本スキルの主眼は A1/A2（子画面・横断の欠落）であり、C/D は明確な場合に拾う（探し回らない）。
- 各出力ファイルの `## 差異一覧` の直前に、使ったコードの**凡例（1行）**を置く（例: `> 分類コード: A1=子画面欠落 / A2=横断欠落 / A3=分岐網羅欠落 / B=記載誤り / C=粒度不足 / D=様式違反 / E=要確認`）。

## 子画面・委譲先の扱い（実在確認の必須）

記載漏れ① の判定でもっとも事故が起きやすいのが「この子画面は**別の仕様書で抽出済みだろう**」という**未確認の思い込み**である。実際、代表画面から `new → ShowDialog/ShowModal` で開く子画面は、原設計書に「〇〇画面を開く」の一行しか無いことが大半で、その詳細が別仕様書にある場合と、どこにも無い場合が混在する。両者を取り違えると、詳細がどこにも無い子画面が「別で抽出済み」と誤判定され静かに落ちる（実例: ある住所編集ダイアログが数百行・外部連携付きの実画面なのに、原設計書は「開く」だけ・独立仕様書も無く、レビューも「別担当で確認」と丸投げして漏れた）。

以下を**必須ルール**とする。

1. **別仕様書カバレッジの実在確認** — 子画面・委譲先を「別仕様書で抽出済みだから対象外」と扱うには、**出力フォルダ（＝突合対象の仕様書が置かれたフォルダ）内をクラス名で `Glob`/`Grep` し、その子画面を主対象とする仕様書ファイルが実在することを確認する**。
   - 実在した → 差異一覧に **「相違/確認」ではなく確認済みメモ**として1行残す（分類「解釈割れ→なし扱い」ではなく、影響列に「詳細は `<実ファイル名>.md` で抽出済み。本タブでは重複記載しない」と明記）。**黙って省略しない**（読み手が「漏れ」と「意図的省略」を区別できるようにする）。
   - 実在しなかった → 原則どおり **記載漏れ①／対処区分「新規抽出」** として上げる。実画面の行数・主な入出力・外部連携の有無を根拠に添える。同一クラスが複数モードを持つ場合（例: 新規登録と編集を1フォームで兼ねる）、別名の仕様書で抽出されていることがあるので**クラス名（実クラス名）で検索**する（画面表示名では見つからない）。
2. **未確認の「他担当範囲」へ差異を丸投げしない** — 「本タブでは深掘りせず、〇〇タブ／〇〇担当で確認」と書いてよいのは、その担当範囲が**この突合の検証単位として実在する**ときだけ。存在しない担当範囲（例: 住所は独立タブではなく基本情報タブ内なのに「住所タブ担当で」と書く）へ回すと、誰も拾わず夹縫落ちする。自分の担当範囲に属する子画面は自分で①判定まで完了させる。

## 医療安全の明示（必須）

報告する各差異について、以下のいずれかに触れる場合は**行頭に `【医療安全】` タグ**を付ける。触れない場合は付記不要。

- 患者識別（PatientID の紐づけ・患者切替・取り違え）
- 薬剤量（用量・用法・単位）
- 検査値
- 外部システム送信（介護度連携・レセプト・PHR・栄養連携 等）
- 処方中止の伝播

## 深刻度・対処区分（各差異に必ず付与）

移植計画・別ドキュメントへの転記に使うため、報告する各差異に **深刻度** と **対処区分** の2項目を必ず付ける。

### 深刻度
- **高** — 医療安全機能（患者識別・検査値・薬剤・外部送信）に関わる欠落/誤り、または移植時に機能欠落・挙動差異を生み得る
- **中** — 移植自体は成立するが、テスト網羅性・抽出の完全性に影響する
- **低** — 様式・可読性・軽微な記載揺れ

### 対処区分（後続アクション。分類コードから既定で決まる）
- **新規抽出** — 未抽出の**子画面・委譲先コントロールを新規に仕様書化**する → `30_queue_phase2_spec_sweep.md` へ積む（スコープ CSV と照合）。**A1 の既定**
- **再抽出** — 当該仕様書を FULL で再抽出する（tier 誤判定含む） → 同じく queue へ積む。**D の一部**
- **横断新設** — 画面横断フローの仕様書を新設する（例: `output/03_…/_flow_order_lifecycle.md`） → queue＋設計判断。**A2 の既定**
- **追補** — 既存仕様書の該当セクションへ追記する。**A3・C の既定**
- **修正** — 誤記載を訂正する。**B の既定**
- **確認** — 有識者確認のみ。結果を仕様書へ反映して完了。**E の既定**

**分類コード → 対処区分の既定対応（原則これに従う）:**

| 分類 | 既定の対処区分 | 補足 |
|------|--------------|------|
| A1 欠落・子画面 | **新規抽出**（queue へ積む） | 未抽出の子画面は「画面まるごと1件」で新規抽出。**内部機能は個別行に列挙しない**（下記） |
| A2 欠落・横断ロジック | **横断新設**（queue＋設計判断） | オーダー中止の会計/在庫/電子処方箋伝播、外部送信、**別フォーム/別画面**への表示依存 等（※同一フォーム内のタブ間表示同期は横断ではなく A3/C 追補） |
| A3 欠落・分岐/網羅 | **追補** | 既存セクション（グリッド列・設定・権限分岐・エラーケース）へ追記 |
| C 粒度不足 | **追補** | 既存セクションを Phase5 テスト化に足る具体度へ補う |
| B 記載誤り | **修正** | 誤記の訂正 |
| D 様式違反 | **再抽出** または 追補 | tier 誤判定・必須セクション欠落は再抽出、軽微は追補 |
| E 要確認 | **確認** | コードだけで仕様/歴史的バグを判断できない |

> **重要（過去の誤運用の訂正）**: 「タブ単位で抽出している」という運用は、**突合レポートの出力ファイルをタブ単位で束ねる**（子画面ごとに出力 md を分けない）という**出力体裁の話**であって、対処区分とは**別の軸**である。未抽出の子画面（例: ある住所編集ダイアログ）は、親タブが既存仕様書に載っていても **A1／対処区分は「新規抽出」（queue へ積む）** が正しい。タブ単位を理由に子画面を「追補」に落とさないこと。
> - **例外**: 子画面が既に独立仕様書で抽出済み（フォルダ内にクラス名の md が実在）→ 差異ではなく確認済みメモ。既存セクションに軽微な補足を足すだけなら A3/C＝追補。

> **重要（新規抽出の粒度＝1画面1件。内部機能を列挙しない）**: 突合は「既存仕様書 vs 実装」の比較であり、**仕様書が丸ごと存在しない子画面**については「この画面は仕様書が無い → 新規抽出」を**画面ごとに1行**だけ挙げる。その画面の**内部機能**（入力項目・入力チェック・郵便番号検索・DB保存に使うサーバーコントロール・画面内の補助機能・フィールド桁数 等）を**個別の差異行として列挙・確認してはならない**。これらは新規抽出（＝後続の抽出フェーズ）で当然に洗い出される内容であり、突合で先取りすると**重複作業**になる。まるごと行の中に「主な入出力・外部連携の有無」を一文で添える程度にとどめる。
> - ただし次は「内部機能」ではないので**別行として残す**:
>   - **横断（A2→横断新設）**: その子画面から**他システム／他画面へ及ぶ**もの（外部送信＝外部システム連携・電子処方箋 等、帳票・監査ログへの波及、他タブ表示への連動 等）。
>   - **要確認（E→確認）**: 業務有識者・原開発者の判断が要る点（物理削除の妥当性・記録保持要件との整合 等）。
>   - **親画面側の A3/B**: 子画面ではなく**親タブ**に属するもの（親タブのグリッド列・トリガー・権限分岐・誤記）。
> - 判定の目安: 「その指摘は、子画面を新規抽出すれば自動的に仕様化されるか？」→ **Yes なら内部機能（まるごと行に集約）／No（他へ波及・要判断・親側）なら別行**。

## 説明の書き方（必須）

結果は**人間のレビュー担当（その機能に詳しくない人・非開発者を含む）が読んで理解できる**ように書く。記号や略記・クラス名の羅列だけで済ませない。各差異は次の3点が平易な日本語で伝わること。

- **何が**: 設計書は何と書いてあり、実コードは実際どうなっているか
- **どう違うか**: その食い違い/漏れ/曖昧さを一文で
- **なぜ問題か（影響）**: 放置するとどんな運用・医療上の不都合が起きるか。専門用語には短い補足を付ける

書き方の要点:
- 主語・目的語を省略しない（「これ」「それ」で済ませない）。画面名・操作・患者への影響を具体的に書く

## 実行フロー

### 1. 仕様書と実装の特定
- 仕様書を `Read` する。
- 対象クラス/画面の実ファイルを **`Glob`/`Grep` で特定**（推測でパスを組み立てない — プロジェクトの「推測禁止」方針に従う）。`.cs` と `.Designer.cs` の両方を対象にする。

### 2. 出力先の決定
- 指定があればそれを使う。
- **省略時の既定**: `%USERPROFILE%\Documents\<機能名>\`（`<機能名>` は仕様書ファイル名から導出。例 `SomeScreen.md` → `Documents\SomeScreen\`）。
- 既存の出力ファイルがあれば、上書き前に同フォルダ内 `old/` へ退避する（`mkdir old` → 既存 `*.md` を move）。

### 3. 並行検証（セクション単位）
- 検証単位ごとに `general-purpose` サブエージェントを**並行起動**する（`superpowers:dispatching-parallel-agents` 準拠。独立領域を1エージェント＝1担当に）。

- 各エージェントは自分の結果 Markdown を出力先に直接 `Write` し、要点3点を返す（メインの文脈を節約）。
- エージェントが**ツール未使用で異常終了**した場合は再起動する（成果物ファイルの更新時刻/存在で確認）。

## Common Mistakes

- **推測でパスを組む** → 41,000+ファイルの大規模コードのため誤参照必至。必ず Glob/Grep で実在確認。
- **バグ探しに逸れる** → このスキルは要件整理であってデバッグではない。null参照・年跨ぎ等の不具合摘出に力を注がない。実装挙動が仕様と食い違えば「相違」、仕様が沈黙していれば「記載漏れ」として拾うにとどめる。
- **設計書が既に正確に書いている点を E で二重確認する** → 設計書がバグ候補・既知事項として正確に記載していれば、設計書＝実装で一致しており突合上の差ではない。E に上げ直さず `（差異なし）`／確認済メモに留める（バグ修正要否の判断はスコープ外）。判定目安「設計書はこの点で実装と食い違っているか？ No なら差異ではない」。
- **対象を数量で丸める** → 「3グリッド」「各画面」等の数だけの見出しは対象が伝わらない。画面名・項目名を名指しし、必要なら控件名/クラス名を括弧添え。「コード変換」「施設分岐」等の抽象語も具体を一言添える。
- **スコープ逸脱** → 純粋なコード品質・規約違反（実害のない空catch/命名等）を混ぜると「突合」でなくなる。報告は「記載漏れ・相違・解釈割れ」の3分類に限定する。
- **代表画面だけ見て子画面・横断を素通り** → 記載漏れの大半はここに潜む。① 委譲先/子画面（`Grep "未確認|委譲|追補|本体未定義"`）、② 画面横断ライフサイクルとサーバーコントロール側、③ enum Col・設定分岐の網羅、を必ず当てる。
- **子画面を「別で抽出済みだろう」と実在確認せず素通り** → もっとも事故が多い。別仕様書の実在は**仕様書フォルダをクラス名で `Glob`/`Grep`**して確認する。実在すれば確認済みメモを残し、無ければ記載漏れ①で上げる。「開くだけ書いてある」子画面を黙って落とさない（実例: ある編集ダイアログの漏れ）。
- **未確認の他担当範囲へ差異を丸投げ** → 「本タブでは深掘りせず〇〇担当で確認」は、その担当範囲が検証単位として実在するときのみ可。実在しない範囲（住所は独立タブでなく基本情報タブ内 等）へ回すと夹縫落ちする。自分の担当の子画面は自分で完了させる。
- **医療安全タグの付け忘れ** → 患者識別/薬剤/検査/外部送信/処方中止に触れる差異は必ず明示。
- **仕様書の行番号を鵜呑み** → 抽出仕様は行番号がズレ、別タブのコードを誤参照していることが多い。実コードで位置を確認する。
- **サブエージェント異常終了の見落とし** → ツール未使用で終了＝未実行。成果物の存在/更新時刻を確認し再起動する。
- **総括で個別ファイルを再読込して文脈を膨張** → 各エージェントの返答要約から総括を組む。

## 実績

本ワークフローは大規模画面（実装数千行＋Designer 数千行・多数タブ）の突合を多エージェント並行で処理できる。区分値(enum)の不一致・存在しないサービスへの委譲記述・重要フラグの誤区分・条件分岐ロジックの未記載 等、ドメイン安全に関わる相違/記載漏れを検出する。

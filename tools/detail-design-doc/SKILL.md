# 詳細設計書の作成（コード → 設計書）

## 概要

既存コードを唯一の情報源として、決まった型の「新規の詳細設計書」を書き起こすためのスキル（**任意の実装を対象とする通用スキル**。GUI 業務アプリ／WinForms 等が主だが、バッチ・常駐・連携等の非GUI機能にも後述「非GUI（バッチ・連携）の読み替え」で適用する）。対象コードの場所は起動時に指定し、プロジェクト固有の探索起点・命名規約は対象プロジェクトの CLAUDE.md／規約に従う。この画面/機能を知らない開発者・レビュー担当者・非開発者が、実装を見なくても仕様と処理内容を理解できるレベルを目指す。

中核原則:

1. **実装が唯一の真実。確度を3段階で明示する。** テーブル名・カラム名・設定名・画面項目名・メッセージ文言は必ず Grep/Glob/Read で実在を確認してから書く（CLAUDE.md「推測禁止」）。各記述は確度で3分する: **確定**（コードに直接の裏付けがある。通常はこれ）／ **推定**（命名・パターン・呼び出し関係から合理的に導けるが直接の裏付けはない。設計意図・rationale 等。本文に書いてよいが必ず〔推定〕を付け、断定形で書かない）／ **未確認**（安全に判断できない。第11章へ隔離し、本文に埋めない）。**コードに存在しない画面項目・テーブル・カラム・設定・メッセージを創作しない**（幻覚の禁止）— 見つからなければ〔推定〕か第11章に落とす。**第11章は「要人間確認の一元収集点」**であり、未確認だけでなく理解不完全（追い切れない）・未処理（共有と判定し別設計書が未作成 等）・低信頼も分類つきで回落する。各所の放棄点（打切り・追跡途中・判定保留）は必ず第11章に1行残す。ただし〔推定〕は本文に置き、真の gap のみ回落する（ダンプ化の防止）。
2. **成果物は「新規の詳細設計書」であって、実装との対照・突合・バグ指摘ではない。** 差異・バグ・課題の洗い出しは別スキル（spec-impl-reconcile）の役割。設計書に対照結果・「既知のバグ」章を混ぜない。
3. **設計段階の文書として書く。コードを前面に出さない。** プログラムコード片・擬似コード・クラス名/メソッド名/行番号の羅列を本文に並べない。書くのは業務仕様（画面項目・操作・処理の流れ・データ項目・条件）。**例外は第6章の SQL**：対象が手書き SQL を用いる場合、SQL は「コード」ではなく「データ入出力仕様の正」なので、実際に発行される形をそのまま記載する。テーブル名・カラム名・画面項目ID（コントロール名）・設定キー名・メッセージ文言も記してよい。コントロール種別は WinForms 型名を英語で記す（TextBox, ComboBox, DataGridView, CheckBox, RadioButton, Button, Label）。実装位置の根拠（file:line）は本文に散らさず付録Bに最小限。
4. **決まった章立ての型（template.md・全11章＋付録）を必ず使う。** 見出し・章順・表の列を変えない。型の再現性がレビュー品質を担保する。
5. **読んだファイルを全部記録する（付録A・網羅の証明）。**
6. **DB 入出力は「ユースケース」として書く（第6章）。読者は AI。** この設計書の後工程は、AI が第6章から DDD アプリケーション層のインターフェースメソッド（Query/Command）を生成することである。したがって DB 操作は「表への CRUD」ではなく「業務としての依頼（UC）」を主語にし、UC ごとに入力/出力項目・事前/事後条件・トランザクション境界・業務ルール・結果とエラーの対応をそろえる。UI のみの責務（Button 有効化・フォーカス・表示切替）は UC に混ぜない。SQL は現行挙動の参考記録であり、移行先の契約は 6.2 のユースケース定義。曖昧語（「適宜」「〜など」「必要に応じて」）を仕様として使わない — AI は解釈で埋めるため、決まっていないなら第11章「未確認」へ。

## このスキルを使う場面 / 使わない場面

- 使う: 実装から新規の詳細設計書を起こす。画面・クラス・機能単位。
- 使わない:
  - 既存仕様書と実装の突合・差異/バグ洗い出し（→ spec-impl-reconcile）
  - コードベース概要ドキュメント生成（→ init）

## 非GUI（バッチ・連携）機能の読み替え

本スキルは通用だが、章立ての型（特に第3・4章）は GUI フォームを前提とする。**非GUI機能（バッチ・常駐監視・HL7/ソケット/ファイル連携等）には、章構成を維持したまま各章の意味を次のように読み替える**（UI 固有章は「該当なし（非GUI）」と明記。この「該当なし」明記は本スキルの既存の欠章規約と同じ扱い）:

| 章 | 非GUIでの読み替え |
|---|---|
| 2 位置づけ・トリガー | 起動契機を UI イベントでなく「スケジューラ起動／常駐監視／受信イベント（ソケット・ファイル着信）／他機能からの呼出」で書く |
| 3 画面設計 / 4 グリッド設計 | 「該当なし（非GUI）」 |
| 5 処理設計 | 主処理フロー（受信→検証→変換→DB更新→送信→ログ）を記す。§5.4 の UI 系観点（ダーティ・フォーカス・モード切替）は該当なし |
| 6 ユースケース | 入力＝受信メッセージ・監視フォルダ・引数・設定値／出力＝送信メッセージ・出力ファイル・DB更新・ログ |
| 7 外部連携（主役） | 相手システム・プロトコル（HL7/独自ソケット/ファイル）・データフォーマット・リトライ/エラー時挙動を必ず記載 |
| 8〜11 | GUI と同様（設定・メッセージ・医療安全・未確認） |

「UI もあるが連携が主」の機能は、両方を該当ぶんだけ埋める（該当なしは明記）。

## 章立てと調査起点の対応

template.md の章 ← 調査起点（プロジェクト固有の格納場所・命名規約は対象プロジェクトの CLAUDE.md／規約を参照する。以下は GUI 業務アプリでの一般的な起点）:

| 章 | 記載内容 | 調査起点 |
|---|---|---|
| 1 概要 / 2 位置づけ | 目的・スコープ・呼び出し元・遷移先（専属/共有の判別） | 対象クラス名を Grep（`new <Form>(` に加え Factory・DI・**文字列型名**の経路も。`new` だけでは呼出元・子画面を取りこぼす → Step 1）／`ShowDialog`。メニュー結線。開く子画面は Step 1 で専属（→本書に取り込む §2.2表A）／共有（→別設計書 §2.2表B）を判別 |
| 3 画面設計 | 画面項目・イベント・バリデーション・コンテキストメニュー・条件付き表示制御 | `<Form>.Designer.cs` の `InitializeComponent()`（宣言・Text・TabIndex・初期 Visible/Enabled）、`Click +=`/`Load +=` 等の結線 + 本体 .cs のハンドラ、入力チェック/ダーティチェックの結線（フォーム基底のしくみ。例: `CheckValidationControls`/`SetValidationControls`）。右クリックは `ContextMenuStrip` 割当と `Opening` ハンドラ（§3.4）。表示分岐は施設判定・設定・権限・患者状態を読む箇所を Grep し §3.5 に集約（判定キー・既定値は §8）。**非GUI機能では本章全体を「該当なし（非GUI）」とする** |
| 4 検索・一覧設計 | グリッドの検索パラメータ・表示列・返却項目 | 列は `.Designer.cs` の Columns 定義（列名・ヘッダ・幅・表示順）。データ取得元は行選択/検索ハンドラ → DB アクセスの SQL を追い、入力条件＝検索パラメータ、取得カラム＝返却項目として3表に展開。**メインクエリの SELECT ≠ 画面の全列**：各列の供給元を3系統（①メインクエリ ②設定で出し分けされる条件列/サブクエリ ③UI層で後追いに別サブシステムを患者ID一括で呼びマージする列。`SetUserData`/holder 代入から辿る）に切り分けて表示列表に明記する。列見出し↔DB列の対応は推測せず列定義で確認。**非GUI機能では「該当なし（非GUI）」** |
| 5 処理設計 | 処理一覧・処理詳細・複雑ロジック・横断的処理 | 各ハンドラ・private メソッド。横断的処理（§5.4）は**業務に関わる横断**（1つの動作/状態変化が複数コントロール・複数出口・複数画面に扇出し、1つ漏らすと事故になるもの）を対象。観点＝モード切替の全体波及・項目間の連動/再計算・状態の横断伝播・同時実行制御（悲観排他/楽観競合・最新化）・ダーティチェック＋未保存離脱ガード・権限/VIP による業務操作の可否・対象切替時の一括再読込・入力チェックの横断ルール（検証時機/複数エラー集約/再計算後の再検証）・処理中の操作抑止と二重送信防止（冪等）・失敗時の画面状態復旧（編集内容保持/ロック解放/ロールバック）・初期表示の状態決定（モード/状態/ロック/権限の合成・優先規則）。監査/アクセスログ・DBトリガ・共通例外・自動ログオフ等の一律の基盤機構は原則対象外（本画面固有の差異があるときのみ1行）。**マージ・名寄せ処理は「勝敗規則」（どちらの値が残るか。例: 日付降順・同キーはID昇順）まで書く**（医療安全直結値は「条件更新」で止めない）。**存在チェックによる「無言スキップ」分岐**（`Exist*`=false で空ブロック・例外もログも出ない）は付け替え漏れの温床として明記する |
| 6 ユースケース設計 | UC一覧・UC定義（入出力/事前事後/Tx境界/エラー対応）・UC→CRUD表・SQL・テーブル | 画面/Form は SQL を直接持たないことが多い。DB を呼ぶ箇所（サーバ呼び出し層・データアクセス層）を探し、その呼び出し先を Read。**DB アクセス経路1つにつき最低1つの UC を起こす**（複数アクセスが1業務依頼を構成するなら1UCにまとめ、Tx境界に明記）。テーブル・カラムの実在はスキーマ/エンティティ定義で確認する（プロジェクトのデータ定義規約に従い、フォルダ名とテーブル名の単複差異等に注意）。**削除/付け替え対象テーブルはワイルドカードで済ませず実体を全列挙する**（不可逆削除は実体一致が必須） |
| 7 外部連携 | 他システム・ファイル・印刷 | 該当コードを追う。**送信/連携は購読側（受信側・Subscriber）まで読む**——「通知」と書いてあっても実際は患者ID/キーを書き換える・送信の条件分岐/中止/件数ガードが購読側にあることがある。**「いつ送るか（トリガ）」だけでなく「何が送信/通知リストに入るか（成員の絞り込み）」も抓む**——「新規＝無条件／更新＝特定項目（担当医・保険・時刻等）が変化したときのみ送信対象」のような送信側の絞り込みが漏れやすい。**連携先が外部EXE起動（CommonExe 等）や反射ロードDLLの場合は「実処理はリポジトリ外＝コードから仕様抽出不能の境界」と明記し §11 へ回落**。無ければ「該当なし（確認済み: …）」 |
| 8 設定値 | 設定キー・既定値・影響＋影響箇所への回参照（集中管理表） | 設定値を読む箇所（設定参照 API 呼び出し）を Grep → 設定定義と既定値の格納場所（プロジェクトの設定定義層）を辿る |
| 9 メッセージ | 文言・分岐 | メッセージ生成・表示のしくみ（メッセージカタログ等）。ダイアログ表示メソッドの戻り値分岐 |
| 10 医療安全（重要ドメイン留意点） | 仕様上の留意点（医療安全・金額・個人情報等） | プロジェクトの規約・レビュー観点（例: `.claude/rules/` があれば参照） |

## 手順

### Step 1. 対象と関連ファイルを収集する

対象の Form/クラスを起点に、上表の調査起点で関連コードを集める。大きい対象（画面項目20超・イベント多数・複数タブ）は、ファイル群を分けて並行 Read する（Designer / イベント本体 / サーバコマンド / モデル・設定を別々に）。

**子画面を「専属」と「共有」に判別する（設計書の範囲確定。重要）。** 対象画面が開く子画面を列挙し、各子画面クラスの**全開放点（どこから開かれるか）**を洗い出す。**判定は開放点の「数」ではなく「帰属」で行う（下記の集合包含テスト）。** 開放点の洗い出しを漏れなく行うのは見落とし防止のためであり、開放点が多いこと自体は判定材料ではない。

**開放経路は `new` だけではない（数え落とし＝誤判定の主因）。** GUI 業務アプリでは画面生成が間接化されていることが多く、`new <子画面>(` の Grep だけでは開放点を取りこぼす。取りこぼすと「共有画面を専属と誤判定 → 不要な章を作る/断リンク化」する。次を**すべて**当たる:

- **直接生成**: `new <子画面>(`
- **Factory / 生成専用クラス経由**: `<子画面>Factory` 等のファクトリの生成・呼び出し箇所。ファクトリが実際に返す画面クラスを Read で特定し、そのファクトリの利用箇所を開放点として数える（本コードベースは多数のフォームがファクトリ経由）
- **DI コンテナ解決**: `Resolve<...>` / コンストラクタ注入等で解決される場合は登録定義と解決箇所
- **反射・文字列型名（最重要の盲点）**: クラスを**文字列のクラス名／FullName**で読み込む経路（例: `Activator.CreateInstance`、`Assembly.*.GetType(名前)`、プロジェクト固有のクラスローダ）。この場合、子画面クラス名は**文字列リテラル**として現れ、`new` Grep でも roslyn `find_references` でも**捕捉できない**。必ず**クラス名（短名・FullName・名前空間断片）を文字列として Grep** する

**数え方の手順**: ①roslyn `find_references`（あれば）で型参照を取る → ②`new`／Factory／DI の Grep を足す → ③**クラス名を文字列リテラルとして Grep**（反射経路のため必須）。`new` 命中が 0 でも「開放点なし・専属」と即断しない。0 は「間接生成されている」信号であり、Factory・文字列型名を追うこと。判定に使った経路と根拠は第11章に残す。以上で得た**全開放点**を、次の**集合包含テスト**で分類する。

**まず `本機能画面集合` を定義する** = 本設計書の対象画面 ＋ 本設計書に取り込む専属画面（その専属の孫画面も再帰的に含む）。「感覚的に関連」ではなく、起点＋取り込む専属子孫という**閉じた列挙可能な集合**であることが濫用防止の要。この集合に対し:

- **専属画面**（全開放点が `本機能画面集合` の内側に収まる ⊆。入口が N 個あってもすべて機能内なら専属。単独では業務的意味を持たない子ダイアログ）→ **本設計書に取り込み、章を設けて詳細化する**（Step 7）。§2.2 では表(A)に本書内の章参照付きで載せる。**「別設計書」に外出ししない**（存在しない設計書への参照＝断リンクを作らない）。
- **共有画面**（`本機能画面集合` の**外側**からの開放点が**1つでもあれば**共有。外部画面が業務的に関連するか否かは問わない。患者検索・職員検索・定型文テンプレート等）→ 従来どおり**別設計書**。§2.2 では表(B)に存在・入出力の要点に加え**判定根拠（機能外のどの開放点で共有としたか）・別設計書の状態**を記す（状態=未作成 は第11章 `未処理` へ回落）。
- **判据の本質は「機能外からの開放点が存在するか（二値）」であり「何回呼ばれるか」ではない。** 複数入口を持つだけの専属子ダイアログ（一覧のツールバー／右クリック／ダブルクリックから開く編集ダイアログ等）を、入口数を理由に共有と誤判定しない。
- 判別が曖昧なもの（ある開放点が集合外かどうか確信が持てない場合。入口が多いことは「曖昧」ではない）は保守的に**共有として扱い、無理に取り込まない**（判定根拠を第11章の `理解不完全` 分類に残す）。
- 専属画面がさらに専属の孫画面を開く場合も同様に取り込む（同一出力フォルダ・index に親子階層を反映）。ただし取り込みは「対象機能に閉じる」範囲まで。深追いで無関係画面まで巻き込まない。

取り込む専属画面については、その Designer / イベント本体 / サーバコマンド / 設定も Step 1 の収集対象に含める。

**証拠の落盤保存（後段の自己校验のための二次利用）。** 探索を並行サブエージェントに委ねる場合、各サブエージェントには「要約を返す」だけでなく **調査した生の事実（発行SQL全文・enum の全列挙・設定キー名と既定値・メッセージ文言・コントロール定義・外部連携の実行条件と実行位置・`file:line`）を `<出力先>/_evidence/<NN>_<topic>.md` に Write する**よう指示する。これは既に生成しているテキストを保存するだけで追加コストはほぼ無く、Step 10 の自己校验が「コードを読み直さずに md と証拠を突合する」ための土台になる。単独で調べた場合も、圧縮前の生事実を同フォルダに残す。`_evidence/` は最終成果物ではない中間物（HTML 化しない）。

### Step 2. 章の型で書く

template.md を骨組みとしてコピーし、各章を埋める。章の順序・見出し・表の列は変えない。各章の書き方・列定義は template.md 内のガイドコメントに従う。

**業務語で書く。**「どのメソッドが呼ばれるか」ではなく「何をするか」。該当がない章（外部連携なし・設定なし等）は削除せず「**該当なし（確認済み: <どこを見たか>）**」と明記する。確認せずに省略しない。

**「表示する／実行する」と書く要素は可達性を確認する（標称挙動をそのまま写さない）。** 画面項目・メッセージ・処理を「表示される／実行される」と書く前に、実コードにそこへ到達する経路が実在するか確認する。控件を「表示」と書くなら `Visible = true`（または表示化する経路）が実コードにあるか確認し、**Visible が false に設定されるだけで true 化されない／フォールバック分岐が前段の `return` で短絡されて到達しない** 等のものは、標称挙動どおりに書かず「死UI・不可達（確認済み: <どこを見たか>）」として第11章へ回落する（設計書に「表示される」と書いた要素が実際は一度も出ない、を防ぐ）。

**同じ可達性を「入力・パラメタ・引継ぎ」にも適用する（配線切れ・デッドパラメタの検出）。** 呼出元が渡す入力（InParam キー・引数・設定・引継ぎ値）を「受け取って使う／条件に渡す」と書く前に、受け側がそのキーを**実際に読み取り、後続（サーバ呼出の InParam・SQL 条件・状態更新）へ転記しているか**を実コードで辿る。**呼出元が渡していても受け側が一度も読まない「デッドパラメタ／配線切れ」**（例: 除外リストのキーが渡るのに引継ぎクエリの条件へ転記されない）は、標称のデータフローが機能する前提で書かず、「受け側で未使用＝当該条件は効かない」という事実を記す（第11章へ回落、または該当 UC/処理に明記）。渡された入力を UC 入力・§前提条件に列挙する際も、未使用のものは「（未使用）」と明記して機能する入力と区別する。

### Step 3. グリッドは第4章に3表そろえて展開する

グリッドごとに **(1)検索パラメータ (2)表示列定義 (3)返却項目** の3表を必ず作る。第6章の SELECT の条件・取得カラムと一致していることを確認する。画面項目定義（3.1）ではグリッドは1行に留める。

**列の網羅基準はメインクエリの SELECT ではなく「画面が実際に表示する全列」。** (2)表示列定義には各列の供給元を明記する: ①メインクエリ ②設定で出し分けされる条件列/サブクエリ（`colVisibleXxx` 等） ③UI層で後追いに別サブシステム（予約・手術・メモ等）を患者ID一括で呼んでマージする列（`SetUserData`/holder への代入から取得元クラス・SQL・テーブルまで辿る）。②③はメインクエリに現れず取りこぼしやすいため、①だけを根拠にしない。列見出し（日本語）↔DB列の対応は推測せず列定義で確認する。

### Step 4. 複雑な業務ロジックは複雑度レベルで書き分ける

**プログラム擬似コードは使わない。** template.md §5.3 のレベルに従う:

- L1 単純 → 日本語の番号手順のみ（1〜2文で足りるものは概要だけ）
- L2 入れ子/多分岐（3分岐以上） → 「条件 → 結果」表
- L3 状態を持つ（中止ステータスの保持/設定/クリア等） → 状態遷移表（または Mermaid 状態遷移図）
- L4 算定・用量・数量・金額の計算 → 日本語の計算手順＋必要なら数式（コードではなく）

**分岐は「主経路」だけで満足しない（記載漏れの最大要因）。** 次を必ず満たす:

- **補集合（else）側を必ず書く。**「条件 → 結果」表は if 側だけでなく else/補集合側（何が出るか。「何もしない」も含む）を明記する。フォールバック連鎖（A が無ければ B、B が無ければ C）は、途中の分岐が `return` 等で短絡して末尾（既定処理）に到達しないケースがないか実コードで確認する。
- **子条件限定の付随処理は guard を条件に含める。** ある付随処理（番号採番・状態更新・通知等）が特定条件下だけで走る（「初回受付時のみ」「特定項目が変化したときのみ」）なら、その guard を明記する。無条件の付随処理として書かない。
- **「自動選択・既定値補完」系を分岐として必ず書く。** 候補1件で自動選択／前回値の継承／特定モードでの自動選択抑止 等は、UI 上は見えにくいが業務挙動を左右する分岐。§3（画面）または §5 に条件つきで書く。
- **同一動作の多入口の後続処理差を書く。** 同じ業務動作（受付確定・保存・送信等）が複数の入口（実行ボタン／閉じる時の保存確認／外部トリガ）から起動される場合、各入口で後続処理（印刷・通知・画面遷移・付随処理）が同一か、入口ごとの差分を明記する。

### Step 5. DB 入出力をユースケースに起こす（第6章）

DB を呼ぶ箇所（サーバ呼び出し・データアクセス層）を列挙し、UI 非依存の業務動詞で UC 命名する（「検索ボタン押下」ではなく「検査結果を検索する」）。UC ごとに template.md §6.2 の型で:

- 入力項目 ⇔ グリッドがあれば §4.1(1) 検索パラメータと1対1一致
- 出力項目 ⇔ §4.1(3) 返却項目と1対1一致（件数上限・打切りフラグ等のメタ情報も出力に含める）
- 事前/事後条件・トランザクション境界・排他を業務語で
- 結果とエラーの対応表：業務エラー（結果として返す・メッセージID対応）と技術例外を区別する

**画面処理（P-xx）と UC は分離する。** P-xx には UI 側の責務（Button 有効化・選択保持・フォーカス）が残ってよいが、UC には持ち込まない。P-xx → UC-xx の対応は 6.1 の一覧で示す。

**UI に業務ロジックが混在している場合の扱い（付録C・移行向け・任意）。** イベントハンドラや画面側メソッドの中に採番・計算・条件判定・DB更新・外部送信などの業務ロジックが埋め込まれている（＝UI と業務が同じ場所に同居している）ことは多い。本スキルは**それを正文へ「混入している」と評判調で書かず、書く時点で責務ごとに分解して各章へ振り分ける**（業務＝UC/§5.3、UI＝P-xx）ことで解消する。加えて **BL 抽出（Strangler Fig 等の移行）を後工程に持つ場合**は、その分解の副産物を**付録C「業務ロジック配置／剥離マップ」**として1表に落とす（純GUI保守目的だけなら任意）。次を守る:

- **事実の配置に限る。** 良し悪し・「リファクタすべき」等の評判は書かない（→ spec-impl-reconcile の領域）。
- **確度は逐列で異なる。** 「業務ロジック（何を）／種別／現在の所在」＝確定、「帰属層／剥离先」＝〔推定〕（剥离後こうなるという設計投影で、切り方は一意でない）、業務意図が読めず帰属できないものは剥离先を空欄にして**第11章 `未確認` へ**（凑うために UC を創作しない）。
- **帰属層は4種に分ける（「剥离先＝UC」だけでは足りない。混入ロジックは行き先が割れる）。** ①応用層（編排＝複数UCの順序・Tx・リトライ制御）②ドメイン規則（採番ポリシー・用量計算・検証/免除ルール）③UC（既存/新規のデータ操作）④UI残留（ボタン制御・確認ダイアログ等＝移行対象外）。ReceMain 検証で1列では表現できないことが判明したため帰属層列を必須化。
- **剥离先は §6 の UC-ID を指す。** §6 に未定義（応用層メソッド等）のみ、付録Cの**署名スケッチ**（名前・入出力・事前事後・Tx・呼出元を宣言形で並べる。**過程式擬似コード＝if/for/代入は書かない**——未来インターフェースの「形」であって実装ではない）で添える。

### Step 6. 医療安全上の留意点を「仕様」として書く（第10章）

患者識別・用量・処方中止/変更の伝播・外部送信・診断/検査/フラグの表示記録・算定期間などについて、この画面の仕様上で気をつけて扱う点・設計意図・制約を記す。とくに**マージ・名寄せの勝敗規則**（残る値の決定規則。血液型・アレルギー・禁忌など医療安全直結値）と**外部送信の実体**（購読側まで読んだ上での送信内容・患者ID書換・中止・件数ガード）は仕様として明記する。

**禁止:** ここを実装バグ・規約違反の指摘欄にしない（「この catch が握りつぶしている」「この SQL が非パラメータ化」等は spec-impl-reconcile へ）。

### Step 7. 出力形式を決める（自適応分割）

**既定の出力先**は `~/Documents/docs/`（`~` はユーザーのホーム。Windows は `%USERPROFILE%\Documents\docs\`）。ユーザーが出力先を明示した場合はそれに従う。ディレクトリが無ければ作成する。以下の `<出力先>` はこの既定ディレクトリを指す。

- **単一 md**（`<出力先>/<対象名>.md`）: 画面項目 ≲15、グリッド ≲1、イベント ≲6、DB操作 ≲5、複雑ロジックなし、**かつ専属画面（Step 1 判定）を持たない** 小規模画面。
- **複数 md**（`<出力先>/<対象名>/`）: 上記を超える場合、**または専属画面を1つ以上取り込む場合**。分け方は2通り:
  - **章別分割**（タブが少ない画面）: `00_index.md`（1・2章＋目次＋付録A）/ `01_画面.md`（3・4章）/ `02_処理.md`（5章）/ `03_データ.md`(6章) / `04_連携_設定_メッセージ.md`（7・8・9章）/ `05_医療安全.md`（10・11章）
  - **タブ/機能別分割**（タブが多い複雑画面。推奨）: `00_index.md`（1・2章＋共通のライフサイクル/設定/DB概要＋付録A）＋ タブごとに `01_基本情報タブ.md` … のように、そのタブの画面項目・グリッド・処理・メッセージ・医療安全をまとめる。読み手がタブ単位で完結して読める。

**専属画面の章ファイル。** Step 1 で取り込むと判定した専属画面は、親画面の章に続けて**専属画面ごとに1つの章ファイル** `0X_<和名>ダイアログ.md` を作る（タブ章と同じ連番の続き。例: 親が `01_基本情報タブ.md` なら `02_住所編集ダイアログ.md`）。章ファイル内は**同じ章立ての型を必要な範囲で**用いる（3 画面設計・4 グリッド・5 処理・6 ユースケース・9 メッセージ・必要なら 10 医療安全）。親と共通のライフサイクル・設定・付録は重複させず `00_index.md` に集約し、専属画面固有ぶんのみ書く。`00_index.md` の目次には親画面章と各専属画面章を並べ、専属の孫画面がある場合はインデント等で親子階層を示す（読み手が「この機能に連なる専属画面の全体像」を一目で掴めること）。

分割の有無に関わらず、`00_index.md`（単一なら巻末）に**付録A 解析対象ファイル一覧**を必ず含める（取り込んだ専属画面の解析ファイルも含める）。file:line が必要なら付録Bに最小限。**BL 抽出（移行）を後工程に持つ場合は付録C 業務ロジック配置／剥離マップ**も `00_index.md`（単一なら巻末）に置く（Step 5 の分解の副産物・任意）。**要確認事項（未確認・理解不完全・未処理・低信頼）は第11章に一元集約する**（収集点は1つ）。第11章の置き場は分割方式による: **章別分割なら `05_医療安全.md`（第10章と同居）／タブ・機能別分割なら横断内容を集約する `00_index.md`**。分割出力では `00_index.md` の目次に「要確認事項は §11 に一元集約」と1行案内し、その所在（上記の置き場）を明示する。

### Step 8. 網羅チェック

Step 1 で集めたすべての **イベントハンドラ・コントロール・グリッド（3表そろっているか）・DB アクセス・設定参照・メッセージ** が設計書のどこかに現れているか照合する。加えて:

- すべての DB アクセス（サーバ呼び出し・直接テーブルアクセス）が、いずれかの UC（6.1）に対応づいているか
- 第4章 検索パラメータ ⇔ UC 入力項目、第4章 返却項目 ⇔ UC 出力項目 ⇔ SQL 取得カラム が一致するか
- 各 UC の「結果とエラーの対応」の業務エラーが、第9章のメッセージIDと対応づいているか
- 更新系 UC にトランザクション境界・排他の記載があるか
- 本文に曖昧語（「適宜」「〜など」「必要に応じて」）が仕様として残っていないか
- 設定分岐が §5/§6 に「設定〈キー名〉が○のとき…」として現れ、§8 の影響箇所列と相互参照になっているか
- §5.4 横断的処理（業務観点）のチェックリスト全11観点が全行埋まっているか（該当なしは「該当なし（確認済み: …）」）。観点＝モード切替波及・項目連動/再計算・状態伝播・同時実行制御（悲観/楽観）・ダーティ＋未保存離脱ガード・権限/VIP・対象切替再読込・入力チェックの横断ルール・処理中の操作抑止/二重送信防止・失敗時の状態復旧・初期表示の状態決定。一律の基盤機構（ログ・DBトリガ・共通例外・自動ログオフ）をノイズとして羅列していないか
- とくに医療系で漏れやすい「二重送信防止（冪等）」「失敗時の編集内容保持・ロック解放」「初期表示の状態決定（権限×ロック×状態×モードの合成・優先規則）」を確認したか（該当なしでも確認済みと明記）
- §5.4 は「全横断の見取り図」として機能しているか。詳細は発生する処理フロー（§5.2 P-xx／§3.5／§6.2 等）に置き、§5.4 は要約＋「詳細箇所」で指す。**双方向**で確認する: (a) フロー等に現れる横断がすべて §5.4 に1行載っているか（載せ忘れ＝散在・漏れ）(b) §5.4 の各行が詳細箇所を指す（または画面全体規模でここに直接記述）になっているか
- §2.2 表(B) の各行に「判定根拠」「別設計書の状態」があり、状態=`未作成` の共有画面がすべて第11章 `未処理` 分類に現れているか（共有への外出しが根拠・状態つきで追跡可能か）
- 要確認事項（未確認・理解不完全・未処理・低信頼）が第11章に一元集約されているか。各所の放棄点（追い切れない・打切り・判定保留）から第11章に1行ずつ回落しているか
- 未確認のまま埋めた箇所がないか（あれば第11章へ移す）
- （付録C を作った場合）剥离先 UC 列が〔推定〕になっているか（確定として断定していないか）・評判調の記述が混じっていないか・帰属できないものが空欄＋第11章 `未確認` に落ちているか・署名スケッチに過程式擬似コード（if/for/代入）が混じっていないか

### Step 9. レビュー用 HTML ビューを生成する

設計書（md）の完成後、**必ず**同フォルダの scripts/build_html.py で HTML ビューを生成する:

```bash
# 単一 md の場合（HTML は md と同じ場所に置く）
python3 <このスキルのフォルダ>/scripts/build_html.py <出力先>/<対象名>.md

# 分割出力の場合（HTML は html/ サブフォルダにまとめ、md と分離する）
for f in <出力先>/<対象名>/[0-9]*.md; do python3 <このスキルのフォルダ>/scripts/build_html.py "$f" --outdir <出力先>/<対象名>/html; done
```

- **分割出力では HTML を `<対象名>/html/` にまとめる**（md 目録を汚さない）。章間ナビ・相互リンクはすべて相対（`01_画面.html` 等）なので、全 HTML が同一フォルダにあればそのまま機能する。`00_index.html` を開けば全章にジャンプできる。
- **md が正本。HTML は閲覧用ビューであり、編集しない。** md を修正したら HTML を再生成する
- 生成した HTML のパスを、成果物の報告に md と並べて記載する
- 変換に失敗した場合（markdown ライブラリ未導入等）は `pip install markdown` を案内し、md のみで完了としてよい（HTML は必須成果物ではない）

### Step 10. 軽量自己校验（既定で実施）

**このステップは既定で毎回実施する**（ユーザーが「自己校验は不要」と明示したときのみ省略）。目的は、生成した md が実装と食い違っていないかを **全量突合（spec-impl-reconcile）の約 1/3 のコスト**で自己点検すること。全量突合の代替ではなく、中〜高 severity を取りこぼさないための簡易ゲートである。**1 サブエージェント**で完結させる。より網羅的・厳密な突合が要る場合（医療安全クリティカル画面・正式移植/評審）は、これで済ませず別スキル `spec-impl-reconcile` を明示的に回す。

設計の要は「二次利用」と「定点深読」の2段構え。片方だけでは効果が出ない:

- **(1) 複用（コード再読なし・圧縮/転写ミスを抓む）**: Step 1 で保存した `_evidence/*.md` と、完成した md を突合する。ここはファイルを読み直さないのでほぼ無料。抓む対象:
  - 第6章の SQL 字面（`SELECT *` 等の要約に潰していないか。列白名単なら列挙されているか）
  - enum・区分値が**代表例止まり**でなく**全列挙**か（グリッド列・種別コード等）
  - 設定キー名・既定値・メッセージ文言の転写ミス
  - 施設分岐・権限分岐が「〜あり」で濁されず具体列挙されているか
  - 〔推定〕を付けずに断定している箇所はないか（推定を確定と偽らない）。とくに第10章（解釈的記述）と §5.3 の図（Mermaid 等・自動生成で誤りやすい低信頼ゾーン）は断定していないか重点確認する
- **(2) 定点深読（`_evidence` に無い＝生成時に深掘りしていない高リスク 3 類＋低コスト 2 点だけ実ファイルを読む）**: 複用では抓めない。ここを省くと効果が出ない。後半 2 点は複用/grep のみで済み 1/3 コストの枠を破らない。
  - **子画面の取り込み・実在確認**（Glob のみ・ほぼ無料）: Step 1 で列挙した子画面（`new` だけでなく Factory・DI・反射/文字列型名の経路も辿って洗い出したもの）と、専属/共有判定・§2.2 の記載を突合する。とくに `new` 命中が 0 の子画面を「開放点なし」で見落としていないか（間接生成の取りこぼし）を確認する。①**専属**と判定した子画面が §2.2 表(A)にあり、対応する章（分割なら `0X_<和名>ダイアログ.md`、単一なら該当節）が実在するか。無ければ＝**専属子画面の取り込み漏れ**（記載漏れ①）。②**共有**とした子画面は §2.2 表(B)にあるか。独立設計書自体は別途でよいが、専属を安易に(B)へ外出しして断リンク化していないか（判定の妥当性）も確認する。
  - **外部連携の実行条件・実行位置（購読側まで）**: 送信/連携メソッドの呼び出し箇所を Read し、①どの経路（保存ボタンのみか、閉じる保存でも走るか）②送信対象患者の絞り込み条件、③**送信/通知リストに入るレコードの絞り込み条件**（新規＝無条件／更新＝特定項目変化時のみ 等）が md と一致するか。可能なら購読側（受信・出力先）まで見て送信内容（患者ID/キーの書換有無）を確認する。医療安全に直結。
  - **write-path の SQL 列**: 更新系サーバコントロールの UPDATE/INSERT 列と、第6章 SQL・§6.5 テーブルの列が一致するか。
  - **（低コスト①）分岐の補集合(else)・子条件限定の付随処理**（複用＋軽微読）: §5.3 の各「条件 → 結果」表について、else/補集合側と「子条件限定の付随処理（初回のみ／特定項目変化時のみ 等）」が書かれているか実コードで反証する。フォールバック連鎖が途中の `return` で短絡して末尾（既定処理）に到達しないケースも確認する。抜けは A3 追補。
  - **（低コスト②）「表示/実行/受取」と書いた要素の可達性**（grep のみ・ほぼ無料）: (a) §3 で表示と書いた控件について `.Visible = true`（または表示化する経路）が実コードに存在するか grep する。false のみ設定・true 化なし／前段 `return` 短絡で到達しない＝死UI・不可達なら §11 へ落ちているか確認する。(b) 呼出元が渡す入力・引継ぎ（InParam キー・引数・除外リスト等）を「使う」と書いたものについて、受け側でそのキー名を grep し、実際に読み取り・後続（サーバ InParam・SQL 条件）へ転記しているか確認する。渡るのに未読＝デッドパラメタ／配線切れなら、標称データフローどおりに書かず事実（当該条件は効かない・入力は未使用）を記す。

出力: `<出力先>/_selfcheck.md`。spec-impl-reconcile と同じ分類コード（A1/A2/A3/B/C/D/E）・深刻度・対処区分で、**1行1件の表**にする。医療安全に触れる差異は行頭 `【医療安全】`。末尾に「本自己校验は複用＋定点深読（高リスク3類＋低コスト2点）に限定。網羅的な突合は spec-impl-reconcile を使うこと」と限界を明記する。差異が見つかれば md 本体を修正し、必要なら HTML を再生成する（Step 9）。加えて、**共有と判定したのに別設計書が未作成のもの**を1類として洗い出し、§2.2 表(B) の状態=`未作成` と第11章 `未処理` 分類に整合して現れているかを確認する（食い違えば md を修正する）。

**限界（必ず守る）**: 軽量自己校验は「生成 md の圧縮/転写ミス」と「高リスク 3 類＋低コスト 2 点（分岐補集合・表示可達性）」を対象とする。医療安全クリティカルな画面や正式移植・評審に載せる画面は、これで済ませず spec-impl-reconcile の全量突合を回すこと。

## よくある失敗（ベースラインで実際に発生）

| 失敗 | 対策 |
|---|---|
| 独自見出しで書く | template.md の型を固定で使う。見出し・列を変えない |
| 実装のバグ・対照・突合結果を混ぜる | 設計書は仕様のみ。→ spec-impl-reconcile |
| クラス名・メソッド名・行番号を本文に並べる | 業務語で「何をするか」。file:line は付録Bに最小限 |
| 擬似コード・コード片を貼る | L2〜L4 の表・日本語手順・数式で。SQL のみ第6章に記載可 |
| DB を SQL 説明・CRUD表だけで済ませる | UC 定義（6.2）が契約。入出力・事前事後・Tx境界までそろえる |
| SQL の列を「…」で省略 | 列名は全列挙。連番の自明な列（PatientFlag01〜21 等）のみ範囲表記可＋その旨注記。非連番の業務列は省かない |
| UC 名が UI 語（「検索ボタン押下処理」等） | 業務動詞で命名。UI 責務は P-xx に残し UC に混ぜない |
| 「適宜」「〜など」で仕様を濁す | AI が解釈で埋める。決まっていなければ第11章「未確認」へ |
| グリッドを画面項目定義に1行で潰す | 第4章に3表（検索パラメータ/表示列/返却項目）を必ず展開し、SQL と一致確認 |
| コントロール種別を日本語訳 | WinForms 型名を英語で（Button/DataGridView/ComboBox 等） |
| 設定・外部連携の章を無言で省略 | 「該当なし（確認済み: …）」 |
| 業務横断（モード切替波及・項目連動/再計算・状態伝播・排他/競合・ダーティ確認・入力チェック横断・二重送信防止・失敗時復旧・初期表示状態決定）を書き漏らす | §5.4 の観点チェックリスト全11観点を全行埋める。無ければ「該当なし（確認済み: …）」 |
| 二重送信防止・失敗時の状態復旧・初期表示の状態決定（合成規則）を検討し忘れる | 医療系で最も漏れやすく事故に直結（重複オーダー送信・編集内容の損失・ロック中の誤操作）。§5.4 の該当行を必ず埋め、詳細は §5.2／§3.5 に置く |
| 一律の基盤機構（ログ・DBトリガ・共通例外・自動ログオフ）を横断処理として羅列しノイズ化 | 全画面共通の基盤は原則 §5.4 に書かない（業務横断が主体）。本画面固有の差異・特別扱いがあるときのみ1行 |
| 複雑な横断処理を §5.4 の1セルに押し込む | §5.4 は業務横断の見取り図。詳細は処理フロー（§5.2 P-xx。L2〜L4 は §5.3）に書き、§5.4 は要約＋「詳細箇所」で指すに留める |
| 処理フローに書いた横断を §5.4 に載せ忘れ、散在・漏れになる | §5.4 は全横断の見取り図（漏れ防止）。フロー（§5.2）・§3.5・§6.2 等に現れる横断はすべて §5.4 に1行載せ相互参照する |
| 施設・環境分岐を「〜あり」で濁す | 対象施設（HospitalID・グループ判定等）を1つずつ具体列挙。§3.5 で切替内容、§8 で判定キー・既定値 |
| 右クリックメニュー・条件付き表示制御を書き漏らす | §3.4 コンテキストメニュー・§3.5 条件付き表示制御に集約。無ければ「該当なし（確認済み: …）」 |
| 設定の影響を §8 の表だけに書く | 分岐する処理側（§5）にも条件として明記。UC の挙動を変えるなら §6.2 の入力/事前条件に昇格 |
| 読んだファイルを残さない | 付録A を必ず付ける |
| テーブル/カラム名を推測 | スキーマ/エンティティ定義で実在確認。未確認は第11章へ |
| 推定を確定として断定する | 〔推定〕を付け断定形にしない。裏付けなき設計意図・rationale は推定として明示（原則1） |
| コードにない項目を創作（幻覚） | 見つからなければ〔推定〕か第11章。存在しない画面項目・表・列・設定・メッセージを書かない |
| 専属の子画面を「別設計書」で外出しして断リンク化 | 専属（本画面/機能からのみ開く子ダイアログ）は本設計書に章立てして取り込む。§2.2 を専属(A)／共有(B)の2表に分ける（Step 1 判定・Step 7） |
| `new` の Grep だけで開放点を数え、Factory/DI/反射経由の子画面・呼出元を取りこぼす | 開放点は new＋Factory＋DI＋**文字列型名の Grep**で洗い出す。`new` 0件を「呼出なし・専属」と即断しない（Step 1・§2.1） |
| 全開放点が機能内なのに入口数が多いだけで共有と誤判定 | 判定は開放点の「帰属」で（集合包含テスト）。全開放点 ⊆ 本機能画面集合 なら入口 N 個でも専属。数で判定しない（Step 1） |
| 共有と判定した画面を根拠も状態も残さず別設計書へ外出しし、reviewer が未対応を検証できない | §2.2 表(B) に判定根拠・別設計書の状態（既存/未作成/対象外）を記し、状態=未作成 は第11章 `未処理` へ回落（Step 1・Step 8・§2.2） |
| タブが多い画面を1枚に | タブ/機能別分割（Step 7） |
| 現状に無い「移行後アーキ」を先取りして書く（実在しない Service/Repository を核に記述） | 実装が唯一の真実。現状のクラス（旧サーバコントロール等）を実在確認して書く。未来アーキは書かない |
| 削除/付け替え対象テーブルをワイルドカード的に濁す | 実体を全列挙（不可逆削除は実体一致必須。§6.3/§6.4） |
| 外部送信を送信側だけ見て「通知」と書く | 購読側（受信側）まで読み、送信内容・患者ID書換・中止・件数ガードを §7 に明記 |
| 外部EXE/反射DLL連携を通常連携と同列に書く | 「実処理はリポジトリ外＝仕様抽出不能の境界」と §7 に明記し §11 へ回落 |
| 非GUI機能に画面章を無理に埋める／画面前提で書く | §3・§4 は「該当なし（非GUI）」とし §5〜§7 を主体に（「非GUI（バッチ・連携）機能の読み替え」節） |
| グリッド列をメインクエリ SELECT だけで網羅したつもりになる | 設定条件列・UI後追いマージ列（予約/手術/メモ等）まで拾う。供給元3系統を明記（Step 3・§4.1(2)） |
| UI に混在した業務ロジックを正文へ「混入している」と評判調で書き仕様を薄める | 書く時点で責務分解（業務＝UC/§5.3・UI＝P-xx）。移行向けは事実配置として付録Cに閉じる（Step 5・付録C） |
| 付録Cの剥离先 UC を確定として断定する／署名スケッチを過程式擬似コードで書く | 剥离先は〔推定〕（設計投影）。スケッチは宣言形（名前・入出力・事前事後・Tx）のみ、if/for/代入は書かない（Step 5・付録C） |
| 分岐を主経路だけ書き、else/補集合・子条件限定の付随処理（初回のみ／特定項目変化時のみ）を落とす | 「条件 → 結果」表に補集合側を必ず書き、子条件限定の付随処理は guard を条件に含める。フォールバック連鎖は途中 `return` の短絡を確認（Step 4・Step 10(2)） |
| 控件/メッセージを「表示される」と書くが Visible=true 経路が無い（死UI）・フォールバックが前段 return で不可達 | 「表示/実行」と書く前に可達性を確認。到達経路が無ければ標称挙動どおり書かず §11 へ回落（Step 2・Step 10(2)） |
| 自動選択・既定値補完（候補1件で自動選択・前回値継承・特定モードで抑止）を書き漏らす | UI で見えにくいが業務挙動を左右する分岐として §3／§5 に条件つきで書く（Step 4） |
| 外部送信を「いつ送るか」だけ書き「何が送信/通知リストに入るか（成員の絞り込み）」を落とす | 送信側の成員絞り込み条件（新規＝無条件／更新＝特定項目変化時のみ 等）も §7 に書く（Step 7・§7） |
| 呼出元が渡す入力・引継ぎ（InParam キー・除外リスト等）を「使う」と書くが受け側が一度も読まない（デッドパラメタ・配線切れ） | 入力・引継ぎも可達性を確認。渡るのに未読なら標称データフローで書かず「未使用＝当該条件は効かない」を明記（Step 2・Step 10(2)） |
